# CodeGraphContext 한눈에 이해하기

## 이 문서의 목표

이 문서는 CodeGraphContext를 처음 접하는 사람이 짧은 시간 안에 아래 내용을 이해하도록 돕기 위해 작성했다.

- CodeGraphContext가 어떤 프로젝트인지
- 왜 그냥 텍스트 검색만으로는 부족한 문제를 풀려고 하는지
- 이 프로젝트가 내세우는 강점이 무엇인지
- 핵심 개념이 무엇인지
- 저장소 안에서 어떤 부분이 실제 제품 가치의 중심인지

대상 독자는 AI 기술의 기본 개념은 알지만, 코드 그래프나 MCP 서버는 깊게 보지 않은 사람을 기준으로 잡았다.

## 한 줄 설명

CodeGraphContext는 로컬 코드베이스를 그래프로 바꿔서, 사람이든 AI든 코드의 구조와 관계를 더 잘 이해하게 돕는 도구다.

더 정확히 말하면, 프로젝트의 파일과 함수와 클래스와 호출 관계를 그래프 DB에 저장한 뒤 CLI와 MCP 서버를 통해 검색, 호출 분석, 상속 분석, 데드코드 탐지 같은 기능을 제공하는 시스템이다.

여기서 그래프 DB는 데이터를 표 형태가 아니라 "노드와 관계" 형태로 저장하는 저장소라고 이해하면 된다.

## 왜 이런 프로젝트가 필요한가

코드를 이해할 때 많은 사람은 보통 아래 방식에 의존한다.

- 파일 이름을 보고 감으로 찾기
- IDE에서 문자열 검색하기
- `grep`이나 `rg`로 패턴 찾기
- AI에게 코드 일부만 붙여 넣고 설명 요청하기

이 접근은 작은 프로젝트에서는 충분히 잘 통한다. 하지만 프로젝트가 커지면 한계가 빨리 드러난다.

- 함수 이름은 찾았지만 누가 이 함수를 호출하는지는 따로 추적해야 함
- 같은 이름의 함수가 여러 파일에 있으면 어떤 것이 실제 대상인지 헷갈리기 쉬움
- 클래스 상속 구조나 모듈 의존성은 문자열 검색만으로 파악하기 어려움
- AI가 답을 해도 전체 코드베이스를 직접 읽은 것이 아니면 틀릴 수 있음
- 코드가 자주 바뀌는 프로젝트에서는 검색 결과와 실제 구조가 계속 어긋날 수 있음

CodeGraphContext는 이 지점을 겨냥한다. 이 프로젝트는 코드를 단순 텍스트 묶음으로 보지 않고, 함수와 클래스와 파일과 관계가 연결된 그래프로 저장해 구조적으로 이해하려고 한다.

## CodeGraphContext가 푸는 문제

예를 들어 어떤 프로젝트에서 아래 질문이 생겼다고 해보자.

- `process_payment`를 누가 호출하나
- `UserService`는 어떤 클래스들을 상속하거나 구현하나
- `config`라는 변수는 어디에서 정의되고 어떤 범위에서 쓰이나
- 지금 프로젝트에서 복잡도가 높은 함수는 무엇인가
- 더 이상 호출되지 않는 함수가 있나

텍스트 검색만으로도 힌트를 얻을 수는 있다. 하지만 이런 질문은 대부분 "코드 조각 찾기"보다 "코드 관계 이해하기"에 가깝다.

CodeGraphContext는 코드를 인덱싱할 때 아래 정보를 함께 저장한다.

- 어떤 파일 안에 어떤 함수와 클래스가 있는지
- 어떤 함수가 어떤 함수를 호출하는지
- 어떤 파일이 어떤 모듈을 가져오는지
- 어떤 클래스가 어떤 클래스를 상속하는지
- 어떤 함수가 어떤 인자를 받는지

그래서 나중에는 단순 검색 결과가 아니라 "구조가 반영된 답"을 꺼낼 수 있다.

## 이 프로젝트의 강점

### 1. 코드 관계를 그래프로 저장한다

이 프로젝트의 가장 큰 강점은 코드를 그냥 문자열로 다루지 않는다는 점이다.

- 파일을 노드로 저장
- 함수와 클래스를 노드로 저장
- `CONTAINS`, `CALLS`, `IMPORTS`, `INHERITS` 같은 관계를 함께 저장

이 구조 덕분에 "어디에 이 이름이 나오나"를 넘어서 "무엇이 무엇과 연결되나"를 물을 수 있다.

### 2. CLI와 MCP를 둘 다 제공한다

CodeGraphContext는 두 가지 방식으로 쓸 수 있다.

- 개발자가 직접 터미널에서 쓰는 CLI 도구
- AI 에이전트나 IDE가 붙는 MCP 서버

여기서 MCP는 AI 도구가 외부 프로그램의 기능을 표준 방식으로 호출할 수 있게 해 주는 연결 규약 정도로 이해하면 충분하다.

즉 혼자 터미널에서 분석할 때도 쓸 수 있고, AI 코딩 도구에 연결해 코드 이해 레이어로 붙일 수도 있다.

### 3. 여러 종류의 분석을 한 도구 안에서 묶어 둔다

이 프로젝트는 단순 검색 도구가 아니다. 실제로는 아래 작업을 한 묶음으로 다룬다.

- 이름 검색
- 패턴 검색
- 호출자와 피호출 함수 분석
- 호출 체인 분석
- 상속 트리 분석
- 변수 범위 분석
- 복잡도 분석
- 데드코드 탐지

즉 "코드 찾기"와 "코드 이해하기"를 같은 데이터 모델 위에서 처리한다.

### 4. 라이브 감시 기능이 있다

코드를 한 번만 읽고 끝내는 도구가 아니라, 작업 중인 폴더를 감시해서 그래프를 계속 최신 상태로 맞출 수 있다.

- 처음 감시 시작 시 전체 스캔 수행
- 파일 생성과 수정과 삭제 감지
- 변경된 파일을 다시 파싱
- 호출 관계와 상속 관계를 다시 연결

이 덕분에 개발 중인 프로젝트에도 맞출 수 있다.

### 5. 데이터 저장소 선택지가 있다

CodeGraphContext는 특정 DB 하나에만 고정되지 않는다.

- KùzuDB
- FalkorDB Lite
- FalkorDB Remote
- Neo4j

사용 환경에 따라 더 가볍게 쓸 수도 있고, 외부 그래프 DB로 붙일 수도 있다.

### 6. 미리 만들어 둔 그래프 번들을 다룰 수 있다

이 프로젝트는 `.cgc` 번들 기능도 제공한다.

- 이미 인덱싱한 그래프를 파일로 내보내기
- 다시 불러오기
- 레지스트리에서 받아오기

즉 큰 저장소를 매번 처음부터 다시 인덱싱하지 않고, 미리 만들어 둔 그래프 스냅샷을 재사용할 수 있다.

## 핵심 개념

CodeGraphContext를 이해할 때 가장 중요한 개념은 여섯 가지다.

| 개념 | 의미 | 왜 중요한가 |
| --- | --- | --- |
| Repository | 인덱싱한 프로젝트 단위 | 그래프의 최상위 범위 |
| File | 실제 소스 파일 | 함수와 클래스의 컨테이너 |
| Function | 함수나 메서드 | 호출 분석의 중심 |
| Class | 클래스 | 상속과 메서드 구조의 중심 |
| Module | import 대상 모듈 | 파일 간 의존성 파악에 필요 |
| Relationship | 노드 사이 연결 | 구조적 질의의 핵심 |

### Repository

Repository는 CodeGraphContext가 인덱싱하는 프로젝트 단위다.

쉽게 말해 "이 그래프가 어떤 코드베이스를 가리키는가"를 나타내는 최상위 노드라고 보면 된다.

### File

File은 실제 코드 파일이다.

예시

- `src/main.py`
- `src/services/payment.ts`
- `app/models/user.rb`

이 노드는 함수와 클래스와 변수의 상위 컨테이너 역할을 한다.

### Function

Function은 함수나 메서드를 뜻한다.

이 프로젝트에서 가장 자주 질의하는 대상도 보통 Function이다.

예시

- 이 함수는 누가 호출하나
- 이 함수는 무엇을 호출하나
- 이 함수의 복잡도는 얼마나 높나

### Class

Class는 클래스 구조를 표현한다.

이 노드가 있으면 아래 같은 질문이 가능해진다.

- 이 클래스는 무엇을 상속하나
- 누가 이 클래스를 상속하나
- 이 클래스 안에 어떤 메서드가 있나

### Module

Module은 파일이 가져오는 외부 또는 내부 모듈을 뜻한다.

이 노드 덕분에 "어떤 파일이 어떤 모듈에 의존하는가"를 그래프 수준에서 볼 수 있다.

### Relationship

Relationship는 노드 사이 연결이다.

대표 예시

- `Repository -> File`의 `CONTAINS`
- `File -> Function`의 `CONTAINS`
- `Function -> Function`의 `CALLS`
- `File -> Module`의 `IMPORTS`
- `Class -> Class`의 `INHERITS`

이 관계가 있어야 CodeGraphContext가 단순 검색 도구가 아니라 코드 구조 분석 도구가 된다.

## CodeGraphContext는 어떻게 동작하나

가장 중요한 흐름은 `인덱싱`과 `질의` 두 가지다.

### 1. 인덱싱

새 프로젝트를 넣으면 대체로 아래 순서로 처리된다.

1. 경로를 받는다
2. 지원하는 확장자의 파일을 모은다
3. `.cgcignore`와 기본 제외 디렉터리 규칙을 적용한다
4. tree-sitter로 언어별 파서를 실행한다
5. 파일과 함수와 클래스와 변수와 모듈 정보를 추출한다
6. 호출 관계와 상속 관계를 연결한다
7. 결과를 그래프 DB에 저장한다

현재 코드는 Python, JavaScript, TypeScript, Go, Java, C, C++, Rust, Ruby, PHP, Kotlin, Scala, Swift, Dart, Perl, Haskell, C# 같은 여러 언어를 다룬다.

핵심은 "코드를 복사해 저장"하는 것이 아니라, 코드를 구조 정보로 변환해 저장한다는 점이다.

### 2. 질의

그래프가 만들어진 뒤에는 CLI나 MCP를 통해 질의한다.

- 이름으로 함수 찾기
- 특정 함수의 호출자 찾기
- 두 함수 사이 호출 체인 찾기
- 상속 트리 보기
- 복잡한 함수 목록 뽑기
- 데드코드 후보 찾기

즉 사용자는 원문 코드를 다시 처음부터 훑지 않고도, 그래프를 기준으로 코드 구조를 질문할 수 있다.

### 3. 감시

프로젝트를 감시 모드로 두면 파일 변경이 발생할 때 그래프도 갱신된다.

- 파일 변경 감지
- 해당 파일 다시 파싱
- 전역 import 맵 갱신
- 호출 관계와 상속 관계 다시 연결

이 과정 덕분에 정적인 분석 도구에 그치지 않고 개발 중인 코드에도 맞춰진다.

## 일반적인 코드 검색 방식과 어떻게 다른가

| 항목 | 일반적인 문자열 검색 | CodeGraphContext |
| --- | --- | --- |
| 기본 단위 | 텍스트 줄과 파일 | 함수와 클래스와 파일과 관계 |
| 주요 관심사 | 이름이나 패턴 찾기 | 구조와 호출 관계와 의존성 이해 |
| 결과 형태 | 파일 경로와 매칭 문자열 | 그래프 기반 분석 결과 |
| AI 연동 | 별도 연결 필요 | MCP 서버로 바로 연결 가능 |
| 변경 반영 | 다시 검색해야 함 | watch 모드로 자동 갱신 가능 |
| 고급 분석 | 직접 추적 필요 | 호출자, 상속, 복잡도, 데드코드 지원 |

CodeGraphContext는 `grep`를 대체하는 모든 것이라기보다, "큰 코드베이스를 구조적으로 이해해야 할 때" 더 잘 맞는 도구라고 보는 편이 정확하다.

## 저장소는 어떻게 구성돼 있나

처음 보는 사람은 저장소를 다섯 부분으로 나눠 이해하면 된다.

### `src/codegraphcontext/cli`

터미널 명령의 진입점이다.

- `cgc index`
- `cgc find`
- `cgc analyze`
- `cgc watch`
- `cgc bundle`
- `cgc mcp start`

사용자 입장에서 가장 먼저 닿는 층이다.

### `src/codegraphcontext/server.py`

MCP 서버 진입점이다.

이 파일은 도구 정의를 읽고, 요청이 들어오면 각 핸들러에 연결해 준다. AI 코딩 도구와 붙일 때 핵심이 되는 층이다.

### `src/codegraphcontext/tools/graph_builder.py`

프로젝트의 핵심 가치가 많이 들어 있는 파일이다.

- 파일 수집
- tree-sitter 파싱
- 그래프 스키마 준비
- 호출 관계 연결
- 상속 관계 연결

즉 코드에서 그래프를 만드는 파이프라인의 중심이다.

### `src/codegraphcontext/tools/code_finder.py`

그래프에서 답을 꺼내는 파일이다.

- 이름 검색
- 내용 검색
- 호출자와 피호출 함수 분석
- 상속 분석
- 모듈 의존성 분석
- 변수 범위 분석
- 복잡도와 데드코드 분석

즉 저장된 그래프를 실제 사용자 질문으로 바꾸는 층이다.

### `src/codegraphcontext/core`

실행 기반을 맡는 층이다.

- DB 선택과 연결
- KùzuDB와 FalkorDB와 Neo4j 구현
- 잡 관리
- 파일 감시
- 번들 입출력

이 층이 있어야 위의 파서와 검색 로직이 실제 환경에서 돌아간다.

## 실제로는 어떻게 쓰나

처음 접한 사람은 아래 흐름으로 이해하면 가장 쉽다.

1. 프로젝트를 인덱싱한다
2. 함수와 클래스와 호출 관계가 그래프 DB에 저장된다
3. CLI나 MCP로 질문한다
4. 문자열 검색보다 구조적인 답을 받는다

예시 명령

```bash
cgc index .
cgc find name process_payment
cgc analyze callers process_payment
cgc analyze tree UserService
cgc analyze dead-code
cgc mcp start
```

이 흐름만 이해해도 이 프로젝트의 핵심 사용 방식은 거의 잡은 셈이다.

## 누가 이 프로젝트를 좋아할 가능성이 큰가

아래 같은 문제를 가진 팀이나 개발자라면 CodeGraphContext가 잘 맞을 수 있다.

- 큰 저장소를 빠르게 이해해야 하는 개발자
- AI 코딩 도구가 로컬 코드 구조를 더 정확히 알길 원하는 팀
- 함수 호출 관계나 상속 구조를 자주 추적하는 백엔드 팀
- 데드코드와 복잡도 같은 정적 분석 결과가 필요한 팀
- 여러 프로젝트를 그래프 형태로 관리하고 싶은 팀

특히 "AI가 내 코드베이스를 실제 구조 기준으로 이해했으면 좋겠다"는 요구와 잘 맞는다.

## 반대로 주의할 점

CodeGraphContext는 강력하지만, 모든 상황에서 가장 단순한 해법은 아니다.

- 작은 스크립트 몇 개만 보는 경우에는 일반 검색이 더 빠를 수 있음
- 처음 인덱싱할 때는 시간과 저장 공간이 필요함
- 그래프 DB와 파서 의존성이 필요함
- DB 종류에 따라 지원 기능과 검색 품질 차이가 있음
- 전체 소스 저장 여부와 변수 인덱싱 여부 같은 설정에 따라 결과가 달라질 수 있음
- 테스트나 실행 환경에 필요한 Python 패키지가 빠지면 바로 동작하지 않을 수 있음

즉 CodeGraphContext는 "코드 구조 이해" 문제를 풀 때 강하지만, 그만큼 설정과 운영 요소도 함께 따라온다.

## 이 프로젝트를 한 문장으로 다시 말하면

CodeGraphContext는 로컬 코드베이스를 그래프로 인덱싱해 사람이든 AI든 함수와 클래스와 호출 관계를 더 정확하게 이해하게 돕는 코드 구조 분석 엔진이다.
