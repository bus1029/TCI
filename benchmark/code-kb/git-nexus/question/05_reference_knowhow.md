# GitNexus 개발 노하우 참고 질문

이 문서는 지식 베이스 구축 시 GitNexus 제품의 핵심 동작 방식과 강점을 파악하고, 우리 제품에 적용할 수 있는 요소들을 정리한 질문 모음입니다.

## 1. 지식 모델링과 고유성 보장 방안

- **Q. 지식의 최소 단위는 무엇이며, 여러 언어의 차이를 공통 스키마로 정규화하는가**
  - **참고할 노하우**: 파일보다 작은 명확한 심벌 단위로 로직을 분할함
  - 언어별 차이를 공통 스키마로 정규화하면서도 고유 특징을 유지하는 설계 확인 필요
- **Q. 어떤 관계를 저장하는가**
  - **참고할 노하우**: 호출, 상속, 임포트 등을 쪼개지 않고 단일 테이블 속성으로 통합 보관함
  - 그래프 질의와 관리를 극도로 단순화하는 모델링 확인 추천
- **Q. 유일성은 어떻게 보장하는가**
  - **참고할 노하우**: 중복 생성을 막기 위해 멱등성 있는 식별자 생성과 맵을 활용함
  - 점진적 갱신 시 오작동을 방지하는 키 설계 기법 분석 요망

## 2. 고강도·다단계 인덱싱 파이프라인

- **Q. 파싱은 어떤 기술을 사용하는가**
  - **참고할 노하우**: 정규식이 아닌 트리시터(tree-sitter) 기반 AST 파싱을 채택함
  - 워커 스레드를 이용한 병렬 파싱으로 속도와 정확도를 높이는 로직 참조 가능
- **Q. 인덱싱이 몇 단계로 나뉘는가**
  - **참고할 노하우**: 스캔부터 파생 모델 추출과 임베딩까지 세분화된 10단계 파이프라인을 구축함
  - 메모리 사용 한계를 제어하면서 대량의 파일을 처리하는 흐름 확인 유용

## 3. 검색 엔진 및 저장소 아키텍처

- **Q. 어떤 저장소를 사용하며, 다른 그래프DB 대신 KùzuDB를 선택한 이유는 무엇인가**
  - **참고할 노하우**: 임베디드 그래프DB를 사용해 외부 인프라 종속성을 제거함
  - KùzuDB 특유의 쿼리 성능과 로컬 통합성을 활용하는 아키텍처 파악 권장
- **Q. 지식 베이스는 하이브리드형인가, 벡터는 무엇을 보조하는가**
  - **참고할 노하우**: 구조적 탐색은 그래프로 처리하고 의미 검색은 임베딩 벡터로 보완함
  - 두 검색 기술의 결합 포인트 차용 가능
- **Q. 다중 저장소 또는 모노레포를 지원하는가**
  - **참고할 노하우**: 로컬 디렉터리에 데이터베이스 폴더를 두고 전역 레지스트리로 통합 제공함
  - 단일 서버 프로세스가 여러 워크스페이스를 유연하게 바라보는 구조 참조 요망

## 4. 확장성과 컨텍스트 제어

- **Q. 대형 저장소에서 인덱싱 시간과 질의 시간을 어떻게 최적화하는가**
  - **참고할 노하우**: 청크 단위 파일 읽기 및 AST 캐싱 기법을 적용함
  - 연결 풀 유지와 필요 시점에 여는 지연 연결 기법 연구 필요
- **Q. 결과 수 제한과 랭킹(우선순위) 기준은 어떻게 관리하는가**
  - **참고할 노하우**: 검색 결과 간 컨텍스트 폭발을 막기 위해 입력 하드캡과 BM25 결합 점수를 사용함
  - 프로세스와 심벌 중요도에 가중치를 부여하는 RRF 랭킹 계산식 참고 가치 높음

## 5. 지능형 에이전트 도구 인터페이스

- **Q. 검색용 도구와 분석용 도구가 분리되어 있는가**
  - **참고할 노하우**: 전역 검색 기능과 심층 영향도 분석 기능을 철저히 분리함
  - 에이전트의 환각을 줄이고 목적에 맞게 도구를 유도하는 계층적 API 설계 관찰 유용
- **Q. 원시 쿼리 탈출구(Escape Hatch)가 있는가**
  - **참고할 노하우**: 기본 도구 제약을 넘어가기 위해 벡터 검색과 직접적인 Cypher 질의 결합 기능 제공
  - 복잡한 구조 질의를 우회 처리하는 고급 도구 열어두기 전략 확인 요망
- **Q. 시스템 프롬프트나 권장 도구 사용 순서 가이드가 제공되는가**
  - **참고할 노하우**: 탐색부터 검증까지 권장 워크플로우를 문서를 통해 LLM에 직접 주입함
  - 스킬 파일이나 초기 가이드라인을 강제하여 에이전트를 유도하는 설계법 참조 필수
