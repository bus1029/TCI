# 5. GitNexus 지능형 에이전트 도구 인터페이스 분석

## 도입 개요
GitNexus의 MCP 인터페이스는 그래프 데이터베이스를 바깥에 그대로 노출하는 얇은 래퍼가 아닙니다. 현재 구현 기준으로 보면 이 계층의 핵심은 세 가지입니다. 첫째, 도구 설명과 응답 본문에 다음 행동 힌트를 넣어 에이전트의 도구 사용 순서를 유도합니다. 둘째, `grep`이나 원시 Cypher만으로는 얻기 어려운 실행 흐름, 심벌 문맥, 변경 영향, 이름 변경 미리보기 같은 고수준 분석 결과를 직접 반환합니다. 셋째, 원시 질의와 파일 수정처럼 위험한 경로에는 읽기 전용 제약, 동명 심벌 해소, 미리보기 기본값 같은 안전장치를 겹쳐 둡니다.

# 시스템 핵심 동작 방식 및 사용 기술

## 1. 도구 설명 자체를 사용 순서 가이드로 쓰는 구조

### 채택 기술 구조
- MCP 도구 설명을 단순 파라미터 목록이 아니라 사용 순서 가이드로 활용하는 구조
- 주요 도구 설명에 `WHEN TO USE`, `AFTER THIS` 문구를 넣어 다음 도구 선택을 유도하는 방식
- 모델의 내부 추론 제어보다 도구 표면에서 안전한 탐색 순서를 제안하는 설계
- `context`, `rename` 등에 `uid`, `file_path`, `repo` 파라미터를 두어 다중 레포와 동명 심벌 상황을 줄이는 인터페이스

### 코드 근거 예시
- `gitnexus/src/mcp/tools.ts`
  - `list_repos` 뒤에 `gitnexus://repo/{name}/context` 리소스 조회를 안내하는 설명
  - `query` 뒤에 `context()` 호출을 권장하는 흐름 정의
  - `context` 뒤에 `impact()` 호출이나 process 리소스 조회로 이어지게 하는 설명
  - `rename` 뒤에 `detect_changes()` 실행을 유도하는 미리보기 기본 설명

### 제품 적용 포인트
- 에이전트용 도구 설계 시 스키마 정확성만으로는 부족하다는 점
- 실제 사용 순서가 중요한 도구에서 "언제 쓰는가"와 "다음에 무엇을 해야 하는가"를 함께 적는 방식
- 검색 도구, 문맥 조회 도구, 변경 영향 도구가 함께 있을 때 권장 호출 순서를 설명문에 고정하는 전략

### 해석과 시사점
- 에이전트가 한 번의 도구 호출 후 멈추는 문제를 설명문 수준에서 완화하려는 설계
- 구현 비용이 낮고 MCP 표준과도 잘 맞지만 강제 규칙이 아닌 힌트 중심 방식
- 설명문 단독보다 응답 후속 힌트와 고수준 결과 구조가 함께 있어야 효과가 커진다는 점

## 2. 저수준 조회 대신 실행 흐름 중심 분석 도구를 제공하는 구조

### 채택 기술 구조
- `query`에서 BM25 검색과 의미 검색을 함께 돌린 뒤 RRF로 병합하는 구조
- 병합된 심벌 결과를 다시 `Process`, `Community`에 연결해 `processes`, `process_symbols`, `definitions`로 재구성하는 방식
- `context`에서 특정 심벌 기준으로 들어오는 참조, 나가는 참조, 실행 흐름 참여 정보를 분류하는 구조
- `impact`에서 관계 그래프를 너비 우선으로 확산해 깊이별 영향과 위험도를 계산하는 방식
- `rename`에서 그래프 기반 참조 수집 뒤 `rg` 기반 텍스트 검색으로 누락 참조를 보강하는 구조

### 코드 근거 예시
- `gitnexus/src/mcp/local/local-backend.ts`
  - `query()`에서 `bm25Search()`와 `semanticSearch()`를 병렬 실행하고 RRF 점수로 병합한 뒤 `Process`, `Community` 기반으로 결과를 재구성하는 흐름
  - `context()`에서 `calls`, `imports`, `extends`, `implements`별 분류 반환
  - `impact()`에서 깊이별 영향 심벌 수집 후 `affected_processes`, `affected_modules`, `risk` 계산
  - `rename()`에서 `context()` 재사용 후 그래프 기반 참조와 ripgrep 기반 참조를 분리 표시하는 구현

### 제품 적용 포인트
- 에이전트에게 원시 검색만 주면 이후 해석과 연결을 모델이 떠안게 된다는 점
- 실행 흐름, 문맥, 변경 영향, 이름 변경 미리보기를 개별 도구로 승격하는 방식
- 코드 수정 도구 도입 시 `impact`, `rename` 미리보기 같은 중간 분석 도구를 별도 API로 두는 전략

### 해석과 시사점
- 그래프 자체보다 그래프를 에이전트가 이해하기 쉬운 결과 구조로 압축해 주는 인터페이스의 강점
- `query -> context -> impact/rename` 흐름으로 실제 개발 절차를 MCP 도구 단위로 옮긴 형태
- 그래프 정확도뿐 아니라 반환 스키마 설계에도 크게 좌우되는 계층 품질

## 3. 응답 뒤 다음 행동을 붙여 스스로 이어가게 만드는 구조

### 채택 기술 구조
- 도구 설명에만 의존하지 않고 실제 도구 응답 뒤에도 다음 행동 힌트를 붙이는 구조
- `query`, `context`, `rename` 등 도구별로 다른 후속 행동 힌트를 생성하는 방식
- `detect_impact`, `generate_map` 같은 MCP 프롬프트로 반복 분석 절차를 재사용하는 구조
- 정적 도구 설명, 동적 응답 힌트, 재사용 가능한 프롬프트를 함께 두는 다층 보정 방식

### 코드 근거 예시
- `gitnexus/src/mcp/server.ts`
  - `getNextStepHint()`에서 도구 이름별 후속 행동 텍스트 생성
  - `CallToolRequestSchema` 핸들러에서 실제 도구 결과 뒤에 힌트를 붙여 반환하는 방식
  - `ListPromptsRequestSchema`, `GetPromptRequestSchema`에서 `detect_impact`, `generate_map` 프롬프트 노출
  - `detect_impact` 프롬프트에서 `detect_changes -> context -> impact` 순서를 단계별로 지정하는 구성

### 제품 적용 포인트
- 시스템 프롬프트만 길게 쓰는 방식보다 실제 도구 응답 시점에 다음 행동을 붙이는 방식
- 반복 조사 절차를 MCP 프롬프트로 굳혀 프롬프트 중복을 줄이는 전략
- 도구 설명, 응답 힌트, 프롬프트를 서로 다른 책임으로 나누는 유지보수 방식

### 해석과 시사점
- 에이전트가 어떤 순서로 움직여야 하는지를 실행 중 계속 상기시키는 구조
- 같은 도구라도 호출 직후의 다음 단계가 정적 문서보다 실행 가능성이 높다는 점
- 설명문, 힌트, 프롬프트가 서로 어긋나면 혼란이 커질 수 있어 세 계층을 함께 관리해야 한다는 점

## 4. 원시 질의와 파일 수정에 안전장치와 동명 해소를 넣는 구조

### 채택 기술 구조
- `cypher`를 읽기 전용 전제로 두고 문자열 수준에서 쓰기 키워드를 한 번 더 차단하는 구조
- `nodeId`에서 추출한 라벨을 `VALID_NODE_LABELS` 집합으로 검증한 뒤 동적 Cypher를 만드는 방식
- 동명 심벌이 여러 개면 임의 선택 대신 `ambiguous` 상태와 후보 목록을 반환하는 `context` 설계
- `dry_run: true`를 기본값으로 두고 `assertSafePath()`로 경로 이탈을 막는 `rename` 구조
- 이름 변경 편집안을 `graph`, `text_search` 신뢰도 태그로 구분해 반환하는 방식

### 코드 근거 예시
- `gitnexus/src/mcp/local/local-backend.ts`
  - `CYPHER_WRITE_RE`로 `CREATE`, `DELETE`, `SET`, `MERGE`, `REMOVE`, `DROP`, `ALTER`, `COPY`, `DETACH` 차단
  - `cypher()`에서 쓰기 질의를 거부하고 지식 그래프가 읽기 전용임을 명시하는 응답
  - `semanticSearch()`에서 `VALID_NODE_LABELS` 대조로 동적 라벨 주입 위험 축소
  - `context()`에서 `uid`, `filePath`, `line`이 포함된 후보 목록 반환
  - `rename()`에서 편집 목록 미리보기 후 `dry_run`이 꺼졌을 때만 실제 파일 적용
- `gitnexus/src/server/api.ts`
  - 기본 바인딩을 `127.0.0.1`로 두고 CORS를 localhost와 배포 UI 도메인으로 제한하는 설정

### 제품 적용 포인트
- 원시 질의 인터페이스를 열 때 데이터베이스 수준 읽기 전용 정책과 질의 수준 필터를 함께 두는 방식
- 동명 심벌이 많은 코드베이스에서 `uid`, `file_path`처럼 대상을 좁히는 인터페이스의 필요성
- 수정 도구에 미리보기 기본값과 신뢰도 태그를 함께 두는 검토 중심 흐름

### 해석과 시사점
- 에이전트를 완전히 신뢰하지 않고 인터페이스 계층에서 오작동 가능성을 줄이려는 설계
- 동명 해소와 미리보기 기본값으로 LLM의 그럴듯한 첫 답 문제를 직접 겨냥한 방식
- 안전장치가 있어도 수정 적용의 정밀도까지 완전히 보장되지는 않으므로 후속 검증 도구가 필요하다는 점

## 5. 한계와 trade-off

### 현재 구현 기준에서 주의할 점
- `query` 스키마의 `task_context`, `goal`과 실제 `LocalBackend.query()` 구현 사이의 불일치
- `detect_changes` 설명과 달리 현재 구현이 `git diff --name-only` 기반 파일 단위 분석에 더 가깝다는 점
- `impact()`가 `context()` 수준의 강한 동명 해소 인터페이스를 아직 제공하지 않는 상태
- `rename()`에서 줄 단위 미리보기와 파일 전체 정규식 치환 적용 사이의 차이
- `cypher` 보호가 문자열 필터 기반이라 복잡한 질의 검증기 수준의 보장은 아니라는 점

### 제품 해석
- 정밀한 정적 분석기보다 실전에서 에이전트가 쓸 수 있는 분석 도구 세트에 가까운 인터페이스 성격
- 완전한 정확성보다 사용성, 단계적 탐색, 안전한 기본값을 우선한 설계 방향
- 로컬 코드 인텔리전스 도구로서는 실용적이지만 문서가 구현보다 더 강하면 실제 능력을 과장하게 된다는 점

# 적용 인사이트
- 에이전트 도구 인터페이스를 검색 API 모음이 아니라 작업 순서를 설계하는 계약으로 보는 관점
- 도구 설명, 응답 후속 힌트, 재사용 프롬프트를 분리해 모델의 도구 사용 흐름을 안정화하는 방식
- 원시 검색보다 `context`, `impact`, 이름 변경 미리보기 같은 해석된 중간 산출물을 직접 반환하는 전략
- 수정 도구 설계 시 동명 해소, 미리보기 기본값, 신뢰도 태그, 읽기 전용 질의 방어를 함께 두는 원칙
- 실제 구현이 파일 단위인지 심벌 단위인지, 미리보기와 실제 적용이 어떻게 다른지까지 코드 기준으로 분리해 적는 벤치마킹 문서 작성 원칙
