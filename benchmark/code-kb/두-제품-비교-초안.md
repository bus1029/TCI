# 한 줄 결론

- 지금 단계의 권장 출발점은 `CodeGraphContext`의 심벌 중심 그래프와 편집 좌표 계약을 중심축으로 두고, `GitNexus`의 대형 레포 처리와 하이브리드 검색 구조를 보강 요소로 흡수하는 방향이다
- `GitNexus`는 장기 확장 방향으로 매우 중요하지만 초기 중심축으로 전부 가져오기에는 인덱싱·운영 계층이 더 무겁다

# 권장 방향

- 1단계: 함수, 클래스, 파일, 호출 관계 같은 핵심 심벌 계약과 좌표 메타데이터를 먼저 고정
- 2단계: 에이전트 도구를 `query`, `context`, `impact` 수준으로 설계하되 읽기 전용 안전장치를 우선 적용
- 3단계: 로컬 인덱스와 기본 FTS를 먼저 붙이고 대형 레포에서는 청크 파싱과 결과 상한으로 확장성 확보
- 4단계: 이후 `Process`(실행 흐름 단위 파생 노드), `Community`(기능 영역 단위 파생 노드), semantic 검색, 멀티레포 레지스트리(여러 레포 인덱스를 한곳에서 발견하는 계층)를 점진적으로 도입

# 지금 하지 말아야 할 것

- 초기 단계에서 `Process`(실행 흐름 단위 파생 노드), `Community`(기능 영역 단위 파생 노드) 같은 상위 추론 레이어를 핵심 데이터 모델로 먼저 고정하지 말 것
- `path`, `line_number`, 심벌 식별자 없이 의미 검색 중심 구조부터 앞세우지 말 것
- 멀티레포 전역 레지스트리와 연결 풀을 초기 필수 범위로 두지 말 것
- 고비용 semantic 계층을 기본 기능처럼 전제하지 말 것

# 의사결정 필요 사항

- 우리 제품의 1차 목표를 코드 검색 보조로 볼지, 리팩터링과 영향 분석까지 수행하는 에이전트 기반 도구로 볼지 결정
- 지식의 기본 단위를 파일보다 작은 심벌 중심으로 고정할지 결정
- 로컬 단일 레포 우선으로 갈지 멀티레포 통합 질의를 초기에 포함할지 결정
- 검색 결과를 편집 가능한 좌표로 반환할지, 프로세스·모듈 단위 요약으로 먼저 반환할지 결정

# 이 문서가 답하려는 질문

- 두 제품은 각각 어떤 코드 지식 문제를 푸는가
- 우리 제품은 어느 설계 축을 중심으로 삼아야 하는가
- 지금 바로 가져올 요소와 나중에 도입할 요소는 무엇인가
- 현재 단계에서 과한 복잡도는 무엇인가

# 문서 목적

이 문서는 `CodeGraphContext`와 `GitNexus`를 비교해 우리 제품이 어떤 코드 지식 표현 방식, 인덱싱 구조, 검색 방식, 에이전트 도구 인터페이스를 선택해야 하는지 판단하기 위한 기준을 정리한다.

핵심 비교 기준은 기능 개수보다 `코드를 무엇으로 저장하는가`, `관계를 어떻게 유지하는가`, `에이전트가 어떤 결과를 실제 작업에 쓸 수 있는가`에 둔다.

# 제품별 한 줄 정의

- `CodeGraphContext`는 심벌 그래프와 MCP 도구를 결합해 코드 분석 에이전트가 바로 쓸 수 있게 만든 로컬 우선 코드 그래프 도구다
- `GitNexus`는 레포 로컬 인덱스, Kuzu 기반 검색, 프로세스 수준 추론(실행 흐름 수준 추론)을 결합해 대형 코드베이스 탐색을 돕는 코드 지식 베이스다

# 이 문서를 읽는 기준

두 제품은 모두 코드 지식을 구조화해 다시 꺼내 쓰게 만들지만 우선순위가 다르다. `CodeGraphContext`는 심벌 관계와 에이전트 도구 표면을 더 직접적으로 다루고, `GitNexus`는 대형 저장소 인덱싱과 검색 결과 압축, 멀티레포 운영에 더 많은 설계를 투자한다.

따라서 이 문서는 어느 제품이 더 좋다고 결론내리기보다, 각 제품이 어떤 코드 지식 문제를 어떤 데이터 모델과 운영 방식으로 푸는지 비교하는 데 초점을 둔다.

# 처음 읽는 사람을 위한 핵심 개념

이 문서에는 두 제품에서만 자주 쓰이는 내부 개념이 나온다. 아래 뜻만 먼저 잡고 읽으면 뒤 비교표와 상세 비교를 훨씬 쉽게 따라갈 수 있다.

## `CodeGraphContext`에서 자주 나오는 말

- `uid`: 심벌의 이름, 파일 경로, 줄 번호를 묶어 만든 안정 식별자
- 편집 좌표: `path`, `line_number`, `end_line`처럼 검색 결과를 실제 수정 위치로 연결하는 정보
- `.cgc` 번들: 인덱싱된 코드 그래프를 내보내고 다시 불러오는 스냅샷 형식
- read-only Cypher fallback: 일반 도구로 부족할 때 쓰는 읽기 전용 원시 그래프 질의

## `GitNexus`에서 자주 나오는 말

- `CodeRelation`: 여러 관계를 하나의 공통 관계 구조로 묶고 `type`으로 구분하는 방식
  - `confidence`: 이 관계를 얼마나 확실하게 봤는지 나타내는 신뢰도
  - `reason`: 이 관계가 왜 만들어졌는지 보여 주는 근거
  - `step`: `Process` 안에서 몇 번째 단계에 놓이는지 나타내는 순서 정보
- `Process`: 여러 심벌을 실행 흐름 단위로 묶은 파생 노드
- `Community`: 밀접하게 연결된 심벌 묶음을 기능 영역처럼 묶은 파생 노드
- `.gitnexus/`: 각 레포 안에 만드는 로컬 인덱스 디렉터리
- `registry.json`: 여러 레포 인덱스 위치를 모아두는 전역 발견 파일
- `impact`: 특정 심벌 변경이 어디까지 퍼지는지 보여주는 영향 분석 결과
- `rename`: 실제 수정 전에 이름 변경 후보를 검토하는 도구
- `dry_run`: 실제 파일 수정 없이 미리보기만 수행하는 모드

# 축별 상세 비교

아래 비교는 제품의 핵심 설계 차이를 기준으로 정리했다.

## 1. 비교 대상과 전제

두 제품은 모두 코드 지식을 저장하고 다시 검색하게 만들지만, 출발점이 다르다.

- 공통점: 로컬 우선 코드 지식 베이스를 만들고 에이전트나 사용자가 다시 활용하게 함
- 차이점: `CodeGraphContext`는 심벌 관계와 도구 호출 표면을, `GitNexus`는 인덱싱 확장성과 검색 경험을 더 강하게 최적화함
- 읽는 기준: 기능 우열보다 `무엇을 지식의 최소 단위로 보는가`, `관계를 어떻게 유지하는가`, `어떤 응답이 실제 작업으로 이어지는가`를 중심으로 비교해야 함

## 2. 제품 정체성

두 제품은 모두 코드 지식을 다루지만, 제품이 자신을 규정하는 방식부터 다르다.

### `CodeGraphContext`

- 중심 문제: 코드 심벌과 관계를 그래프로 만들고 에이전트가 이를 직접 질의하게 함
- 강점: 심벌 중심 구조, 구조 질의, MCP 도구 표면, 편집 좌표 중심 응답
- 한계: 대형 레포 확장성과 멀티레포 운영은 상대적으로 단순한 편

### `GitNexus`

- 중심 문제: 대형 코드베이스를 인덱싱하고 검색 결과를 에이전트가 소비하기 좋은 형태로 압축함
- 강점: 레포 로컬 인덱스, FTS 중심 검색, 프로세스·커뮤니티 추론(실행 흐름·기능 영역 수준의 파생 추론), 멀티레포 운영
- 한계: 초기 제품의 중심축으로 가져가기에는 파이프라인과 운영 계층이 더 무거움

### 비교 해석

- `CodeGraphContext`는 코드 분석 에이전트에 더 가깝다
- `GitNexus`는 코드 탐색과 검색 운영 제품에 더 가깝다

### 우리 제품 시사점

- 코드 수정과 영향 분석까지 잇는 제품이면 `CodeGraphContext` 축이 더 중요함
- 대형 저장소 탐색과 검색 경험이 먼저면 `GitNexus` 축이 더 중요함

## 3. 지식 표현 단위

두 제품의 차이는 `무엇을 코드 지식의 최소 단위로 보느냐`에서 크게 갈린다.

### `CodeGraphContext`

- 기본 단위: 함수, 클래스, 변수, 인터페이스, 파라미터, import
- 모델링 방식: 언어별 파싱 결과를 공통 심벌 계약으로 정규화
- 검색 대상: 심벌과 관계, 심벌의 문맥 메타데이터

### `GitNexus`

- 기본 단위: 파일, 폴더, 함수, 클래스, 메서드, 인터페이스, 구조체
- 파생 단위: `Process`(실행 흐름 단위 파생 노드), `Community`(기능 영역 단위 파생 노드)
- 검색 대상: 파일 구조, 심벌 구조, 상위 추론 레이어

### 비교 해석

- `CodeGraphContext`는 심벌 자체를 가장 중요한 지식 단위로 본다
- `GitNexus`는 파일 구조와 심벌 구조를 함께 보존하고 그 위에 상위 추론 레이어를 얹는다

### 우리 제품 시사점

- 초기 지식 모델은 `CodeGraphContext`처럼 심벌과 좌표를 먼저 고정하는 편이 안전함
- 프로세스와 커뮤니티 같은 상위 단위는 `GitNexus`처럼 후처리 레이어로 추가하는 편이 현실적임

## 4. 관계 모델과 고유성 보장

관계를 어떻게 저장하고 중복 생성을 어떻게 막는지도 두 제품의 성격을 갈라놓는다.

### `CodeGraphContext`

- 관계 구조: `CALLS`, `IMPORTS`, `CONTAINS`, `HAS_PARAMETER`, `INHERITS` 같은 타입별 관계 유지
- 고유성 방식: `name + path + line_number` 기반 식별을 `uid`(안정 식별자) 단일 키로 흡수
- 특징: 관계의 의미가 스키마에서 바로 드러나고 구조 질의 표현력이 높음

### `GitNexus`

- 관계 구조: `CodeRelation`(공통 관계 인터페이스) 하나에 `type`, `confidence`(신뢰도), `reason`(생성 근거), `step`(프로세스 안의 순서) 속성으로 통합
- 고유성 방식: 사람이 읽을 수 있는 결정적 문자열 ID(같은 입력이면 항상 같은 ID) 사용
- 특징: 도구 표면과 질의 표면이 단순해지고 관계 메타데이터를 일관되게 붙이기 쉬움

### 비교 해석

- `CodeGraphContext`는 관계 의미를 스키마 수준에서 드러내는 쪽에 가깝다
- `GitNexus`는 관계 인터페이스를 단순화하고 메타데이터를 풍부하게 넣는 쪽에 가깝다

### 우리 제품 시사점

- 코드 분석 정확도와 구조 질의가 중요하면 `CodeGraphContext`식 관계 모델이 유리함
- 저장 엔진 제약을 줄이고 도구 표면을 단순화하고 싶다면 `GitNexus`식 공통 관계 인터페이스가 유리함
- 현실적인 절충안은 논리 모델에서는 관계 의미를 유지하고 저장 계층에서는 통합 관계 테이블을 검토하는 방식이다

## 5. 원본 저장소와 인덱스 구조

두 제품은 저장 구조에서도 철학 차이가 뚜렷하다.

### `CodeGraphContext`

- 저장 기본값: 로컬 그래프 저장소
- 백엔드 전략: KùzuDB, FalkorDB Lite, Neo4j 같은 선택적 백엔드 추상화
- 설계 의도: 로컬 우선 경로를 유지하면서 조직 환경에 따라 원격 백엔드로 확장

### `GitNexus`

- 저장 기본값: 레포 루트 아래 `.gitnexus/` 로컬 인덱스
- 통합 전략: `~/.gitnexus/registry.json`으로 여러 레포를 통합 발견
- 설계 의도: 저장은 분산하고 discovery만 통합해 멀티레포 경험 제공

### 비교 해석

- `CodeGraphContext`는 저장소 백엔드 교체에 더 열려 있다
- `GitNexus`는 레포 로컬 저장과 멀티레포 운영의 결합이 더 선명하다

### 우리 제품 시사점

- 초기 제품은 `GitNexus`처럼 레포 가까이에 인덱스를 두는 구조가 운영상 단순함
- 장기적으로 엔터프라이즈 확장을 염두에 둔다면 `CodeGraphContext`식 백엔드 추상화도 함께 검토할 가치가 있음

## 6. 인덱싱과 갱신 모델

두 제품은 코드를 최신 상태로 반영하는 방식에서도 차이가 크다.

### `CodeGraphContext`

- 갱신 흐름: watcher -> 파일 단위 교체 -> 저장소 전역 관계 재링크
- 보조 장치: job 상태 추적, `.cgcignore`, `.cgc` 번들(인덱싱된 그래프 스냅샷) export/import
- 특징: 완전한 미세 증분보다 관계 무결성을 우선

### `GitNexus`

- 갱신 흐름: 저장소 스캔 -> 바이트 예산 청크 파싱 -> CSV 스트리밍 -> bulk load
- 보조 장치: AST 캐시, 워커 풀, 임베딩 재사용, 노드 수 상한
- 특징: 파일 단위 실시간 갱신보다 대형 레포 재인덱싱의 안정성과 비용 통제에 초점

### 비교 해석

- `CodeGraphContext`는 개발 중 자주 바뀌는 코드를 따라가는 구조에 가깝다
- `GitNexus`는 대형 저장소를 안정적으로 다시 인덱싱하는 구조에 가깝다

### 우리 제품 시사점

- IDE 보조형 제품이면 `CodeGraphContext`식 watcher와 전역 리링크가 더 적합함
- 레포 분석 배치 작업이 중심이면 `GitNexus`식 청크 파이프라인과 bulk load가 더 적합함
- 초기에는 `CodeGraphContext`식 증분 갱신을, 대형 레포 단계에서는 `GitNexus`식 청크 재인덱싱 경로를 병행하는 전략이 현실적임

## 7. 검색과 문맥 구성

검색은 두 제품 모두 중요하지만, 검색 결과를 어떤 단위로 다시 묶어 주느냐에서 차이가 난다.

### `CodeGraphContext`

- 검색 출발점: 심벌 이름, 코드 내용, 관계 질의
- 문맥 구성: caller, callee, class hierarchy, complexity, parameter 같은 구조 질의 결과
- 마무리 방식: 작업별 DTO와 `summary` 중심 응답

### `GitNexus`

- 검색 출발점: FTS 기본 검색 + 선택적 semantic 검색
- 문맥 구성: RRF 결합, `Process`(실행 흐름 단위 파생 노드), `Community`(기능 영역 단위 파생 노드), `definitions` 단위 재구성
- 마무리 방식: 프로세스 그룹화, 결과 상한, 기본 본문 비포함

### 비교 해석

- `CodeGraphContext`는 구조 질의 자체가 곧 문맥 조립에 가깝다
- `GitNexus`는 검색 결과를 다시 에이전트가 소비하기 쉬운 묶음으로 압축한다

### 우리 제품 시사점

- 코드 수정과 영향 분석에 직결되는 문맥이면 `CodeGraphContext`식 구조 질의가 더 중요함
- 검색 결과가 너무 많아 토큰이 자주 터지는 환경이면 `GitNexus`식 그룹화와 상한 설계가 더 중요함

## 8. 에이전트 인터페이스와 안전장치

두 제품 모두 MCP를 활용하지만, 에이전트 사용 흐름을 설계하는 방식은 다르다.

### `CodeGraphContext`

- 인터페이스 중심: stdio 기반 MCP 서버와 작업별 도구 분리
- 강점: 인덱싱, watch, 검색, 관계 분석, job 조회를 같은 표면에서 제공
- 안전장치: read-only Cypher fallback(읽기 전용 원시 그래프 질의), fail-fast 초기화, 도구별 입력 스키마

### `GitNexus`

- 인터페이스 중심: 도구 설명과 응답 힌트로 권장 사용 순서를 유도
- 강점: `query -> context -> impact -> rename` 흐름이 실제 개발 절차와 가깝다. 즉 검색 -> 문맥 확인 -> 영향 분석 -> 이름 변경 미리보기 순서로 이어진다
- 안전장치: read-only Cypher, 동명 심벌 후보 반환, `dry_run`(실제 수정 없는 미리보기) 기본값, 신뢰도 태그

### 비교 해석

- `CodeGraphContext`는 도구 집합 자체를 넓게 제공하는 쪽에 가깝다
- `GitNexus`는 에이전트가 실수하지 않도록 다음 행동까지 유도하는 쪽에 가깝다

### 우리 제품 시사점

- 운영 도구와 검색 도구를 한 표면에 넣고 싶다면 `CodeGraphContext`가 더 참고할 만하다
- 분석 절차를 단계형 워크플로우로 설계하고 싶다면 `GitNexus`가 더 참고할 만하다
- 초기 제품은 `CodeGraphContext`식 운영 도구 집합 위에 `GitNexus`식 후속 행동 힌트를 얹는 조합이 가장 실용적이다

## 9. 확장성, 운영, 멀티레포

운영 모델을 보면 두 제품이 실제로 어디에 더 강한지 선명해진다.

### `CodeGraphContext`

- 운영 중심: 로컬 개발 환경, 백엔드 교체, 장기 작업 상태 추적
- 확장 장치: backend factory, bundle import/export, ignore 규칙, feature flag
- 특징: 단일 프로세스 환경에서 실용적인 운영성을 우선

### `GitNexus`

- 운영 중심: 대형 레포 메모리 상한 제어, 멀티레포, 읽기 전용 연결 풀
- 확장 장치: 청크 예산, AST 캐시 정리, 워커 타임아웃, 전역 레지스트리
- 특징: 큰 저장소를 안정적으로 다루는 보호 장치가 많음

### 비교 해석

- `CodeGraphContext`는 로컬 분석 도구로서의 단순성과 유연성이 강점이다
- `GitNexus`는 큰 코드베이스와 여러 레포를 버티는 운영 규칙이 강점이다

### 우리 제품 시사점

- 개인 또는 소규모 팀의 IDE 보조 도구면 `CodeGraphContext`식 운영 구조로도 충분할 수 있다
- 조직 단위 멀티레포 탐색까지 염두에 둔다면 `GitNexus`식 레지스트리와 연결 관리가 더 필요하다

## 10. 출처 추적과 편집 가능성

코드형 지식 베이스에서는 출처 추적이 곧 편집 가능성으로 이어진다.

### `CodeGraphContext`

- 중심 장치: `path`, `line_number`, `end_line`, `source`, `context`
- 강점: 검색 결과를 실제 파일 편집으로 바로 연결하기 쉽다
- 성격: 에이전트 편집 정확도를 높이기 위한 좌표 중심 계약

### `GitNexus`

- 중심 장치: `uid`, `filePath`, `line`, 이름 변경 미리보기(`rename`), 영향 분석(`impact`)
- 강점: 동명 심벌 해소와 변경 전 검토 흐름이 비교적 잘 드러난다
- 성격: 검색 결과를 안전한 검토 절차로 연결하는 인터페이스 중심 계약

### 비교 해석

- `CodeGraphContext`는 정밀한 편집 좌표 제공에 강하다
- `GitNexus`는 변경 검토와 영향도 설명에 강하다

### 우리 제품 시사점

- 코드 편집 에이전트라면 `CodeGraphContext`식 좌표 계약을 먼저 고정해야 함
- 이후 실제 수정 도구를 붙일 때는 `GitNexus`식 `impact`(영향 분석), `rename`(이름 변경 미리보기), 미리보기 기본값을 흡수하는 편이 안전하다

## 우리 제품 관점의 해석

두 제품을 종합하면, 우리 제품은 어느 제품을 그대로 복제할지보다 어느 층을 중심축으로 삼을지 먼저 정해야 한다.

### 핵심 판단

- 코드 편집까지 잇는 분석 에이전트가 1차 목표면 `CodeGraphContext` 축이 중심이 된다
- 대형 저장소 탐색과 검색 경험이 1차 목표면 `GitNexus` 축이 중심이 된다
- 현재 단계에서 가장 현실적인 방향은 `CodeGraphContext`를 중심축으로 두고 `GitNexus`를 확장 레이어로 흡수하는 조합이다

### 1. 바로 가져올 수 있는 요소

- `CodeGraphContext`의 공통 심벌 계약과 좌표 메타데이터
- `CodeGraphContext`의 MCP 도구 분리와 read-only fallback 질의(읽기 전용 원시 그래프 질의)
- `GitNexus`의 청크 기반 인덱싱과 bulk load 발상
- `GitNexus`의 결과 상한과 프로세스 단위 응답 압축
- `GitNexus`의 `impact`(영향 분석), `rename`(이름 변경 미리보기), `dry_run`(실제 수정 없는 미리보기) 중심 안전한 수정 흐름

### 2. 조합해서 가져와야 할 요소

- `CodeGraphContext`의 관계 모델과 `GitNexus`의 결과 그룹화
- `CodeGraphContext`의 watcher 기반 갱신과 `GitNexus`의 대형 레포 재인덱싱 경로
- `CodeGraphContext`의 로컬 우선 MCP 서버와 `GitNexus`의 멀티레포 discovery 구조

### 3. 지금 단계에서 과할 수 있는 요소

- 초기 단계에서 `Process`(실행 흐름 단위 파생 노드), `Community`(기능 영역 단위 파생 노드)를 핵심 스키마로 먼저 도입하는 것
- 멀티레포 연결 풀과 전역 레지스트리를 초기 제품의 필수 구조로 강제하는 것
- semantic 검색을 기본 필수 기능으로 전제하는 것
- 여러 그래프 백엔드와 저장 전략을 동시에 완벽하게 추상화하는 것

### 4. 현재 기준의 권장 방향

- 추천 출발점: `CodeGraphContext` + `GitNexus` 일부 패턴
- 이유 1: 코드 편집과 영향 분석으로 이어지려면 `CodeGraphContext`의 심벌 좌표 계약이 먼저 필요함
- 이유 2: 대형 코드베이스를 고려하면 `GitNexus`의 청크 파이프라인과 결과 압축 설계가 곧 필요해질 가능성이 큼
- 확장 방향: 초기에는 심벌 그래프와 MCP 도구에 집중하고, 이후 멀티레포와 프로세스 추론, semantic 검색을 점진적으로 추가하는 편이 좋음
- 보류 판단: 지금 단계에서 `GitNexus` 전체 구조를 통째로 도입하는 것은 복잡도 대비 이득이 작을 수 있음

# 결론

### 핵심 요약

- `CodeGraphContext`는 코드 분석 에이전트 코어에 가깝다
- `GitNexus`는 대형 레포 탐색과 검색 운영 계층에 가깝다
- 두 제품은 경쟁 관계라기보다 서로 다른 층에서 강점을 가진다

### 우리 제품이 지금 내려야 할 판단

- 먼저 심벌 계약과 편집 좌표를 고정해야 함
- 다음으로 갱신 모델을 watcher 중심으로 갈지 재인덱싱 중심으로 갈지 정해야 함
- 그다음에 검색 결과를 구조 질의 중심으로 줄지 프로세스 묶음 중심으로 줄지 정해야 함
- 마지막 단계에서 멀티레포와 semantic 검색, 상위 추론 레이어를 얹는 것이 자연스럽다

### 주의할 점

- 이 순서를 거꾸로 잡으면 검색은 화려해져도 실제 코드 작업 연결성이 약해질 수 있음
- 특히 초기 단계에서 상위 추론 레이어와 멀티레포 운영을 먼저 들이면 핵심 심벌 계약이 흔들릴 가능성이 큼
