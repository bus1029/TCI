# Code KB 두 제품 원페이저

## 목적

이 문서는 내일 상급자 질문에 바로 답할 수 있도록 `CodeGraphContext`와 `GitNexus`의 차이와 우리 팀 시사점을 한 장으로 압축한 요약본이다.

## 한 줄 결론

- 지금 단계의 추천 축은 `CodeGraphContext + GitNexus 일부 패턴`
- 이유는 코드 편집과 영향 분석으로 이어지려면 `CodeGraphContext`의 심벌 중심 그래프와 편집 좌표 계약이 먼저 필요하기 때문
- `GitNexus`는 대형 레포 탐색과 결과 압축 측면에서 매우 중요하지만, 지금 바로 전체 구조를 중심축으로 두기에는 인덱싱과 운영 계층이 더 무겁다

## 제품별 한 줄 정의

| 제품 | 한 줄 정의 | 가장 잘 푸는 문제 |
| --- | --- | --- |
| `CodeGraphContext` | 심벌 그래프와 MCP 도구를 결합한 로컬 우선 코드 분석 도구 | 코드 구조 질의, 호출 관계 분석, 편집 좌표 기반 작업 연결 |
| `GitNexus` | 레포 로컬 인덱스와 실행 흐름 중심 탐색을 결합한 코드 지식 베이스 | 대형 코드베이스 탐색, 영향 분석, 프로세스 단위 결과 압축 |

## 제품별 핵심 특징

### `CodeGraphContext`

- 핵심 정체성: 심벌 그래프 기반 코드 분석 도구
- 강점 1: 함수, 클래스, 파일, 호출 관계를 그래프로 저장해 구조 질의에 강함
- 강점 2: CLI와 MCP를 함께 제공해 개발자와 AI 에이전트가 같은 인덱스를 재사용 가능
- 강점 3: `path`, `line_number`, `end_line` 같은 편집 좌표를 강하게 다뤄 실제 코드 수정으로 연결하기 쉬움
- 우리 팀 적용 포인트: 공통 심벌 계약, 관계 타입 모델, 편집 좌표 메타데이터, read-only fallback 질의
- 한계: 대형 레포 운영, 멀티레포 discovery, 결과 압축 계층은 상대적으로 단순한 편

### `GitNexus`

- 핵심 정체성: 대형 레포 탐색과 영향 분석에 강한 코드 지식 베이스
- 강점 1: `query -> context -> impact -> rename` 흐름으로 실제 개발 절차에 가까운 워크플로우를 제공함
- 강점 2: `Process`, `Community`, 결과 상한, 프로세스 그룹화로 검색 결과를 에이전트가 소비하기 쉽게 압축함
- 강점 3: `.gitnexus/` 로컬 인덱스와 `registry.json` 기반 멀티레포 운영 구조를 갖춤
- 우리 팀 적용 포인트: 청크 기반 인덱싱, bulk load 발상, 영향 분석 흐름, `dry_run` 중심 안전장치, 대형 레포 보호 장치
- 한계: 초기 핵심 스키마로 전부 가져오면 상위 추론 레이어와 운영 복잡도가 빠르게 커짐

## 두 제품의 차이

| 비교 축 | `CodeGraphContext` | `GitNexus` |
| --- | --- | --- |
| 지식의 기본 단위 | 함수, 클래스, 변수, 파라미터 같은 심벌 | 파일 + 심벌 + `Process` + `Community` |
| 관계 모델 | `CALLS`, `IMPORTS`, `INHERITS` 같은 타입별 관계 | `CodeRelation` 하나에 `type`, `confidence`, `reason`, `step`를 담는 통합 관계 |
| 검색 철학 | 구조 질의 자체가 문맥 조립에 가까움 | 검색 결과를 실행 흐름과 기능 묶음으로 압축 |
| 갱신 철학 | watcher 기반 갱신과 전역 재링크 | 청크 기반 재인덱싱과 bulk load |
| 편집 연계성 | 좌표 중심 계약이 강함 | 영향 분석과 이름 변경 미리보기 흐름이 강함 |
| 운영 강점 | 로컬 분석 도구의 단순성과 유연성 | 대형 레포와 멀티레포를 버티는 보호 장치 |

## 우리 팀 기준 권장 선택

### 지금 먼저 가져올 것

- `CodeGraphContext`의 공통 심벌 계약
- `CodeGraphContext`의 편집 좌표 메타데이터
- `CodeGraphContext`의 MCP 도구 분리와 read-only fallback 질의
- 호출 관계, 상속 관계, 포함 관계 같은 구조 질의 중심 설계

### 나중에 확장할 것

- `GitNexus`의 청크 기반 대형 레포 인덱싱 경로
- `GitNexus`의 결과 상한과 프로세스 단위 응답 압축
- `GitNexus`의 `impact`, `rename`, `dry_run` 중심 안전한 수정 흐름
- `GitNexus`의 멀티레포 discovery와 연결 관리

### 지금 하지 않을 것

- 초기 단계에서 `Process`, `Community`를 핵심 데이터 모델로 먼저 고정하는 것
- semantic 검색을 기본 필수 기능처럼 전제하는 것
- 멀티레포 전역 레지스트리와 연결 풀을 초기 필수 구조로 강제하는 것
- 여러 그래프 백엔드와 저장 전략을 동시에 완벽하게 추상화하는 것

## 예상 질문과 짧은 답변

### Q. 두 제품 중 무엇이 더 좋나

우열의 문제라기보다 푸는 범위가 다르다. 코드 편집과 영향 분석까지 이어지는 에이전트 도구를 먼저 만들려면 `CodeGraphContext`가 중심축에 더 가깝다.

### Q. 왜 `GitNexus`를 바로 중심으로 쓰지 않나

`GitNexus`는 강력하지만 대형 레포 운영과 상위 추론 레이어까지 함께 가져온다. 지금 단계에서는 심벌 계약과 편집 좌표를 먼저 고정하는 편이 더 안전하다.

### Q. `CodeGraphContext`와 `GitNexus`의 가장 큰 차이는 무엇인가

`CodeGraphContext`는 심벌 구조와 좌표 기반 분석에 강하고, `GitNexus`는 실행 흐름 단위 검색 압축과 영향 분석 워크플로우에 강하다.

### Q. 우리 팀이 바로 벤치마킹해야 할 기술 포인트는 무엇인가

심벌 단위 그래프, 좌표 메타데이터, 구조 질의, read-only 안전장치, 영향 분석 흐름, 결과 상한 설계다.

### Q. 장기적으로 `GitNexus`에서 배워야 할 것은 무엇인가

대형 레포 재인덱싱, 프로세스 단위 결과 묶음, `impact`와 `rename` 중심 검토 절차, 멀티레포 운영 구조다.

## 최종 메시지

지금 우리 팀은 코드 지식 베이스의 기본기를 먼저 세워야 한다. 따라서 출발점은 `CodeGraphContext`의 심벌 그래프와 편집 좌표 계약을 중심으로 잡고, 이후 `GitNexus`의 대형 레포 처리와 결과 압축, 영향 분석 흐름을 확장 레이어로 흡수하는 방향이 적절하다.
