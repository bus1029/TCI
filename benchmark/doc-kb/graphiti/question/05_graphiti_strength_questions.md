# Graphiti 질문에서 참고할만한 개발사항들

## 1. Temporal Context Graph 표현 방식

**Q. 왜 Graphiti는 문서 청크 저장소 대신 `Episode -> Entity -> Fact` 구조를 KB의 기본 단위로 택했는가**

- 참고할 노하우
    - 원문 문서를 그대로 검색하는 방식이 아니라, 에이전트가 재사용하기 좋은 사실 단위로 분해하는 방식
    - 문서형 RAG와 다르게 지식의 최소 표현 단위를 관계와 시간 축으로 설계하는 이유

**Q. Episode를 provenance의 기준점으로 삼는 구조는 어떤 장점을 주는가**

- 참고할 노하우
    - 모든 파생 사실이 어떤 입력에서 나왔는지 역추적할 수 있게 만드는 방식
    - 장기 메모리 시스템에서 설명 가능성과 디버깅 가능성을 높이는 구조

**Q. `CommunityNode`, `SagaNode` 같은 상위 계층을 둔 이유는 무엇인가**

- 참고할 노하우
    - 개별 fact 저장을 넘어, 주제 묶음과 순차 사건 흐름까지 표현하는 방식
    - 단순 triplet 저장소보다 에이전트 메모리에 더 적합한 그래프 계층 설계

## 2. 사실 변경과 증분 갱신

**Q. 왜 사실 변경을 삭제가 아니라 invalidation으로 처리하는가**

- 참고할 노하우
    - 현재 사실과 과거 사실을 함께 보존하는 temporal KB 패턴
    - 시간에 따라 바뀌는 사용자 상태, 조직 정보, 정책 정보를 안전하게 관리하는 방식

**Q. `add_episode()`와 `add_episode_bulk()`를 분리한 이유는 무엇인가**

- 참고할 노하우
    - 정교한 단건 갱신과 처리량 중심의 벌크 적재를 분리하는 운영 전략
    - 동일한 KB라도 ingest 경로를 하나로 통합하지 않고 목적별로 나누는 이유

**Q. 노드/엣지 dedupe와 contradiction 처리를 ingestion 파이프라인 안에 넣은 이유는 무엇인가**

- 참고할 노하우
    - 추출 품질보다 정규화와 충돌 처리의 중요도를 높게 보는 설계
    - LLM 추출 결과를 그대로 저장하지 않고 그래프 품질을 보정하는 계층적 방식

## 3. 검색과 문맥 구성

**Q. Hybrid search와 graph traversal은 각각 어떤 역할을 맡고 어떻게 함께 동작하는가**

- 참고할 노하우
    - BM25, 벡터 검색, BFS를 분리한 뒤 조합하는 검색 설계
    - 단순 Top-K 검색이 아니라 에이전트가 쓸 만한 그래프 문맥을 구성하는 방식

**Q. 왜 `search()`와 `search_()`를 분리했는가**

- 참고할 노하우
    - 빠른 fact 검색과 구조화된 graph context 반환을 분리하는 인터페이스 전략
    - 에이전트 통합 시 단순 답변 보조와 고급 추론 보조를 나누는 방식

**Q. RRF, MMR, cross-encoder, node-distance 같은 reranker를 recipe로 제공하는 이유는 무엇인가**

- 참고할 노하우
    - 검색 품질 제어를 코드 상수와 설정 객체로 재사용 가능하게 만드는 패턴
    - 특정 도메인과 비용 제약에 따라 검색 전략을 갈아끼우는 구조

**Q. provenance가 붙은 fact를 Episode로 역추적할 수 있게 하는 것이 왜 중요한가**

- 참고할 노하우
    - 에이전트 메모리에서 환각 억제와 감사 가능성을 확보하는 방식
    - 검색 결과를 바로 원문 상황으로 되돌리는 컨텍스트 복원 전략

## 4. 추상화와 백엔드 전략

**Q. 왜 Graphiti는 graph DB, LLM, embedder, cross-encoder를 모두 교체 가능한 계층으로 분리했는가**

- 참고할 노하우
    - 제품이 특정 모델 또는 특정 DB에 잠기지 않도록 만드는 추상화 전략
    - 비용, 성능, 배포 환경에 따라 구성 요소를 바꾸면서 코어 로직은 유지하는 방식

**Q. driver/provider 추상화는 어떤 수준까지 백엔드 차이를 흡수하고, 어디서부터 provider별 제약이 드러나는가**

- 참고할 노하우
    - Neo4j, FalkorDB, Kuzu, Neptune을 하나의 엔진 아래 묶는 현실적 다중 백엔드 전략
    - 이상적인 공통 API와 실제 provider parity 사이의 간극을 다루는 방법

**Q. namespace + operations 구조로 리팩터링하려는 이유는 무엇인가**

- 참고할 노하우
    - 데이터 모델과 DB I/O를 분리해 아키텍처를 장기적으로 안정화하는 방식
    - 백엔드 추가와 테스트를 쉽게 만들기 위한 점진적 구조 개선 전략

## 5. 에이전트 메모리와 인터페이스 설계

**Q. Graphiti는 왜 답변 생성 엔진이 아니라 에이전트 메모리 엔진으로 자신을 정의하는가**

- 참고할 노하우
    - 생성 모델과 메모리 인프라를 분리하는 제품 경계 설정
    - KB 자체가 답변을 하지 않고, 에이전트에게 더 나은 컨텍스트를 공급하는 구조

**Q. MCP 서버와 FastAPI 서버를 별도 레이어로 둔 이유는 무엇인가**

- 참고할 노하우
    - 같은 코어 엔진을 다른 소비 채널에서 재사용하는 인터페이스 설계
    - 라이브러리, 서비스, MCP 도구를 분리해 확장하는 제품 구조

**Q. group 단위 직렬 처리와 saga 연결은 왜 agent memory에서 중요한가**

- 참고할 노하우
    - 시간 순서가 깨지면 메모리 품질이 무너지는 문제를 운영 계층에서 방어하는 방식
    - 대화 로그를 단순 저장이 아니라 사건 흐름으로 보존하는 구조

## 6. 운영 안정성과 품질 장치

**Q. `SEMAPHORE_LIMIT` 같은 동시성 제어를 제품 핵심 설정으로 두는 이유는 무엇인가**

- 참고할 노하우
    - LLM 기반 KB에서 처리량과 rate limit 안정성을 함께 맞추는 실전 패턴
    - 모델 호출 비용과 ingestion 속도를 운영 변수로 노출하는 방식

**Q. tracing, telemetry, 테스트 스킵 정책까지 포함해 provider 성숙도를 어떻게 관리하는가**

- 참고할 노하우
    - 기능 지원 여부와 실제 운영 신뢰도를 구분해서 보는 관점
    - 멀티 provider 시스템에서 문서, 코드, 테스트 신호를 함께 읽는 방법

**Q. 입력 검증과 검색 필터 방어를 왜 helpers와 테스트 계층에 함께 넣었는가**

- 참고할 노하우
    - label injection, group_id 오염 같은 위험을 초기에 차단하는 보안 기본기
    - 지식 베이스 제품에서도 query safety를 명시적으로 다루는 방식

## 이 문서로 최종 확인하고 싶은 것

- Graphiti의 핵심 아이덴티티가 왜 문서형 RAG 앱이 아니라 temporal context graph 엔진인지
- 시간 변화와 provenance를 함께 다루는 fact model이 어떤 제품 차별점을 만드는지
- 검색 품질이 BM25, 벡터 검색, graph traversal, reranker 조합 위에서 어떻게 만들어지는지
- 다중 백엔드와 다중 모델 추상화가 어떤 운영 유연성을 주는지
- 에이전트 메모리 엔진으로서 Graphiti가 어떤 인터페이스와 운영 규칙을 가지는지
