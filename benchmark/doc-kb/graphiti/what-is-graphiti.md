# Graphiti 한눈에 이해하기

## 이 문서의 목표

이 문서는 Graphiti를 처음 접하는 사람이 짧은 시간 안에 아래 내용을 이해하도록 돕기 위해 작성했다.

- Graphiti가 어떤 프로젝트인지
- 왜 일반적인 RAG나 벡터 검색만으로는 부족한 문제를 풀려고 하는지
- 이 프로젝트가 내세우는 강점이 무엇인지
- 핵심 개념이 무엇인지
- 저장소 안에서 어떤 부분이 실제 제품 가치의 중심인지

대상 독자는 AI 기술을 어느 정도 알고 있고 MCP가 무엇인지도 알지만, knowledge graph나 temporal memory 시스템은 깊게 보지 않은 사람을 기준으로 잡았다.

## 한 줄 설명

Graphiti는 AI 에이전트를 위한 시간 기반 메모리 그래프 엔진이다.

더 정확히 말하면, 대화 로그나 문서 같은 입력을 저장하고 그 안에서 개체와 사실을 뽑아 그래프로 관리하면서, 시간이 지나며 바뀌는 사실까지 추적하는 시스템이다.

## 왜 이런 프로젝트가 필요한가

많은 AI 시스템은 정보를 다룰 때 주로 아래 두 방식을 쓴다.

- 문서를 잘게 나눠 벡터 DB에 넣고 검색하는 RAG
- 애플리케이션 코드에서 별도 메모리 테이블이나 캐시를 직접 관리하는 방식

이 접근은 단순한 문서 검색에는 유용하지만, 에이전트 메모리로 쓰기에는 몇 가지 한계가 있다.

- 사실이 바뀌었을 때 이전 사실과 현재 사실을 함께 다루기 어렵다
- 정보가 어떤 입력에서 나왔는지 provenance를 추적하기 어렵다
- 문서 조각은 찾을 수 있어도 개체와 관계를 구조적으로 다루기 어렵다
- 사용자의 선호, 조직 변화, 작업 이력처럼 시간에 따라 변하는 지식을 표현하기 어렵다

Graphiti는 바로 이 지점을 겨냥한다. 이 프로젝트는 "무엇이 관련 있는가"만 찾는 것이 아니라, "언제부터 사실이었는가", "무엇이 이미 바뀌었는가", "어떤 원문에서 나온 정보인가"까지 같이 다루려 한다.

## Graphiti가 푸는 문제

예를 들어 어떤 에이전트가 사용자와 오래 상호작용한다고 가정해보자.

- 1월에는 사용자가 "나는 커피보다 차를 더 좋아해"라고 말했다
- 3월에는 "이제는 아침마다 커피를 마셔"라고 말했다
- 4월에는 팀 이동으로 소속 조직이 바뀌었다

단순 벡터 검색 시스템은 이 세 정보를 관련 문장으로는 찾을 수 있다. 하지만 아래 질문에 안정적으로 답하기는 어렵다.

- 지금 기준으로 사용자의 최신 선호는 무엇인가
- 예전 선호는 언제까지 유효했는가
- 조직 변경은 어떤 입력에서 확인되었는가
- 이 사용자와 관련된 다른 개체나 사건은 무엇인가

Graphiti는 이런 정보를 `Episode`, `Entity`, `Fact` 단위로 저장하고, 사실이 바뀌면 이전 사실을 invalidation 처리해 시간 축을 가진 메모리로 유지한다.

## 이 프로젝트의 강점

### 1. 시간성을 다루는 메모리

Graphiti의 가장 큰 강점은 사실이 고정돼 있다고 가정하지 않는 점이다.

- 사실의 시작 시점 기록
- 더 이상 유효하지 않게 된 시점 기록
- 현재 사실과 과거 사실 동시 보관

이 덕분에 에이전트는 단순히 "기억한다"를 넘어서 "지금 무엇이 맞는지"와 "예전에는 무엇이 맞았는지"를 구분할 수 있다.

### 2. 원문 provenance 유지

Graphiti는 정보를 바로 그래프로만 바꾸지 않고, 먼저 입력 원문을 `Episode`로 저장한다.

그래서 나중에 아래 추적이 가능하다.

- 이 사실이 어느 대화나 문서에서 나왔는가
- 같은 사실이 여러 입력에서 반복 확인되었는가
- 문제가 생겼을 때 어떤 입력이 원인이었는가

에이전트 메모리 시스템에서 provenance는 신뢰성과 디버깅 가능성에 직접 연결된다.

### 3. 구조화된 검색

Graphiti의 검색 결과는 단순 문서 조각만이 아니다. 검색 시 아래 객체를 함께 다룬다.

- 관계와 사실을 나타내는 edge
- 개체를 나타내는 node
- 원문 입력인 episode
- 상위 요약 단위인 community

즉 검색 결과가 "관련 문장 몇 개"로 끝나지 않고, 에이전트가 더 풍부한 컨텍스트를 조합할 수 있는 형태로 나온다.

### 4. LLM과 그래프 검색의 결합

Graphiti는 LLM 추출만 쓰는 시스템도 아니고, 그래프 DB만 쓰는 시스템도 아니다.

- LLM으로 개체와 관계를 추출
- dedupe와 invalidation으로 그래프 정리
- 임베딩과 그래프 탐색을 함께 사용해 검색
- reranker로 결과 품질 보정

이 조합 덕분에 비정형 입력을 구조화하면서도, 검색 단계에서 단순 키워드 매칭 이상의 품질을 노릴 수 있다.

### 5. 통합 가능한 아키텍처

이 프로젝트는 특정 벤더 하나에 강하게 묶이지 않으려는 방향을 갖고 있다.

- 그래프 DB 교체 가능
- LLM 제공자 교체 가능
- embedder와 reranker 교체 가능
- MCP 서버 형태로 외부 에이전트에 노출 가능

즉 Graphiti는 "하나의 완성 앱"이라기보다, 에이전트 메모리 레이어로 붙일 수 있는 엔진에 가깝다.

## 핵심 개념

Graphiti를 이해할 때 가장 중요한 개념은 다섯 가지다.

| 개념 | 의미 | 왜 중요한가 |
| --- | --- | --- |
| Episode | 원문 입력 단위 | provenance의 기준점 |
| EntityNode | 사람, 조직, 장소, 문서, 주제 같은 개체 | 메모리의 주요 대상 |
| EntityEdge | 개체 사이의 사실이나 관계 | 시간성 있는 지식 표현 |
| CommunityNode | 관련 개체 묶음을 요약한 상위 노드 | 큰 맥락 탐색에 유용 |
| Saga | 순차적인 episode 흐름 | 대화나 작업 단계 연결 |

### Episode

Episode는 Graphiti에 들어오는 원본 정보 단위다.

예시

- 사용자 메시지
- 문서 일부
- JSON 이벤트

Graphiti는 Episode를 먼저 저장한 뒤, 그 안에서 개체와 사실을 추출한다. 그래서 나중에 그래프의 모든 정보가 어느 입력에서 비롯됐는지 되짚을 수 있다.

### EntityNode

EntityNode는 그래프 안의 개체다.

예시

- 사람
- 회사
- 팀
- 제품
- 위치
- 주제

에이전트 메모리 관점에서는 "누가", "무엇이", "어디가"에 해당하는 대상이라고 보면 된다.

### EntityEdge

EntityEdge는 두 개체 사이의 사실이나 관계다.

예시

- `사용자 A`는 `팀 B`에 속한다
- `사용자 A`는 `커피`를 선호한다
- `제품 X`는 `회사 Y`가 만든다

중요한 점은 이 edge가 단순 연결선이 아니라 fact와 시간 정보를 함께 가진다는 점이다. 그래서 Graphiti는 관계 그래프이면서 동시에 사실 관리 시스템 역할도 한다.

### CommunityNode

CommunityNode는 서로 많이 연결된 EntityNode 집합을 상위 수준에서 요약한 노드다.

쉽게 말해, 관련 개체들을 묶어서 더 큰 주제나 문맥 단위로 보여주기 위한 계층이다. 대규모 그래프에서 전체 맥락을 빠르게 파악할 때 유용하다.

### Saga

Saga는 순서가 있는 episode들의 흐름을 표현한다.

예를 들면 아래 같은 상황에 맞다.

- 여러 턴에 걸친 대화
- 단계가 있는 작업 수행 기록
- 하나의 사건을 따라가는 연속 입력

즉 Graphiti는 개별 사실만 저장하는 것이 아니라, 사건의 흐름도 다룰 수 있다.

## Graphiti는 어떻게 동작하나

가장 중요한 흐름은 `Episode ingestion`과 `Search` 두 가지다.

### 1. Episode ingestion

새 정보가 들어오면 대체로 아래 순서로 처리된다.

1. 입력을 Episode로 받는다
2. LLM으로 개체를 추출한다
3. 기존 그래프와 비교해 같은 개체를 dedupe한다
4. LLM으로 개체 사이 관계를 추출한다
5. 기존 사실과 비교해 중복을 줄이고, 모순되는 옛 사실은 invalidation 처리한다
6. 임베딩을 생성한다
7. 그래프 DB에 저장한다
8. 필요하면 community와 saga 연결도 갱신한다

핵심은 "새 텍스트를 그냥 추가 저장"하는 것이 아니라, 기존 메모리와 비교하면서 그래프를 유지보수한다는 점이다.

### 2. Search

질문이 들어오면 Graphiti는 여러 방법을 조합해 관련 컨텍스트를 찾는다.

- BM25
- cosine similarity
- BFS 기반 그래프 탐색
- reranker 기반 재정렬

결과도 하나의 타입으로만 끝나지 않는다.

- 관련 사실
- 관련 개체
- 관련 episode
- 관련 community

그래서 에이전트는 단순 검색 결과가 아니라, 답변 생성에 쓸 수 있는 구조화된 기억 묶음을 받게 된다.

## 일반적인 RAG와 어떻게 다른가

| 항목 | 일반적인 RAG | Graphiti |
| --- | --- | --- |
| 기본 단위 | 문서 청크 | episode, node, edge |
| 주요 관심사 | 관련 문서 회수 | 시간성 있는 사실과 관계 관리 |
| 변경 처리 | 최신 문서 재색인 중심 | 기존 사실 invalidation 가능 |
| provenance | 보통 문서 단위 | episode 단위로 더 직접적 |
| 검색 결과 | 주로 텍스트 조각 | 사실, 개체, 원문, 커뮤니티 |
| 관계 표현 | 약함 | 그래프 구조로 명시적 표현 |

Graphiti는 RAG를 대체하는 모든 것이라기보다, "에이전트 메모리"나 "동적으로 변하는 지식"을 다룰 때 더 잘 맞는 접근이라고 보는 편이 정확하다.

## 저장소는 어떻게 구성돼 있나

처음 보는 사람은 저장소를 세 부분으로 나눠 이해하면 된다.

### `graphiti_core`

프로젝트의 핵심 라이브러리다.

여기에 실제 제품 가치가 가장 많이 들어 있다.

- Episode ingestion
- node와 edge 모델
- dedupe와 invalidation
- search
- community build
- DB driver 추상화
- LLM, embedder, reranker 추상화

### `server`

FastAPI 기반 예시 서버다.

Graphiti를 HTTP API로 감싸는 얇은 래퍼에 가깝다. 실전 제품 서버라기보다 사용 예시와 테스트용 진입점에 가깝다.

### `mcp_server`

Graphiti를 MCP 도구로 노출하는 서버다.

에이전트 통합 관점에서는 이 부분이 특히 중요하다. 외부 에이전트가 Graphiti를 메모리 도구처럼 사용할 수 있게 해준다.

## 누가 이 프로젝트를 좋아할 가능성이 큰가

아래 같은 문제를 가진 팀이라면 Graphiti가 특히 잘 맞을 수 있다.

- 장기 대화 메모리를 가진 AI 에이전트를 만들고 싶은 팀
- 사용자의 선호와 상태 변화까지 기억해야 하는 제품을 만드는 팀
- 조직 정보, 고객 이력, 사건 변화를 시간 축으로 다뤄야 하는 팀
- 문서 검색을 넘어서 개체와 관계를 구조적으로 검색하고 싶은 팀
- MCP 기반 에이전트에 메모리 서비스를 붙이고 싶은 팀

## 반대로 주의할 점

Graphiti는 강력하지만, 모든 상황에 가장 단순한 해법은 아니다.

- 단순 FAQ 검색만 필요하면 더 가벼운 RAG가 충분할 수 있다
- LLM 추출 품질에 영향을 받는다
- 그래프 DB와 임베딩, reranker까지 포함하면 운영 복잡도가 올라간다
- provider마다 기능 성숙도가 완전히 같지는 않다
- bulk ingestion과 single ingestion은 의미가 완전히 같지 않다

즉 Graphiti는 "구조화된 장기 메모리" 문제를 푸는 데 강하지만, 그만큼 설계와 운영에서 생각할 요소도 더 많다.

## 이 프로젝트를 한 문장으로 다시 말하면

Graphiti는 AI 에이전트가 시간에 따라 변하는 사실을 기억하고, 그 출처를 추적하며, 관계 중심으로 검색할 수 있게 해주는 메모리 그래프 엔진이다.
