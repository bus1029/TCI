# Anything-llm 개발 노하우 참고 질문

이 문서는 사내 지식 베이스 구축 시 Anything-llm 제품의 핵심 동작 방식과 강점을 파악하고, 우리 제품에 적용할 수 있는 설계 요소들을 정리한 질문 모음이다.

## 1. 다종 문서 파싱 및 지식 변환 파이프라인

**Q. 다종 문서(PDF, DOCX, 엑셀 등)를 어떤 파서로 처리하는가?**
- 참고할 노하우
  - 서드파티 패키지 조합을 통해 다종 포맷(표가 포함된 PDF, 엑셀 등)을 텍스트로 치환하는 모듈화 로직을 벤치마킹함

## 2. 최적화된 검색 파이프라인과 저장소 아키텍처

**Q. RDBMS, Vector DB, 파일 시스템 등 다중 저장소를 어떻게 조합해서 쓰는가?**
- 참고할 노하우
  - 무거운 텍스트 원문은 파일 시스템으로 분리함
  - 관계 메타데이터는 RDBMS에, 검색 수치만 Vector DB에 적재함
  - 위 분산을 통해 DB 트랜잭션 마비와 볼륨 폭주를 막아내는 삼원화 아키텍처를 도입함

**Q. 쿼리 검색 속도 최적화 및 과금 절감을 위해 캐싱(Caching) 레이어를 어떻게 배치했는가?**
- 참고할 노하우
  - 복수 임직원의 중복 질의에 대비해 캐싱 프레임워크를 차용함
  - LLM API를 매번 과금하며 호출하지 않고 DB 단에서 응답을 가로채어 지연 시간과 비용을 낮춤

**Q. LLM 모델이나 Vector DB 벤더를 무중단 교체할 수 있는 추상화 레이어(Adapter Pattern)는 어떻게 구현했는가?**
- 참고할 노하우
  - 클라우드 공급자 스펙에 종속되는 단점을 회피하도록 설계함
  - 핵심 로직 변경 없이 상용 모델과 폐쇄망 모델을 스위칭할 수 있는 확장형 패턴을 적용함

## 3. 환각 억제 및 컨텍스트 안정성

**Q. 지식 단위(청크)별 원본 문서 출처(Citation) 매핑을 위한 메타데이터를 어떻게 주입하는가?**
- 참고할 노하우
  - AI 에이전트의 환각 리스크를 없애기 위해 응답 신뢰도를 보장하는 방식을 채택함
  - 텍스트 분할 단계부터 원본 링크 및 파일명을 청크 헤더 영역에 강제 삽입함

**Q. LLM 모델의 한계 토큰을 초과하지 않도록 런타임에 동적으로 프롬프트 컨텍스트를 제한하는 구조인가?**
- 참고할 노하우
  - 워크스페이스 전역 하드 리밋(Hard Limit) 방호선 설계를 도입함
  - 방대한 검색 결과가 유입되더라도 반환되는 문서 토큰 범위를 미리 잘라내어 토큰 오버플로우 에러를 원천 시봉함

## 4. 엔터프라이즈 보안 및 사내 융합 확장성

**Q. 특정 부서 단위에 연동된 세밀한 문서 접근 권한 제어(Document-level Security, RBAC)가 가능한가?**
- 참고할 노하우
  - 조직 내 민감 정보가 교차 노출되는 보안사고를 막기 위해 폐쇄형 권한 모델을 본받음
  - 유저 ID 그룹과 문서 폴더 스키마 격벽(Silo)을 철저히 단절함

**Q. 별도의 사내 포털이나 웹페이지에 지식 베이스를 조각 위젯(Embeddable Widget) 형태로 삽입하고 보안 통신하는가?**
- 참고할 노하우
  - 독립된 앱으로 제한하지 않고 기존 레거시 시스템 어디에든 챗봇을 유연하게 삽입 연동할 수 있는 구조를 마련함
  - 이를 위해 CORS 스크립트 발급 API 레이어를 별도로 구축함
