# Basic Memory 개발 노하우 참고 질문

이 문서는 사내 지식 베이스 구축 시 Basic Memory 제품의 핵심 설계 방식과 강점을 파악하고, 우리 제품에 적용할 수 있는 개발 노하우를 정리하기 위한 질문 모음이다.

기본 접근은 `docs/questions/01~04`의 질문들 중 Basic Memory의 강점을 가장 잘 드러내는 항목을 다시 추리고, 이 제품에만 두드러지는 설계 축을 추가로 보완하는 방식이다.

## 1. Markdown 기반 지식 표현 방식

**Q. 사람도 직접 읽고 수정할 수 있는 Markdown 파일을 어떻게 장기적인 지식 단위로 유지하는가**
- 참고할 노하우
  - 별도 전용 포맷이 아니라 일반 Markdown을 source of truth로 삼는 방식
  - AI가 만든 지식과 사람이 직접 고친 지식을 같은 저장소에서 공존시키는 방식

**Q. Observation, Relation, Permalink를 어떤 규칙으로 추출하고 왜 이런 모델을 택했는가**
- 참고할 노하우
  - 문서 전체를 벡터 덩어리로만 다루지 않고 사실과 링크를 구조 단위로 분해하는 방식
  - 검색, 문맥 구성, 링크 안정성을 높이기 위한 최소 문법 설계

**Q. 파일 경로와 별개로 안정적인 식별자를 어떻게 유지하며, 파일 이동이나 이름 변경 시 링크를 어떻게 보호하는가**
- 참고할 노하우
  - 파일 시스템 친화적인 운영과 안정적인 식별 체계를 함께 가져가는 방식
  - 경로 기반 저장소에서 permalink를 별도 축으로 운영하는 이유

## 2. Local-first 저장 구조와 동기화

**Q. 파일이 원본이고 DB는 인덱스인 구조에서 두 계층의 일관성을 어떻게 유지하는가**
- 참고할 노하우
  - source of truth와 검색 인덱스를 분리하면서도 운영 복잡도를 낮추는 방식
  - 파일 기반 시스템이 검색 성능을 잃지 않도록 만드는 기본 원칙

**Q. watcher, checksum, watermark 기반 증분 sync를 어떤 흐름으로 묶었는가**
- 참고할 노하우
  - 대량 노트에서도 전체 재색인을 피하고 변경분만 빠르게 반영하는 방식
  - watcher와 sync service를 분리해 운영 안정성을 확보하는 구조

**Q. 파일 직접 수정, 이동, 삭제가 발생해도 인덱스와 그래프를 어떻게 복구하는가**
- 참고할 노하우
  - 사람이 에디터에서 직접 작업해도 시스템 정합성을 유지하는 복구 흐름
  - 지식 베이스를 앱 내부 데이터가 아니라 파일 시스템 위에서 운영하는 방식

**Q. `doctor` 같은 검증 루프를 왜 제품 안에 포함시켰는가**
- 참고할 노하우
  - 파일 ↔ DB 루프가 실제로 건강한지 확인하는 운영 점검 방식
  - 지식 베이스 제품에서 테스트와 런타임 검증을 연결하는 방법

## 3. 검색과 문맥 구성

**Q. Hybrid search와 graph traversal은 각각 어떤 역할을 맡고, 어떻게 함께 동작하는가**
- 참고할 노하우
  - 키워드 매칭과 의미 검색, 관계 확장을 분리해 설계하는 방식
  - 단순 Top-K 검색을 넘어서 실제로 쓸 수 있는 문맥을 만드는 구조

**Q. `memory://` 주소 체계를 왜 도입했고, LLM 도구 호출과 어떻게 연결하는가**
- 참고할 노하우
  - LLM이 후속 도구 호출에서 일관된 지식 식별자를 쓰게 만드는 방식
  - 세션 히스토리 대신 저장된 지식을 다시 탐색하는 memory 설계

**Q. 검색 결과에 permalink, file_path, matched chunk 같은 출처 정보를 어떤 수준까지 남기는가**
- 참고할 노하우
  - 완전한 citation 시스템이 아니더라도 사람이 원문으로 되돌아가기 쉽게 만드는 방식
  - 환각 억제를 위해 최소한 어떤 추적성을 확보해야 하는지 판단하는 기준

**Q. 검색 결과 수, 청크 수, preview 길이를 어떤 규칙으로 제한해 컨텍스트 윈도우를 관리하는가**
- 참고할 노하우
  - 토큰 예산을 정적 규칙과 제품 상수로 운영하는 실전 패턴
  - 방대한 검색 결과가 들어와도 LLM 호출이 무너지지 않게 막는 방식

## 4. MCP 도구와 AI 협업 인터페이스

**Q. 왜 거대한 단일 에이전트 대신 atomic하고 composable한 MCP 도구 세트로 설계했는가**
- 참고할 노하우
  - `write_note`, `search_notes`, `build_context`, `recent_activity` 같은 작은 도구를 조합하는 방식
  - 상위 LLM 클라이언트가 더 유연하게 판단하도록 만드는 인터페이스 철학

**Q. MCP tool → typed client → API → service 구조로 나눈 이유는 무엇인가**
- 참고할 노하우
  - 도구 계층과 도메인 로직을 느슨하게 결합하는 구조
  - CLI, API, MCP가 같은 코어 로직을 재사용하게 만드는 아키텍처 패턴

**Q. AI 대화를 일회성 채팅 로그가 아니라 재사용 가능한 지식으로 남기게 만드는 제품 설계는 무엇인가**
- 참고할 노하우
  - 대화 내용을 노트와 그래프 자산으로 승격시키는 흐름
  - 인간과 AI가 같은 지식 저장소를 함께 가꾸는 협업 모델

## 5. 로컬 우선 확장과 클라우드 연결

**Q. 로컬 우선 제품이면서도 프로젝트별로 local/cloud routing을 공존시키는 이유는 무엇인가**
- 참고할 노하우
  - 전체 시스템을 한 번에 클라우드로 넘기지 않고 프로젝트 단위로 전환하는 확장 전략
  - local-first 철학을 해치지 않고 SaaS 기능을 추가하는 방식

**Q. 동일한 도구 호출을 유지한 채 라우팅만 바꾸는 클라이언트 추상화는 어떻게 설계했는가**
- 참고할 노하우
  - 사용자 경험을 바꾸지 않고 실행 경로만 바꾸는 클라이언트 설계
  - per-project routing이 제품 복잡도를 어떻게 흡수하는지 보여주는 사례

**Q. 폐쇄망 또는 온프레미스 환경에서 외부 전송 없이 운영하려면 어떤 구성과 제약을 이해해야 하는가**
- 참고할 노하우
  - local mode, 로컬 임베딩, Docker 배포를 조합해 오프라인 운영하는 방식
  - 클라우드 확장 기능이 있어도 기본 운영 철학은 로컬에 두는 구조

**Q. 스냅샷, 백업, 복구 흐름은 local-first 제품에서 어떤 수준까지 제품이 책임지고 어디부터 운영자가 맡는가**
- 참고할 노하우
  - 클라우드 경로와 로컬 경로의 책임 경계를 분리해 보는 방식
  - 파일 저장소와 인덱스 저장소를 함께 복구해야 하는 제품의 운영 관점

## 6. 구조 품질과 스키마 진화

**Q. 자유로운 Markdown 위에 schema inference, validation, drift 분석을 왜 얹었는가**
- 참고할 노하우
  - 지나치게 엄격한 입력 강제 없이도 지식 구조 품질을 높이는 방법
  - 장기간 운영 시 노트 구조가 무너지는 문제를 감지하는 방식

**Q. 지식 베이스를 단순 저장소가 아니라 점점 더 정돈된 시스템으로 진화시키기 위해 어떤 품질 장치를 넣었는가**
- 참고할 노하우
  - 포맷 규칙, schema, sync 검증, 검색 품질 보정이 서로 어떻게 연결되는지 파악하는 질문
  - 초기에는 유연하게 시작하고 점차 엄격도를 높이는 제품 전략

## 이 문서로 최종 확인하고 싶은 것

- Basic Memory의 핵심 아이덴티티가 왜 범용 RAG 앱이 아니라 local-first knowledge infrastructure인지
- Markdown 파일 원본과 검색 인덱스를 함께 운영하는 실전 설계 원칙
- 그래프 기반 문맥 구성과 MCP 도구 설계가 어떻게 맞물리는지
- 로컬 우선 구조를 유지한 채 클라우드 확장을 붙이는 방법
- 자유로운 Markdown에 구조 품질을 더하는 운영 노하우
