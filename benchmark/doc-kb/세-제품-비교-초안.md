# 한 줄 결론

- 지금 단계의 권장 출발점은 `AnythingLLM`의 문서 수집 구조와 `Basic Memory`의 저장 원칙을 결합하는 방향이다
- `Graphiti`는 장기 확장 방향으로는 중요하지만 초기 범위의 중심축으로 두기에는 복잡도가 높다

# 권장 방향

- 1단계: 문서 수집과 공통 저장 계약을 먼저 정리
- 2단계: 파일 원본과 인덱스의 경계를 분명히 설정
- 3단계: 검색과 문맥 조립을 분리해 구조를 단순화
- 4단계: 이후 필요할 때 provenance(출처 역추적 정보), 상태 전이, 에이전트 메모리 요소를 점진적으로 확장

# 지금 하지 말아야 할 것

- 초기 단계에서 완전한 temporal fact graph(시간 축을 가진 사실 그래프)를 먼저 도입하지 말 것
- 모든 provider를 같은 수준으로 지원하는 추상화를 초기에 요구하지 말 것
- 고비용 reranker(검색 결과 재정렬기)와 복잡한 그래프 탐색을 조기 도입하지 말 것
- 문서형 지식 베이스의 기본기보다 메모리 엔진 수준의 복잡도를 앞세우지 말 것

# 의사결정 필요 사항

- 우리 제품의 1차 목표를 문서형 지식 베이스로 볼지 결정
- 파일 원본을 진실 원본으로 둘지 결정
- 워크스페이스 운영과 권한 모델을 어느 수준까지 포함할지 결정
- 장기적으로 에이전트 메모리까지 확장할지 결정

# 이 문서가 답하려는 질문

- 세 제품은 각각 어떤 문제를 푸는가
- 우리 제품은 어느 설계 축에 더 가까운가
- 지금 바로 가져올 요소와 나중에 도입할 요소는 무엇인가
- 현재 단계에서 과한 복잡도는 무엇인가

# 문서 목적

이 문서는 `AnythingLLM`, `Basic Memory`, `Graphiti`를 비교해 우리 제품이 어떤 지식 표현 방식, 저장 구조, 검색 방식, 운영 모델을 선택해야 하는지 판단하기 위한 기준을 정리한다.

핵심 비교 기준은 기능 개수보다 `제품이 지식을 무엇으로 보는가`, `어떻게 갱신하는가`, `어떤 문맥을 에이전트에 공급하는가`에 둔다.

# 제품별 한 줄 정의

- `AnythingLLM`은 다종 문서를 수집해 워크스페이스 단위로 검색하고 운영하는 문서형 지식 베이스 제품이다
- `Basic Memory`는 Markdown 파일을 원본으로 두고 사람과 AI가 함께 가꾸는 로컬 우선 지식 인프라다
- `Graphiti`는 `Episode`(원문 입력), `Entity`(개체), `Fact`(사실)를 시간 축 위에 유지하는 에이전트 메모리 엔진이다

# 이 문서를 읽는 기준

이 세 제품은 같은 문제를 푸는 도구가 아니다. `AnythingLLM`은 문서형 검색과 운영, `Basic Memory`는 장기 지식 저장과 협업, `Graphiti`는 시간성을 가진 에이전트 메모리에 더 가깝다.

따라서 이 문서는 어떤 제품이 더 좋다고 결론내리기보다, 각 제품이 어떤 문제를 어떤 데이터 모델과 운영 방식으로 푸는지 비교하는 데 초점을 둔다.

# 처음 읽는 사람을 위한 핵심 개념

이 문서에는 제품 고유 개념과 내부 구현처럼 보이는 표현이 일부 나온다. 아래 뜻만 먼저 잡고 읽으면 뒤 비교표와 상세 비교를 훨씬 쉽게 따라갈 수 있다.

## `AnythingLLM`에서 자주 나오는 말

- `워크스페이스`: 문서, 설정, 검색 범위를 함께 묶는 작업 공간
- `청크`: 긴 문서를 검색 가능한 작은 조각으로 나눈 단위
- `pinned 문맥`: 검색 결과와 별개로 답변에 항상 함께 넣는 고정 참고 문맥
- `벡터 ID 브리지`: 문서와 실제 벡터 조각 ID를 연결해 삭제와 재색인을 쉽게 만드는 매핑 계층

## `Basic Memory`에서 자주 나오는 말

- `observation`: Markdown 노트 안에서 추출하는 구조화된 사실 한 줄
- `relation`: 노트와 다른 노트 또는 개체를 잇는 연결 정보
- `permalink`: 파일 경로와 별개로 유지되는 안정 식별자
- `memory://` 주소: 저장된 지식을 다시 찾기 위한 논리 주소
- `doctor`: 파일 원본, 인덱스, 검색 상태가 서로 맞는지 점검하는 검증 루프

## `Graphiti`에서 자주 나오는 말

- `Episode`: 대화, 문서, 이벤트 같은 원문 입력 한 건
- `Entity`: Episode나 문서에서 추출한 사람, 조직, 개념 같은 개체
- `Fact`: 개체 사이의 사실 또는 관계
- `provenance`: 어떤 원문 입력에서 나왔는지 되짚을 수 있게 하는 출처 추적 정보
- `invalidation`: 기존 사실을 지우지 않고 과거 사실로 전환하는 처리

# 축별 상세 비교

아래 비교는 제품의 핵심 설계 차이를 기준으로 정리했다.

## 1. 비교 대상과 전제

세 제품은 모두 지식을 저장하고 다시 꺼내 쓰게 만들지만, 출발점과 저장 단위가 서로 다르다.

- 공통점: 검색 가능한 지식 자산을 만들고 AI나 사용자가 다시 활용하게 함
- 차이점: 문서 운영, 파일 기반 지식 저장, 시간성 있는 메모리라는 서로 다른 문제를 풂
- 읽는 기준: 기능 우열보다 `무엇을 지식으로 보는가`와 `어떤 운영 모델을 택했는가`를 중심으로 비교해야 함

## 2. 제품 정체성

세 제품은 모두 지식을 다루지만, 제품이 자신을 규정하는 방식부터 다르다.

### `AnythingLLM`
- 중심 문제: 다양한 문서를 빠르게 수집하고 검색 가능한 워크스페이스로 운영
- 강점: 다종 입력 수용력, 워크스페이스 기반 운영, 문서형 검색 경험
- 한계: 장기 메모리나 시간성 있는 사실 관리보다는 문서 운영에 더 가까움

### `Basic Memory`
- 중심 문제: 사람이 직접 읽고 고칠 수 있는 지식 저장소를 오래 유지
- 강점: 파일 원본 소유권, 로컬 우선 구조, 사람과 AI의 공동 편집
- 한계: 대규모 범용 문서 수집기보다는 저장소형 제품에 가까움

### `Graphiti`
- 중심 문제: 변화하는 사실과 관계를 장기 메모리로 유지
- 강점: 시간성, provenance(출처 역추적 정보), 그래프 기반 문맥 조립
- 한계: 문서형 KB보다 구조와 운영 복잡도가 높음

### 비교 해석
- `AnythingLLM`은 문서 운영 문제를 푼다
- `Basic Memory`는 지식 저장소 소유권 문제를 푼다
- `Graphiti`는 장기 메모리와 상태 변화 문제를 푼다

### 우리 제품 시사점
- 먼저 풀 문제를 `문서 수집`, `지식 축적`, `시간성 메모리` 중 어디에 둘지 정해야 함
- 이 질문에 답하지 않으면 뒤의 기능 비교가 흐려짐

## 3. 지식 표현 단위

세 제품의 차이는 결국 `무엇을 지식의 최소 단위로 보느냐`에서 가장 크게 드러난다.

### `AnythingLLM`
- 기본 단위: 문서 JSON(여러 입력을 공통 형식으로 정리한 중간 문서 객체)과 청크(검색용 작은 조각)
- 원문 처리: 포맷별 파서를 거쳐 공통 문서 계약으로 정규화
- 검색 대상: 청크와 벡터 메타데이터

### `Basic Memory`
- 기본 단위: Markdown 노트
- 내부 구조화: observation(구조화된 사실)과 relation(연결 정보)을 별도 추출
- 검색 대상: 노트, observation, relation이 함께 검색 축을 이룸

### `Graphiti`
- 기본 단위: `Episode`(원문 입력), `Entity`(개체), `Fact`(사실)
- 원문 처리: 원문은 Episode로 남기고 재사용 단위는 사실 그래프로 전환
- 검색 대상: fact, node, episode, community 같은 그래프 레이어

### 비교 해석
- `AnythingLLM`은 문서를 검색 가능한 단위로 분해한다
- `Basic Memory`는 읽기 쉬운 노트에 약한 구조를 더한다
- `Graphiti`는 원문을 사실 그래프로 바꿔 메모리 단위를 다시 정의한다

### 우리 제품 시사점
- 문서 자체를 주 자산으로 볼지 정해야 함
- 사람이 읽는 노트를 주 자산으로 볼지 정해야 함
- 사실 그래프를 주 자산으로 볼지 정해야 함
- 이 선택이 이후 저장 구조와 검색 구조를 결정함

## 4. 원본 저장소와 인덱스 구조

세 제품은 원본과 인덱스를 어디에 두는지에서도 철학 차이가 뚜렷하다.

### `AnythingLLM`
- 원본 저장: 파일 기반 문서 JSON
- 인덱스 구조: 관계형 메타데이터와 벡터 저장소 분리
- 설계 의도: 원문, 제어 평면, 검색 계층을 나눠 운영성 확보

### `Basic Memory`
- 원본 저장: Markdown 파일이 진실 원본
- 인덱스 구조: 검색과 링크 해석을 위한 재구축 가능한 DB
- 설계 의도: 파일 소유권 유지와 검색 성능 확보를 동시에 추구

### `Graphiti`
- 원본 저장: Episode 원문 보존
- 인덱스 구조: 그래프 DB의 노드와 엣지가 사실상의 핵심 저장 모델
- 설계 의도: 저장 구조와 검색 구조를 사실 그래프 중심으로 통합

### 비교 해석
- `AnythingLLM`은 저장 계층 분리로 운영성을 확보한다
- `Basic Memory`는 원본 소유권을 파일 시스템에 남긴다
- `Graphiti`는 저장과 검색을 사실 그래프 구조로 통합한다

### 우리 제품 시사점
- 원본과 파생 데이터를 분리하려면 `AnythingLLM`과 `Basic Memory` 패턴이 유리함
- 그래프가 핵심 자산이면 `Graphiti`처럼 저장 모델부터 그래프 중심이어야 함

## 5. 갱신과 동기화 방식

세 제품은 갱신을 다루는 방식에서도 완전히 다른 운영 감각을 보여 준다.

### `AnythingLLM`
- 갱신 흐름: 수집 -> 파싱 -> 청킹 -> 임베딩 -> 재색인
- 핵심 관심사: 새 문서를 검색 가능한 상태로 빠르게 전환
- 보조 장치: 벡터 재사용 캐시로 비용 절감

### `Basic Memory`
- 갱신 흐름: 파일 감시기(watcher) -> checksum(내용 해시값) 비교 -> 증분 동기화 -> 복구
- 핵심 관심사: 파일 원본과 인덱스 정합성 유지
- 보조 장치: 이동, 삭제, 직접 수정까지 복구 가능한 sync 경로

### `Graphiti`
- 갱신 흐름: 추출 -> dedupe(중복 정리) -> contradiction 처리 -> invalidation(기존 사실을 과거 사실로 전환하는 처리)
- 핵심 관심사: 새 입력이 기존 사실을 어떻게 바꾸는지 정확히 반영
- 보조 장치: 단건 경로와 벌크 경로 분리

### 비교 해석
- `AnythingLLM`은 문서를 검색 가능 상태로 만드는 갱신이 중요하다
- `Basic Memory`는 파일 원본과 인덱스 정합성 유지가 중요하다
- `Graphiti`는 사실 상태 전이의 정확도가 중요하다

### 우리 제품 시사점
- 문서 업로드 후 검색 가능화가 핵심이면 `AnythingLLM`
- 사용자가 원본 파일을 직접 고치는 환경이면 `Basic Memory`
- 사실의 시간 이력까지 보존해야 하면 `Graphiti`

## 6. 검색과 문맥 구성

검색은 세 제품 모두 중요하지만, 검색 결과를 무엇으로 이어 붙이느냐에서 차이가 커진다.

### `AnythingLLM`
- 검색 출발점: 워크스페이스 범위 제한
- 문맥 구성: pinned 문서(고정 참고 문서), parsed 파일(정식 적재 전이거나 임시로 문맥에 들어가는 파싱 결과 파일), similarity search 결과 조합
- 마무리 방식: 프롬프트 압축으로 모델 입력 안정화

### `Basic Memory`
- 검색 출발점: hybrid retrieval
- 문맥 구성: relation traversal(관계 확장 탐색)과 `memory://` 주소 기반 후속 탐색
- 마무리 방식: 검색과 문맥 구성을 분리해 기억 인터페이스화

### `Graphiti`
- 검색 출발점: BM25, 벡터 검색, BFS(그래프 이웃 확장 탐색)
- 문맥 구성: edge, node, episode, community를 함께 조립
- 마무리 방식: `search()`와 `search_()`를 나눠 빠른 사실 검색과 더 구조화된 그래프 문맥 회수를 구분

### 비교 해석
- `AnythingLLM`은 문서 검색 결과를 프롬프트 재료로 조립한다
- `Basic Memory`는 검색 결과를 다시 탐색 가능한 지식 주소로 연결한다
- `Graphiti`는 그래프 레이어를 함께 조립해 구조화된 문맥을 만든다

### 우리 제품 시사점
- 후속 탐색 흐름이 중요하면 `Basic Memory`와 `Graphiti`가 더 참고할 만함
- 문서형 답변 보조가 목적이면 `AnythingLLM` 방식이 더 단순하고 빠름

## 7. 출처 추적과 신뢰성

세 제품은 모두 출처를 다루지만, 신뢰성을 만드는 방식은 서로 다르다.

### `AnythingLLM`
- 중심 장치: 청크 헤더 메타데이터와 벡터 메타데이터 보존
- 신뢰성 강화: query 모드(근거가 없으면 답을 거절하는 검색 중심 모드)에서 무근거 응답 차단
- 성격: 답변 근거 보강과 정책적 안전장치

### `Basic Memory`
- 중심 장치: `permalink`(안정 식별자), `file_path`, `matched_chunk`(실제 매칭된 텍스트 조각)
- 신뢰성 강화: 사람이 직접 원문으로 되돌아갈 수 있는 추적성
- 성격: citation보다 원문 복귀 가능성 중심

### `Graphiti`
- 중심 장치: Episode provenance(원문 출처 추적 정보)와 fact 역추적(이 사실이 어느 원문 입력에서 나왔는지 되짚는 구조)
- 신뢰성 강화: fact, entity, episode를 상호 복원 가능한 구조로 유지
- 성격: citation보다 감사 가능성과 이력 추적 중심

### 비교 해석
- `AnythingLLM`은 답변 근거 보강과 무근거 응답 차단에 강하다
- `Basic Memory`는 사람 중심의 원문 추적성에 강하다
- `Graphiti`는 시스템 수준의 provenance(출처 역추적 정보)와 이력 추적에 강하다

### 우리 제품 시사점
- 사용자용 근거 표시가 중요하면 `AnythingLLM`
- 사람이 원문을 직접 검토하는 흐름이면 `Basic Memory`
- 시스템 감사와 사실 역추적까지 필요하면 `Graphiti`

## 8. 인터페이스와 AI 협업 방식

세 제품은 AI와 사용자가 지식을 다루는 접점에서도 서로 다른 철학을 보인다.

### `AnythingLLM`
- 인터페이스 중심: 앱과 워크스페이스
- 협업 방식: 제품 안에서 문서를 넣고 검색하고 대화
- 강한 채널: 챗 인터페이스, 임베드, 워크스페이스 운영

### `Basic Memory`
- 인터페이스 중심: 작은 MCP 도구 조합
- 협업 방식: 검색, 읽기, 쓰기, 문맥 확장을 AI가 조합
- 강한 채널: 사람과 AI의 공동 편집 저장소

### `Graphiti`
- 인터페이스 중심: 메모리 엔진과 소비 채널 분리
- 협업 방식: 코어는 메모리 공급에 집중하고 답변 생성은 외부가 담당
- 강한 채널: 라이브러리, FastAPI, MCP 같은 재사용 가능한 표면

### 비교 해석
- `AnythingLLM`은 완성된 제품 인터페이스가 강하다
- `Basic Memory`는 협업 도구 인터페이스가 강하다
- `Graphiti`는 재사용 가능한 엔진 인터페이스가 강하다

### 우리 제품 시사점
- 사용자가 바로 쓰는 앱을 만들려면 `AnythingLLM`
- 사람과 AI의 공동 편집 흐름을 만들려면 `Basic Memory`
- 여러 채널에서 재사용할 메모리 코어가 필요하면 `Graphiti`

## 9. 운영 안정성, 권한, 확장성

운영 모델을 보면 세 제품이 실제로 어디에 강한지 더 선명하게 드러난다.

### `AnythingLLM`
- 운영 중심: 워크스페이스 격리와 전역 역할
- 확장 장치: provider adapter(공급자 교체용 추상화), 임베드 설정, 외부 삽입 경로
- 특징: 문서 단위 ACL보다 워크스페이스 단위 경계가 중심

### `Basic Memory`
- 운영 중심: 로컬 우선 저장과 파일 소유권
- 확장 장치: `doctor`(정합성 점검 루프), 파일 감시기(watcher), 프로젝트별 local/cloud routing(같은 도구 호출을 유지한 채 실행 경로만 나누는 방식)
- 특징: 사용자의 저장소를 침범하지 않으면서 정합성을 유지

### `Graphiti`
- 운영 중심: 시간 일관성과 멀티 provider 대응
- 확장 장치: 동시성 제어, tracing(실행 추적), telemetry(운영 관측 정보), provider 추상화
- 특징: group 단위 순차 처리(같은 메모리 묶음을 시간 순서대로 처리하는 규칙)처럼 메모리 품질을 위한 운영 규칙이 강함

### 비교 해석
- `AnythingLLM`은 제품 운영 경계와 권한 모델이 강하다
- `Basic Memory`는 저장소 정합성과 사용자 소유권을 함께 지키는 운영 모델이 강하다
- `Graphiti`는 멀티 provider와 시간 일관성을 버티는 운영 장치가 강하다

### 우리 제품 시사점
- 기업 내부 배포와 워크스페이스 운영이 핵심이면 `AnythingLLM`
- 개인 또는 팀 단위 지식 저장소와 로컬 소유권이 핵심이면 `Basic Memory`
- 에이전트 메모리를 여러 환경에서 운영해야 하면 `Graphiti`

## 우리 제품 관점의 해석

세 제품을 종합하면, 우리 제품은 어느 제품을 그대로 복제할지보다 어느 층을 중심축으로 삼을지 먼저 정해야 한다.

### 핵심 판단
- 문서 수집과 운영이 먼저면 `AnythingLLM` 축이 중심이 됨
- 파일 원본과 장기 지식 저장이 먼저면 `Basic Memory` 축이 중심이 됨
- 시간성 있는 사실 관리와 에이전트 메모리가 먼저면 `Graphiti` 축이 중심이 됨
- 현재 문서 기반 지식 베이스를 만드는 단계라면 `AnythingLLM`과 `Basic Memory`의 조합이 더 현실적임

### 1. 바로 가져올 수 있는 요소

- `AnythingLLM`의 공통 문서 계약과 저장 계층 분리
- `Basic Memory`의 파일 원본과 인덱스 분리
- `Basic Memory`의 `permalink` 중심 추적성
- `Graphiti`의 provenance(출처 역추적 정보)를 일급 객체로 다루는 태도
- 검색과 문맥 조립을 분리하는 설계 원칙

### 2. 조합해서 가져와야 할 요소

- `AnythingLLM`의 문서 수집 파이프라인과 `Basic Memory`의 로컬 우선 저장 구조
- `Basic Memory`의 사람 친화적 지식 표현과 `Graphiti`의 구조화된 문맥 조립
- `AnythingLLM`의 기업형 운영 경계와 `Graphiti`의 메모리 provenance(출처 역추적 정보)

### 3. 지금 단계에서 과할 수 있는 요소

- 초기 단계에서의 완전한 temporal fact graph
- 모든 provider를 같은 수준으로 지원하려는 과도한 추상화
- 고비용 reranker(검색 결과 재정렬기)와 복잡한 그래프 탐색의 조기 도입

### 4. 현재 기준의 권장 방향

- 추천 출발점: `AnythingLLM` + `Basic Memory`
- 이유 1: 문서 수집과 저장 계약은 `AnythingLLM`이 더 실전적임
- 이유 2: 원본 소유권과 파일 기반 운영 원칙은 `Basic Memory`가 더 선명함
- 확장 방향: 장기적으로 에이전트 메모리까지 가려면 `Graphiti`의 provenance(출처 역추적 정보), 상태 전이, 검색-문맥 분리 원칙을 점진적으로 흡수하는 편이 좋음
- 보류 판단: 지금 단계에서 `Graphiti` 전체 구조를 바로 도입하는 것은 복잡도 대비 이득이 작을 수 있음

# 결론

### 핵심 요약
- `AnythingLLM`은 문서 운영 제품에 가깝다
- `Basic Memory`는 지식 저장 인프라에 가깝다
- `Graphiti`는 에이전트 메모리 엔진에 가깝다

### 우리 제품이 지금 내려야 할 판단
- 먼저 문서 수집과 저장 계약의 경계를 정해야 함
- 다음으로 원본 소유권과 인덱스 구조를 정해야 함
- 그다음에 검색과 문맥 조립 방식을 정해야 함
- 마지막 단계에서 그래프와 시간성, 에이전트 메모리를 얹는 것이 자연스러움

### 주의할 점
- 이 순서를 거꾸로 잡으면 구조는 화려해져도 제품은 불안정해질 가능성이 큼
- 특히 초기 단계에서 메모리 엔진 수준의 복잡도를 먼저 들이면 문서형 지식 베이스의 기본기가 약해질 수 있음
