# TCI C4 Diagram 검토 방법

## 목적

이 문서는 TCI의 C1, C2, C3 다이어그램을 제품 문서와 대조해 검토할 때 사용하는 공통 방법을 정리한 가이드다.

목표는 다음 두 가지다.

- 다이어그램이 제품 문서의 의도와 범위를 정확히 반영하는지 검증
- 다른 에이전트가 같은 기준과 순서로 재검토할 수 있게 절차를 표준화

## 기본 원칙

- 다이어그램의 미려함이 아니라 `문서 정합성`을 검토
- `좋은 설계인지`보다 `문서와 맞는 설계인지`를 먼저 본다
- Notion 설명과 로컬 `puml`을 따로 본다
- 둘이 다르면 `설명 문서와 소스 간 불일치`로 별도 기록
- 문서에 없는 기능을 다이어그램이 앞서 표현하면 `과잉 해석`으로 본다
- 문서에 있는데 다이어그램에 없으면 `누락`으로 본다
- C4 레벨이 올라갈수록 `모든 기능을 다 그렸는지`보다 `해당 레벨에 필요한 책임과 경계가 맞는지`를 본다

## 검토 입력

각 다이어그램 검토 시 반드시 아래 두 가지를 함께 본다.

- Notion 페이지
- 로컬 `puml` 파일

이유는 다음과 같다.

- Notion에는 설계 판단, 설명, 경계 근거가 있음
- `puml`은 실제 다이어그램 소스이며 종종 Notion과 다를 수 있음

## 검토 절차

### 1. 대상 확인

먼저 정확한 Notion 페이지를 찾는다.

- 검색 결과 제목이 요청한 페이지와 정확히 일치하는지 확인
- 유사한 제목의 다른 페이지를 잘못 읽지 않도록 주의

그다음 로컬 `puml` 파일을 읽는다.

### 2. Notion 설명과 `puml`을 먼저 비교

제품 문서와 대조하기 전에, 같은 다이어그램의 두 표현이 서로 맞는지 본다.

우선 확인할 것:

- 컴포넌트/컨테이너/외부 시스템 개수
- 책임 설명
- 관계 방향
- 빠진 연결선
- Notion에는 있는데 `puml`에 없는 요소
- `puml`에는 있는데 Notion 설명에는 없는 요소

이 단계에서 다르면 바로 기록한다.

예시 유형:

- Notion은 `IDE Plugin 제거`라고 쓰는데 `puml`에는 남아 있음
- Notion은 외부 시스템 5개라고 쓰는데 `puml`에는 6개가 있음
- Notion은 `내부 컴포넌트 10개`라고 쓰는데 실제 목록은 11개임

### 3. 제품 문서와 대조

그다음 다이어그램을 제품 문서와 대조한다.

이때 보는 기준은 3가지다.

- 누락
- 오해
- 과잉 해석

#### 누락

문서에 중요한데 다이어그램에 없는 경우

예시:

- `파일 업로드`
- `Policy Gate 외부 시스템`
- `직접 업로드 경로`
- `문서화/질의응답 정체성`

#### 오해

문서의 의도는 맞지만 역할이나 책임을 잘못 표현한 경우

예시:

- 사용자가 `Web Application`을 거쳐야 하는데 특정 백엔드 컨테이너에 직접 붙어 있음
- `파일 업로드`가 외부 시스템 입력처럼 그려짐
- `KB Loader`가 중앙 적재자처럼 설명되지만 실제로는 다른 컴포넌트가 직접 적재함

#### 과잉 해석

문서에는 없는 동작을 설계가 먼저 넣은 경우

예시:

- Jira에 분석 코멘트 write-back
- 문서/티켓 시스템의 webhook 수신
- 외부 위키 발행을 확정 기능처럼 표현

### 4. C4 레벨에 맞는지 확인

같은 누락이라도 레벨에 따라 판단이 다르다.

#### C1

C1에서는 아래를 본다.

- 사용자 종류가 맞는지
- 시스템의 핵심 정체성이 맞는지
- 외부 시스템 경계가 맞는지
- 중요한 입력/출력 채널이 빠지지 않았는지

C1에서 굳이 안 넣어도 되는 것:

- 내부 엔진 세부 기능
- 저장소 종류
- 세션, 캐시, 큐 같은 구현 세부

#### C2

C2에서는 아래를 본다.

- 컨테이너 분해가 제품 흐름과 맞는지
- 컨테이너 책임이 과도하게 겹치지 않는지
- 외부 시스템과의 연결점이 정확한지
- 데이터 흐름이 현실적인지

C2에서 자주 보는 문제:

- 외부 시스템이 사라짐
- 사용자 직접 업로드 경로가 틀어짐
- 저장소 종류가 일관되지 않음
- 알림/이벤트 반환 경로가 빠짐

#### C3

C3에서는 아래를 본다.

- 컴포넌트 책임이 문서 기능과 맞는지
- 실제로 동작하려면 필요한 입력/의존 관계가 그려졌는지
- 상위 C2와 경계가 맞는지
- 각 컴포넌트가 읽고 쓰는 대상이 논리적으로 맞는지

C3에서 자주 보는 문제:

- 책임은 있는데 입력 관계가 없음
- 저장 책임이 여러 컴포넌트에 흩어짐
- 상위 다이어그램에서 정한 진입 경계를 깨뜨림
- 설명에는 있는데 관계선이 없음

## 레벨별 체크리스트

### C1 체크리스트

- 핵심 사용자 3종이 맞는가
- TCI의 제품 정체성이 `분석 레이어 + 컨텍스트 + 문서화 + 비즈니스 로직`으로 드러나는가
- Git, Ticket, Docs/Wiki, CI/CD, AI Agent, Policy Engine 같은 주요 외부 시스템이 반영됐는가
- 직접 업로드와 같은 중요한 입력 방식이 지나치게 사라지지 않았는가
- Notion 설명과 `puml`이 같은 설계를 가리키는가

### C2 체크리스트

- `수집 -> 처리 -> 분석 -> 지식 저장 -> 질의/자동화` 흐름이 자연스러운가
- Knowledge Base가 단일 지식 모델 방향과 맞는가
- 외부 시스템이 적절히 분리되어 있는가
- 사용자 진입이 Web Application을 경유하는가
- 파일 업로드, Policy Engine, 알림 푸시처럼 사용자 흐름을 닫는 요소가 빠지지 않았는가

### C3 체크리스트

- 각 컴포넌트가 문서의 기능 항목과 매핑되는가
- 입력 관계가 실제로 존재하는가
- 출력이 실제 어디에 저장/전달되는지 보이는가
- 상위 C2에서 정한 경계를 깨지 않는가
- Notion 설명의 책임과 `puml`의 관계선이 서로 맞는가

## 특히 자주 확인한 항목

TCI 검토에서는 아래 항목을 반복적으로 확인했다.

### 경계

- 사용자는 항상 `Web Application`을 통해 진입하는가
- 외부 `Policy Engine`이 별도 외부 시스템으로 유지되는가
- AI Agent, IDE Plugin, 직접 업로드 경로가 서로 충돌하지 않는가

### 입력

- Diff 입력이 어디서 오는가
- 문서/티켓/업로드 파일이 어떤 파이프라인을 타는가
- webhook, 스케줄, 수동 실행의 진입점이 분리되어 있는가

### 저장

- Graph, Vector, Object, Search의 역할이 명확한가
- 누가 인덱스를 갱신하는가
- 대화 이력, 평가, 문서화 답변이 어디에 저장되는가
- 스냅샷과 버전 이력이 어디서 관리되는가

### 출력

- PR 본문, 문서 초안, 리포트가 실제 어디에 쓰이는가
- 알림이 실제 어느 채널로 가는가
- Web Application으로의 알림 반환 경로가 있는가

## Findings 작성 방식

리뷰 문서는 아래 구조를 유지한다.

### 1. 총평

짧게 한 줄로 평가한다.

예시:

- `현재 다이어그램은 문서와 대체로 일치`
- `큰 방향은 맞지만 직접 충돌이 있음`

### 2. 핵심 findings

가장 중요한 것부터 적는다.

각 finding에는 아래를 포함한다.

- 무엇이 문제인지
- 어떤 문서와 충돌하는지
- 왜 문제인지
- 권고안

### 3. 문서 대비 잘 맞는 부분

부정적 피드백만 쓰지 않는다.

- 다이어그램이 문서와 잘 맞는 부분도 명시
- 이후 수정 시 유지해야 할 설계를 보호하는 역할

### 4. 결론

항상 아래 두 가지를 남긴다.

- 현재 기준 판단
  - `부분적으로 일치`
  - `대체로 일치`
  - `직접 충돌 존재`
- 우선순위 높은 수정 항목 3~5개

## 표현 규칙

- 리뷰는 `설계 비판`이 아니라 `문서 정합성 검토`라는 톤 유지
- 모호하면 `가능한 해석`을 먼저 적고 단정은 피함
- 레벨이 다른 다이어그램 간 충돌은 별도 지적
- `오류`와 `확장 포인트`를 구분
- 다이어그램이 반드시 다 담아야 하는 것과, 상위 레벨이라 생략 가능한 것을 구분

## 권장 파일 구조

각 검토 결과는 다이어그램 폴더에 아래처럼 둔다.

- `01_context-diagram/context-diagram-review.md`
- `02_container-diagram/container-diagram-review.md`
- `03_component-diagram/<container-name>/<name>-review.md`

검토 방법 문서는 이 파일처럼 `architecture` 루트에 둔다.

## 재검토 시 권장 순서

다시 검토할 때는 아래 순서를 권장한다.

1. 제품 문서 3종 읽기
2. C1 검토
3. C2 검토
4. C3 검토
5. 이전 레벨과의 정합성 재확인
6. 공통 이슈 묶기

이 순서를 지키면 아래 문제를 빨리 잡을 수 있다.

- C1에서 제거된 요소가 C2/C3에는 살아 있음
- C2에서 외부 시스템이 사라졌는데 C3에는 어댑터가 남아 있음
- C3에서 사용자가 백엔드에 직접 붙어 있음

## 마지막 확인 질문

리뷰를 끝내기 전에 아래 질문에 답할 수 있어야 한다.

- 이 다이어그램은 문서와 같은 제품을 설명하고 있는가

- 문서에 없는 설계를 앞서 가정하고 있는가
- 상위 레벨 다이어그램과 모순이 없는가
- 다른 에이전트가 이 리뷰를 읽고 수정 우선순위를 바로 이해할 수 있는가
