# **제품 포지셔닝**

## **1. 제품 포지셔닝**

## **1.1 정의 및 역할**

**Engineering Context Platform + Change Intelligence Platform**

with **Business Logic Intelligence** and **Documentation Automation**

> **정의**
> 
> 
> Change Intelligence: 코드 변경이 시스템에 미치는 영향/리스크를 분석해서 변경 판단 근거를 생성
> 
> Engineering Context Platform: 코드·문서·이슈·정책 등 다양한 개발 자산을 연결, 의사 결정 정보 제공
> 
> Business Logic Intelligence: TCI 고유 차별점 강화
> 
> Documentation Automation: 결과물을 실제 업무 흐름으로 연결
> 

> **역할**
> 
> 
> 개발자뿐 아니라 리뷰어, 운영/기획 등 비개발자에게도 이해 가능한 형태로 변경 근거와 설명을 정리해주는 Engineering Context Layer 역할 수행
> 
> 또한 AI Agent가 쓸 수 있는 형태의 컨텍스트 제공, 코드 생성/수정 시 코드베이스 이해도 향상 지원
> 

## **1.2 포지셔닝 구조**

### **Engineering Context Platform**

코드, 문서, 이슈, 정책, 변경 이력 등 다양한 개발 자산을 연결하여 시스템 이해와 의사결정에 필요한 맥락 정보를 제공하는 플랫폼

### **핵심 가치**

- 개발 맥락의 분산 문제 해소
- 코드와 비코드 자산을 연결한 단일 탐색 경로 제공
- 사람과 AI가 모두 활용 가능한 구조화된 컨텍스트 제공

### **Change Intelligence Platform**

코드 구조, 의존 관계, 변경 단위를 기준으로 시스템 영향 범위와 리스크를 분석하여 변경 판단 근거를 제공하는 플랫폼

### **핵심 가치**

- 변경 영향 판단의 경험 의존성 감소
- 리뷰와 승인 전 리스크 식별
- 영향 범위 기반의 검토 생산성 향상

### **Business Logic Intelligence**

코드에 구현된 비즈니스 규칙, 흐름, 조건 분기, 정책 로직을 분석해 사람이 이해 가능한 형태로 정리하는 기능

### **핵심 가치**

- 복잡한 레거시 시스템의 업무 규칙 이해 지원
- PM/PO와 개발자 간 구현 이해 간극 축소
- 요구사항 변경과 구현 영향의 근거 기반 검토 지원

### **Documentation Automation**

분석 결과를 설명 가능한 산출물로 변환하여 리뷰, 승인, 공유, 보고, 인수인계에 활용할 수 있게 하는 기능

### **핵심 가치**

- 분석 결과의 실무 활용성 강화
- 리뷰 리포트, 설명 자료, 문서 초안 자동화
- 개인 지식의 조직 자산화

## **2. 포지셔닝 근거**

### **2.1 개발 맥락 분산**

개발 조직은 코드 외에도 Jira, Confluence, 설계 문서, 운영 문서, 변경 이력 등 다양한 자산 수집

하지만 이 정보들은 서로 다른 도구에 흩어져 있어, 실제 변경 판단은 여전히 개인 경험과 암묵지에 의존하는 경우가 많음

### **2.2 AI는 생성은 잘하지만 맥락이 부족**

GitHub Copilot, Cursor, Codeium 같은 AI 기반 개발 도구 빠르게 확산 중

하지만 대규모 코드베이스의 구조, 서비스 간 의존성, 비즈니스 규칙, 정책 문맥까지 정확히 이해하는 것은 한계가 있음

### **2.3 엔터프라이즈 환경에서 중요한 지점, 변경 판단**

복잡한 시스템일수록 중요한 것은 단순 코드 작성 속도가 아니라 변경 판단이 더 중요

# **4. TCI 참고 제품 및 참고 관점**

TCI는 각 영역의 대표 제품을 참고하되, 특정 기능 하나를 모방하는 것이 아니라 개발 자산 통합, 구조 분석, 변경 판단, 설명 가능한 결과 제공을 하나의 흐름으로 연결하는 방향으로 차별화

### **4.1 데이터 소스 통합(Context) / AI 컨텍스트 주입**

코드, 문서, 이슈, 외부 지식 등 분산된 개발 자산을 연결하거나, AI가 활용할 수 있는 형태로 컨텍스트를 주입하는 제품군

### **대표 레퍼런스**

- Sourcegraph Cody
- Port

### **시장 특성**

- 코드베이스와 문서, 이슈, 외부 데이터 등을 연결해 개발 맥락을 제공
- AI 질의응답, 코드 생성, 탐색 과정에서 더 정확한 응답을 유도
- 조직 자산을 엔티티와 관계 중심으로 통합 관리하는 UX 제공

### **참고 포인트**

- Sourcegraph Cody: AI 작업 시점의 컨텍스트 주입 방식
- Port: 서비스, 팀, 이슈, 인시던트 등 조직 자산을 관계형으로 통합하는 방식

### **TCI와의 차이**

이 제품군은 컨텍스트 연결에는 강하지만, 코드 변경이 시스템에 미치는 영향과 리스크를 분석해 변경 판단 근거를 제공하는 기능은 제한적이다.

TCI는 컨텍스트 통합을 넘어 변경 영향 분석과 판단 지원까지 연결하는 것을 목표로 한다.

### **4.2 코드 구조 분석 / 아키텍처 시각화**

코드 구조, 호출 관계, 의존성, 서비스 관계, 시스템 아키텍처를 분석하고 시각적으로 탐색하게 해주는 제품군

### **대표 레퍼런스**

- CAST Imaging
- CodeSee
- CodeCompass
- Structurizr
- Backstage

### **시장 특성**

- 코드 구조와 시스템 관계를 시각적으로 보여줘 복잡한 코드베이스 이해를 지원
- 서비스 맵, 코드베이스 맵, 호출 관계, 아키텍처 다이어그램 등 다양한 형태의 구조 가시화 제공
- 온보딩, 리뷰, 리팩토링, 구조 설명에 유용

### **참고 포인트**

- CAST Imaging: 엔터프라이즈 시스템 구조와 의존성 가시화
- CodeSee: 코드베이스 맵, 서비스 맵, PR 변경 시각화 UX
- CodeCompass: 정적 구조 탐색, 호출/참조/상속 분석
- Structurizr: C4 기반 계층형 아키텍처 설명 UX
- Backstage: 서비스/시스템/문서 관계를 포털 안에서 연결해 보여주는 UX

### **TCI와의 차이**

이 제품군은 구조를 보여주는 데 강하지만, 대부분 문서·이슈·정책·변경 이력까지 결합한 맥락 제공이나 변경 의사결정용 영향 분석, 비즈니스 로직 설명까지는 충분히 포괄하지 않는다.

TCI는 구조 분석을 기반으로 실제 검토와 판단에 필요한 근거 정보까지 제공하는 방향으로 차별화한다.

### **3.3 코드 이해 / 문서화**

코드 지식과 구조 이해 결과를 문서 형태로 정리하고, 팀 내 공유와 온보딩에 활용할 수 있게 해주는 제품군

### **대표 레퍼런스**

- Swimm

### **시장 특성**

- 코드와 설명 문서를 연결해 지식 전달과 인수인계를 지원
- 특정 기능, 모듈, 흐름에 대한 설명 자산을 축적하는 데 강점
- 개발자 온보딩과 코드베이스 이해 지원에 적합

### **참고 포인트**

- 코드 이해 결과를 문서와 연결하는 방식
- 단순 분석 화면이 아니라, 재사용 가능한 지식 자산으로 축적하는 방식
- 사람 중심의 설명 경험과 온보딩 UX

### **TCI와의 차이**

Swimm과 같은 제품은 코드 지식의 문서화와 전달에는 강점이 있지만, TCI는 여기서 더 나아가 코드 구조 분석, 변경 영향 분석, 리뷰 근거 제공, AI 컨텍스트 생성까지 함께 제공하는 방향을 가진다.

즉 TCI는 문서화 자체보다, 분석 결과를 문서와 설명 가능한 산출물로 연결하는 것에 더 초점을 둔다.

---

### **3.4 PR 리뷰**

Pull Request 단위의 변경 내용을 요약하고, 리뷰 포인트와 잠재 리스크를 제시해 리뷰 생산성을 높이는 제품군

### **대표 레퍼런스**

- CodeRabbit

### **시장 특성**

- PR 리뷰 자동화
- 리뷰 코멘트 생성 및 누락 가능 영역 식별
- 리뷰어의 검토 속도와 생산성 향상

### **참고 포인트**

- PR 단위 요약 UX
- 검토 포인트 제안 방식
- 변경 내용을 리뷰 가능한 형태로 구조화하는 방식

### **TCI와의 차이**

CodeRabbit은 PR 리뷰 단계의 생산성 향상에 초점이 있다. 반면 TCI는 리뷰 단계에만 머무르지 않고,

변경 전 구조 이해 -> 영향 분석 -> 관련 문서/이슈 연결 -> 리뷰 근거 제공까지 포함하는 더 넓은 흐름을 지원한다.

## **4. TCI의 차별화 방향**

### **4.1 단일 기능 도구가 아니라 변경 판단 중심의 통합 제품**

기존 제품은 검색, 문서화, 구조 시각화, 리뷰 자동화 등 각기 다른 문제 초점

반면 TCI는 개발 자산 통합 -> 구조 이해 -> 변경 영향 분석 -> 검토/설명 산출물 생성을 하나의 흐름으로 연결.

### **4.2 코드 자체보다 “변경 의사결정”에 더 집중**

전통적인 정적 분석 도구는 코드 품질과 보안 규칙에 집중하고, 코드 검색 제품은 탐색 효율에 집중

TCI는 그보다 더 직접적으로 “이 변경이 안전한가, 어디를 더 봐야 하는가”라는 질문에 답하는 데 집중

### **4.3 사람과 AI를 동시에 지원**

Sourcegraph Cody 같은 제품은 AI 활용에 강하고, Port나 Backstage는 사람 중심 포털에 강함

TCI는 개발자, 리뷰어, PM/PO가 보는 근거 정보와 AI Agent가 활용할 수 있는 컨텍스트를 함께 제공하는 방향

### **4.4 구조 분석을 넘어 비즈니스 맥락까지 연결**

CodeSee, CodeCompass, CAST Imaging은 구조 분석과 시각화에는 강점이 있지만,

문서, 이슈, 요구사항, 정책, 비즈니스 규칙까지 함께 연결해 변경 판단에 활용하는 데는 한계가 있음

TCI는 코드 구조와 비즈니스 맥락을 함께 연결하는 것이 차별화 포인트

### **4.5 분석 결과를 실무 산출물로 전환**

많은 분석 도구는 결과를 화면에서 보여주는 데 그침

TCI는 분석 결과를 리뷰 리포트, 설명 자료, 문서 초안, 근거 링크, 질의응답 형태로 연결해 실제 리뷰, 승인, 보고, 요구사항 검토까지 이어지게 하는 방향
