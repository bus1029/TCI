# 기능 설계 착수 판단 기준

## 목적

이 문서는 기능 설명과 상세 기능 목록만 주어진 상태에서, 해당 기능이 바로 설계 단계로 넘어갈 수 있는지 판단하는 기준을 정리한다.

여기서 말하는 설계 단계는 다음을 포함한다.

- API 설계
- 데이터 모델 설계
- 컴포넌트 분해
- 이벤트 및 동기화 시퀀스 설계
- 보안 및 운영 정책 설계

기능 목록이 아래 기준을 충족하지 못하면, 개념 설계나 아키텍처 탐색은 가능하더라도 상세 설계와 구현 착수는 보류하는 것이 맞다.

## 판단 원칙

- 기능 카탈로그와 설계 명세는 다르다
- `무엇을 할지`만 있으면 기능 목록이다
- `어떻게 동작할지`, `어디까지 지원할지`, `어떤 제약이 있는지`까지 있어야 설계 입력이 된다
- 설계 단계에서 열린 질문이 많으면 구현보다 재해석 비용이 커진다

## 설계 착수 가능 수준

### 1. 기능 목록 수준

다음 정도만 정의된 상태다.

- 기능명
- 한 줄 설명
- 서브 기능 목록

이 수준에서는 가능한 작업이 제한된다.

- 가능: 기능 분해, 대략적 시스템 구성 검토, 후보 아키텍처 비교
- 불가: API 확정, DB 스키마 확정, 작업 큐 설계 확정, 운영 시나리오 확정

### 2. 설계 입력 수준

다음이 함께 정의된 상태다.

- 범위
- 사용자 시나리오
- 핵심 엔티티
- 상태 전이
- 외부 연동 방식
- 실패 처리 방식
- 비기능 요구사항
- 완료 기준

이 수준이면 상세 설계에 착수할 수 있다.

## 설계 착수 판단 기준

### 1. 지원 범위가 명확한가

확인할 내용

- 어떤 외부 시스템을 지원하는가
- MVP 범위와 제외 범위가 무엇인가
- SaaS만 지원하는지 self-hosted도 포함하는가
- 특정 공급자만 우선 지원하는가

없으면 생기는 문제

- 연동 방식이 기능마다 달라짐
- 인증 모델과 API 제약을 설계할 수 없음
- 구현 난이도와 일정 산정이 왜곡됨

### 2. 관리 단위와 사용자 대상이 명확한가

확인할 내용

- 사용자가 무엇을 연결하는가
- 관리 단위가 조직, 프로젝트, 저장소, 브랜치 중 무엇인가
- 누가 이 기능을 사용하는가
- 사용자 역할별 권한 차이가 있는가

없으면 생기는 문제

- 도메인 모델이 흔들림
- UI 흐름과 권한 모델을 설계할 수 없음

### 3. 사용자 시나리오와 성공 조건이 정의돼 있는가

확인할 내용

- 사용자가 처음 기능을 사용하는 흐름
- 정상 완료의 기준
- 재시도와 실패 복구 흐름
- 사용자가 확인하게 될 결과 화면 또는 상태

없으면 생기는 문제

- 기능이 연결돼도 무엇이 완료인지 판단할 수 없음
- UX와 백엔드 상태 모델이 분리됨

### 4. 핵심 엔티티와 상태 전이가 정의돼 있는가

확인할 내용

- 어떤 엔티티가 필요한가
- 각 엔티티의 주요 속성은 무엇인가
- 상태가 어떻게 바뀌는가
- 어떤 이벤트가 상태 전이를 유발하는가

예시 엔티티

- Repository
- Branch
- Snapshot
- SyncJob
- WebhookEvent
- Credential

없으면 생기는 문제

- DB 스키마와 API 리소스를 설계할 수 없음
- 비동기 처리 상태를 추적할 수 없음

### 5. 데이터 수집과 처리 방식이 정의돼 있는가

확인할 내용

- 어떤 데이터를 수집하는가
- 수집 시점이 언제인가
- 전체 수집과 증분 수집을 어떻게 구분하는가
- 원본 데이터와 가공 데이터의 저장 기준은 무엇인가

없으면 생기는 문제

- 스냅샷 정의가 사람마다 달라짐
- 저장 비용과 처리 비용을 예측할 수 없음

### 6. 외부 연동 방식이 정의돼 있는가

확인할 내용

- 연동 방식이 OAuth, API Token, SSH Key 중 무엇인가
- Webhook, Polling, 수동 실행 중 무엇을 쓰는가
- 공급자별 차이를 어떻게 추상화할 것인가
- 외부 API 제한과 장애를 어떻게 다룰 것인가

없으면 생기는 문제

- 커넥터 구조를 설계할 수 없음
- 운영 중 유실과 중복 처리가 빈번해짐

### 7. 웹과 IDE 동기화의 의미가 명확한가

확인할 내용

- 웹과 IDE 중 어디서 연결을 생성하는가
- 어느 쪽이 원본 상태를 가진가
- 어떤 데이터를 양방향으로 동기화하는가
- 충돌 시 우선순위는 무엇인가

없으면 생기는 문제

- 같은 기능을 두 채널에서 다르게 해석함
- 권한, 상태, 캐시 정책이 충돌함

### 8. 보안과 인증 정책이 정의돼 있는가

확인할 내용

- 인증 정보를 어디에 저장하는가
- 암호화와 복호화 정책은 무엇인가
- 키 회전과 만료 처리는 어떻게 하는가
- 최소 권한 원칙을 어떻게 적용하는가

없으면 생기는 문제

- 연동 기능을 구현해도 운영 승인 단계에서 막힘
- 민감 정보 저장 구조를 다시 뜯어고쳐야 함

### 9. 실패 처리와 운영 정책이 정의돼 있는가

확인할 내용

- 웹훅 유실 시 복구 방식
- 중복 이벤트 수신 시 처리 정책
- 장시간 동기화 작업의 타임아웃 기준
- 재시도 횟수와 백오프 정책
- 운영자가 확인할 로그와 모니터링 지표

없으면 생기는 문제

- 기능은 동작해도 운영 불가능한 상태가 됨
- 장애 재현과 복구가 어려워짐

### 10. 비기능 요구사항이 정의돼 있는가

확인할 내용

- 최대 저장소 수
- 최대 저장소 크기
- 동시 처리량
- 허용 가능한 동기화 지연 시간
- 응답 시간 기대치

없으면 생기는 문제

- 설계가 과소 또는 과대해짐
- 큐, 워커, 저장소 구조를 결정할 근거가 없음

### 11. 완료 기준과 검증 기준이 정의돼 있는가

확인할 내용

- 연결 성공을 무엇으로 판단하는가
- 스냅샷 생성 완료를 무엇으로 판단하는가
- 이벤트 반영 성공 여부를 어떻게 검증하는가
- 실패 시 사용자에게 어떤 상태를 보여주는가

없으면 생기는 문제

- 테스트 케이스를 정의할 수 없음
- 구현 완료 여부를 합의할 수 없음

## 빠른 체크리스트

아래 질문에 대부분 `예`라고 답할 수 있어야 상세 설계에 들어갈 수 있다.

- 지원 대상 Git 공급자가 정의돼 있는가
- MVP 포함 범위와 제외 범위가 정의돼 있는가
- 사용자와 관리 단위가 정의돼 있는가
- 대표 사용자 시나리오가 정의돼 있는가
- 주요 엔티티와 상태 전이가 정의돼 있는가
- 스냅샷의 의미가 정의돼 있는가
- 이벤트 수집 방식이 정의돼 있는가
- 웹과 IDE 동기화 범위가 정의돼 있는가
- 인증 정보 저장 및 암호화 정책이 정의돼 있는가
- 재시도, 중복 처리, 복구 정책이 정의돼 있는가
- 성능과 용량 기준이 정의돼 있는가
- 완료 조건과 검증 기준이 정의돼 있는가

`예`가 9개 이상이면 상세 설계 가능, 6개에서 8개면 보완 후 설계, 5개 이하면 기능 목록 단계로 본다.

### 현재 상태 평가

이미 있는 정보

- Git 기반 저장소를 연동한다는 목적
- SSH/HTTPS 연결이 필요하다는 점
- 브랜치/태그 선택이 필요하다는 점
- Commit, Push, PR 이벤트를 감지한다는 점
- 커밋 메타데이터와 코드 스냅샷이 필요하다는 점
- 멀티 레포 관리가 필요하다는 점

부족한 정보

- GitHub, GitLab, Bitbucket, self-hosted 중 지원 대상
- 연결 단위와 권한 모델
- 스냅샷 정의
- Webhook 유실 복구 방식
- 웹과 IDE 동기화의 정확한 범위
- 인증 정보 저장 및 회전 정책
- 최대 레포 크기와 동기화 지연 목표
- 연결 성공과 동기화 완료의 판정 기준

판단

- 아키텍처 탐색 가능
- 상세 설계 착수는 보류
- 요구사항 보강이 먼저 필요
