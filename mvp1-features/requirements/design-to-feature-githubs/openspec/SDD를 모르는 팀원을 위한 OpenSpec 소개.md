# SDD를 모르는 팀원을 위한 OpenSpec 소개

- 문서 목적: SDD와 `Fission-AI/OpenSpec`을 처음 접하는 팀원이 핵심 개념만 빠르게 이해하도록 설명
- 대상 독자: 내일 공유 세션에 참석하는 TCI 팀원

## 한 줄 정의

OpenSpec은 기능 변경을 바로 구현으로 넘기지 않고, 먼저 변경 제안과 요구사항 변화, 설계, 작업 목록을 문서로 정리하게 만드는 오픈소스 워크플로우다.

이 문서를 끝까지 읽고 기억해야 할 핵심은 하나다. OpenSpec은 코드를 대신 써주는 도구라기보다, 기존 시스템에 어떤 변경을 넣을지 추적 가능한 문서로 바꾸는 틀이다.

## 먼저 SDD가 무엇인가

SDD는 `Spec-Driven Development`의 약자다. 쉽게 말하면 코드를 먼저 쓰는 대신, 명세를 먼저 만들고 그 명세를 기준으로 설계와 구현을 진행하는 방식이다.

핵심 흐름은 단순하다.

- 무엇을 왜 바꾸는가
- 무엇이 달라지는가
- 어떻게 구현할 것인가
- 어떤 작업 순서로 진행할 것인가

즉, 구현 전에 팀이 같은 기준 문서를 보고 움직이게 만드는 방식이라고 보면 된다.

## 왜 이런 방식이 필요한가

기능 논의가 바로 구현으로 넘어가면 보통 아래 문제가 생긴다.

- 변경 이유는 알겠는데 정확히 무엇이 바뀌는지 문서가 남지 않음
- 기존 동작과 새 동작이 뒤섞여서 범위가 흔들림
- 설계 없이 구현하다가 영향 범위를 뒤늦게 발견함
- 작업은 시작됐는데 완료 기준이 모호함

OpenSpec은 이 문제를 줄이기 위해, 구현 전에 변경 단위를 문서로 고정한다.

## OpenSpec의 핵심 흐름

OpenSpec의 기본 흐름은 아래처럼 이해하면 된다.

`propose → apply → archive`

TCI에서 당장 중요하게 볼 것은 `propose` 단계다. 이 단계에서 이미 핵심 문서가 한 번에 만들어진다.

| 단계 | 이 단계에서 답하는 질문 |
| --- | --- |
| `propose` | 이번에 무엇을 왜 바꾸는가 |
| `apply` | 이 변경을 실제 코드와 테스트에 어떻게 반영할 것인가 |
| `archive` | 완료된 변경을 현재 기준선에 어떻게 반영할 것인가 |

필요하면 그 사이에 `explore`, `verify`, `sync` 같은 단계를 더 넣을 수 있다. 하지만 발표에서는 `propose → apply → archive`만 이해해도 충분하다.

## OpenSpec은 무엇이 다른가

Spec Kit과 비교했을 때 OpenSpec의 가장 큰 차이는 "새 문서를 만드는 도구"라기보다 "기존 기준선 대비 변경을 관리하는 도구"라는 점이다.

핵심 구조는 아래 두 층이다.

- `openspec/specs/`
  - 현재 시스템이 따라야 하는 기준선
- `openspec/changes/`
  - 아직 제안 중이거나 구현 중인 변경 묶음

즉, OpenSpec은 현재 기준선과 새 변경안을 분리해서 관리한다. 이 점이 기존 기능 수정이 많은 프로젝트에서 특히 유용하다.

## 산출물은 이것만 기억하면 된다

발표에서는 세부 명령보다 아래 문서 역할만 기억하면 충분하다.

| 문서 | 역할 |
| --- | --- |
| `proposal.md` | 왜 이 변경이 필요한지 정리 |
| `spec.md` | 어떤 요구사항이 추가되거나 수정되는지 정리 |
| `design.md` | 어떻게 구현할지 정리 |
| `tasks.md` | 실제 구현 작업 목록 정리 |

여기서 가장 중요한 것은 `spec.md`다. OpenSpec에서는 이 문서가 기존 기준선 대비 무엇이 달라지는지를 보여 주는 핵심 계약 역할을 한다.

## OpenSpec에서 꼭 알아야 할 개념 하나

OpenSpec은 요구사항 변화를 `delta spec`으로 관리한다. 쉽게 말하면 기존 문서를 통째로 다시 쓰는 대신, 바뀐 부분만 적는 방식이다.

대표 섹션은 아래처럼 나뉜다.

- `ADDED`
- `MODIFIED`
- `REMOVED`
- `RENAMED`

이 구조 덕분에 "새 기능을 처음 만드는 경우"보다 "기존 기능을 보강하거나 규칙을 수정하는 경우"를 더 잘 다룬다.

## TCI에 왜 잘 맞는가

TCI가 원하는 것은 기능 아이디어를 설계 착수 가능한 수준까지 끌어올리고, 기존 워크플로우에 어떤 변화가 생기는지 분명하게 남기는 것이다. 이 기준에서 OpenSpec은 잘 맞는다.

- 변경 이유와 범위를 `proposal`로 먼저 고정함
- 요구사항 변화를 `delta spec`으로 남겨 기존 동작과 새 동작을 분리함
- `design`과 `tasks`까지 이어져 구현 착수 판단이 쉬움
- 기존 시스템을 계속 수정해 가는 `brownfield` 작업에 강함

즉, TCI처럼 기존 연동 규칙이나 기존 기능 흐름을 계속 손보는 프로젝트에 적합하다.

## 그대로 쓰면 안 되는 이유

OpenSpec이 유용하다고 해서 기본 형태를 그대로 도입하면 되는 것은 아니다.

- 기본 워크플로우는 가볍지만 설계 전 정제 강도는 Spec Kit보다 약함
- `clarify`, `research`, `data-model` 같은 보강 문서는 기본 제공이 아님
- 팀이 품질 게이트를 직접 정하지 않으면 문서 품질 편차가 커질 수 있음

즉, OpenSpec은 좋은 기본 뼈대이지만, TCI에 맞는 문서 규칙과 검토 기준을 덧붙여 써야 한다.

## TCI 기준으로는 이렇게 보는 편이 맞다

초기 도입에서는 OpenSpec을 "자동 구현 도구"보다 "변경 제안을 설계와 작업 계획으로 바꾸는 문서화 워크플로우"로 보는 편이 정확하다.

권장 방식은 아래와 같다.

- 초기에는 `proposal`, `spec`, `design`, `tasks` 품질 검토에 집중
- 구현 자동화는 작은 변경부터 제한적으로 사용
- 나중에는 TCI 전용 schema나 추가 문서 규칙을 붙여서 운영

## 발표 때 이렇게 설명하면 된다

OpenSpec은 기존 시스템에 들어갈 변경을 문서로 관리하는 SDD 도구다. 새 기능을 바로 개발하지 않고, 먼저 왜 바꾸는지, 무엇이 달라지는지, 어떻게 구현할지, 어떤 작업으로 나눌지를 한 change 단위로 정리한다. 우리에게 중요한 이유는, 기존 기능 수정이 많은 TCI에서 변경 영향과 요구사항 차이를 더 선명하게 남길 수 있기 때문이다.

## 최종 정리

OpenSpec의 핵심은 하나다. 아이디어를 바로 코드로 옮기지 않고, "현재 기준선"과 "제안 중인 변경"을 분리한 문서로 먼저 정리하게 만든다는 것이다.

TCI 기준으로는 도입 가치가 높다. 특히 기존 기능 변경을 관리하는 용도로 강하다. 다만 초기에는 구현 자동화보다 변경 문서화 워크플로우로 제한해서 쓰는 것이 가장 현실적이다.
