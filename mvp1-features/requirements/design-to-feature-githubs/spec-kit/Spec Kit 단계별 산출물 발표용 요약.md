# Spec Kit 단계별 산출물 발표용 요약

- 문서 목적: 팀 공유 자리에서 각 단계 산출물의 의미를 한눈에 설명하기 위한 발표 보조 문서
- 전제: 실제 산출물 예시는 라이브로 직접 보여주고, 이 문서는 "지금 보고 있는 파일이 왜 필요한가"를 빠르게 이해시키는 데 집중

## 먼저 핵심만

Spec Kit의 산출물은 전부 같은 목적을 가진다. 아이디어를 바로 구현하지 않고, `무엇을 만들지 → 뭐가 아직 모호한지 → 어떻게 만들지 → 무슨 작업을 할지`를 문서로 고정하는 것이다.

따라서 각 단계 산출물은 아래처럼 이해하면 된다.

- `specify`: 요구사항을 정리
- `clarify`: 빠진 질문을 메움
- `plan`: 기술 설계로 바꿈
- `tasks`: 실행 작업으로 쪼갬

TCI에서는 이 네 단계가 가장 중요하다.

## 전체 흐름 한눈에 보기

| 단계 | 대표 산출물 | 한 줄 의미 | 시연 때 팀원이 봐야 할 것 | 다음 단계로 넘기는 것 |
| --- | --- | --- | --- | --- |
| `specify init` | `.specify/`, `specs/` | Spec Kit 작업장 생성 | 문서를 쌓는 기본 구조가 생김 | 작업 환경 |
| `constitution` | `constitution.md` | 프로젝트 공통 원칙 정의 | 어떤 기준으로 문서를 쓸지 | 공통 규칙 |
| `specify` | `spec.md`, `requirements.md` | 기능 요구사항 초안 작성 | 사용자 가치, 요구사항, 성공 기준 | 명세 초안 |
| `clarify` | 갱신된 `spec.md` | 모호한 부분 정리 | 빠진 질문이 어떻게 반영됐는지 | 정리된 명세 |
| `checklist` | 관점별 체크리스트 | 특정 관점에서 보강 점검 | 보안, UX, API 관점 누락 여부 | 보완 포인트 |
| `plan` | `plan.md`, `research.md`, `data-model.md`, `contracts/`, `quickstart.md` | 기술 설계 입력 생성 | 구조, 데이터, 외부 연동, 검증 흐름 | 설계 세트 |
| `tasks` | `tasks.md` | 구현 작업 분해 | 작업이 어떻게 묶이고 순서화됐는지 | 실행 계획 |
| `analyze` | 분석 리포트 | 문서 간 충돌 점검 | spec, plan, tasks 간 누락과 불일치 | 수정 포인트 |
| `implement` | 코드, 테스트 | 실제 구현 | 문서가 코드로 바뀐 결과 | 구현 결과 |

## 발표에서는 이렇게 보면 된다

발표 중에는 모든 단계를 똑같이 길게 설명할 필요가 없다. 아래 우선순위로 보면 된다.

### 꼭 보여줄 단계

- `specify`
- `clarify`
- `plan`
- `tasks`

### 짧게만 언급할 단계

- `specify init`
- `constitution`
- `checklist`
- `analyze`
- `implement`

## 단계별 설명

## 1. `specify init`

### 산출물

- `.specify/`: Spec Kit 템플릿과 설정이 들어가는 시스템 작업 디렉터리
- `specs/`: 기능별 산출물이 실제로 쌓이는 작업 디렉터리

### 의미

문서를 만드는 단계가 아니라, Spec Kit이 앞으로 문서를 쌓을 작업장을 만드는 단계다.

### 시연 때 포인트

- `specs/` 아래에 기능별 문서가 쌓인다는 점
- `.specify/`는 사용자가 읽는 기능 문서보다 시스템 설정에 가깝다는 점

## 2. `constitution`

### 산출물

- `.specify/memory/constitution.md`: 프로젝트 공통 원칙을 적는 기준 문서

### 의미

이 프로젝트에서 어떤 기준으로 spec과 plan을 쓸지 정하는 상위 규칙 문서다.

### 시연 때 포인트

- 개별 기능 문서가 아니라 프로젝트 공통 원칙이라는 점
- 이후 plan 단계에서 이 규칙이 기준선이 된다는 점

## 3. `specify`

### 산출물

- `specs/<feature>/spec.md`: 기능 요구사항, 사용자 시나리오, 성공 기준을 담는 핵심 명세 문서
- `specs/<feature>/checklists/requirements.md`: 이 spec이 plan 단계로 넘어갈 만큼 잘 정의됐는지 보는 기본 검토표

### 의미

기능 설명을 읽고 요구사항 문서로 바꾸는 단계다. 구현 방법보다 "무엇을 왜 만드는가"를 고정한다.

### 시연 때 포인트

- 사용자 시나리오가 있는지
- 기능 요구사항이 있는지
- 성공 기준이 있는지
- 구현 상세보다 요구사항 중심으로 쓰였는지

### 한 줄로 정리

`spec.md`는 이 기능의 요구사항 계약서다.

## 4. `clarify`

### 산출물

- 갱신된 `spec.md`: clarify에서 나온 답변이 반영된 최신 요구사항 문서

### 의미

초안 spec의 빈칸을 메워, 설계 가능한 수준의 명세로 다듬는 단계다.

### 시연 때 포인트

- 어떤 질문이 추가로 나왔는지
- 그 질문의 답이 spec 본문에 실제로 반영됐는지

### 한 줄로 정리

`clarify`는 질문 목록을 만드는 단계가 아니라, spec을 더 명확하게 고치는 단계다.

## 5. `checklist`

### 산출물

- `checklists/security.md`: 보안과 권한 관점에서 spec 누락을 점검하는 체크리스트
- `checklists/ux.md`: 사용자 흐름과 사용성 관점에서 spec 누락을 점검하는 체크리스트
- `checklists/api.md`: 외부 연동과 인터페이스 관점에서 spec 누락을 점검하는 체크리스트

### 의미

기본 요구사항 검토 외에, 특정 관점에서 spec의 누락을 더 깊게 점검하는 보조 단계다.

### 시연 때 포인트

- 체크리스트가 구현 테스트가 아니라 명세 검토용이라는 점
- 보안, UX, API처럼 관점별로 누락을 찾는다는 점

## 6. `plan`

### 산출물

- `plan.md`: 전체 설계 방향, 제약, 검증 포인트를 정리한 기술 계획 문서
- `research.md`: 주요 기술 선택과 대안 검토 이유를 남기는 설계 근거 문서
- `data-model.md`: 핵심 데이터 개체와 관계를 정리한 문서
- `contracts/`: 외부 시스템과 주고받는 입력·출력 약속을 정리한 문서 묶음
- `quickstart.md`: 사람이 따라 해보며 검증할 기본 흐름을 적은 문서

### 의미

정리된 spec을 실제 기술 설계 입력으로 바꾸는 단계다.

### 시연 때 포인트

- `plan.md`: 전체 설계 방향
- `research.md`: 왜 이렇게 결정했는지
- `data-model.md`: 어떤 데이터가 필요한지
- `contracts/`: 외부 연동 약속이 무엇인지
- `quickstart.md`: 사람이 따라 해볼 검증 흐름이 무엇인지

### 한 줄로 정리

`spec`이 "무엇을 만들지"라면, `plan` 산출물 묶음은 "어떻게 만들지"다.

## 7. `tasks`

### 산출물

- `tasks.md`: 설계를 실제 구현 가능한 작업 단위로 분해한 실행 계획 문서

### 의미

설계를 실제 구현 가능한 작업 단위로 나누는 단계다.

### 시연 때 포인트

- 작업이 Phase나 사용자 스토리 기준으로 묶였는지
- 공통 기반 작업과 기능별 작업이 구분되는지
- 바로 backlog나 issue로 넘길 수 있는 수준인지

### 한 줄로 정리

`tasks.md`는 설계를 실행 계획으로 바꾼 문서다.

## 8. `analyze`

### 산출물

- 분석 리포트: spec, plan, tasks 사이의 충돌과 누락을 알려주는 점검 결과

### 의미

`spec.md`, `plan.md`, `tasks.md`가 서로 맞물리는지 점검하는 단계다.

### 시연 때 포인트

- spec에는 있는데 tasks에는 빠진 것이 있는지
- plan에는 있는데 spec과 연결되지 않는 내용이 있는지

## 9. `implement`

### 산출물

- 실제 코드: 앞 단계 문서를 바탕으로 구현된 기능 결과물
- 테스트 코드: 요구사항과 설계가 의도대로 동작하는지 검증하는 결과물
- 반영된 문서: 구현 결과에 맞춰 업데이트된 명세와 운영 문서

### 의미

앞 단계 문서를 바탕으로 실제 결과물을 만드는 단계다.

### 시연 때 포인트

- 구현의 출발점이 코드가 아니라 앞 단계 문서라는 점
- TCI에서는 이 단계보다 그 이전 문서 품질이 더 중요하다는 점

## 산출물끼리 어떻게 이어지나

한 기능은 보통 아래 순서로 이어진다.

1. `spec.md`에서 요구사항을 정한다
2. `clarify`로 빠진 질문을 메운다
3. `plan` 산출물로 설계 구조를 정한다
4. `tasks.md`로 구현 작업을 나눈다
5. 필요하면 `analyze`로 문서 간 충돌을 점검한다
6. 마지막에 `implement`로 실제 코드를 만든다

즉, 앞 단계 산출물은 다음 단계의 입력이다.

## 팀원이 꼭 기억하면 좋은 포인트

- `spec.md`는 요구사항 문서
- `plan` 산출물 묶음은 설계 문서
- `tasks.md`는 실행 계획
- `clarify`는 TCI에서 특히 중요한 관문
- Spec Kit의 핵심은 문서를 많이 만드는 것이 아니라, 구현 전에 필요한 생각을 문서로 고정하는 것

## TCI 발표용 권장 설명 순서

실제 발표에서는 아래 순서가 가장 이해하기 쉽다.

1. `spec.md`
2. `clarify`로 수정된 spec
3. `plan.md`와 `contracts/`
4. `tasks.md`
5. 필요하면 `analyze`와 `implement`는 짧게 언급

## 티켓 시스템 연동 시연용 복붙 프롬프트

아래 프롬프트는 [tci-final-feature-list.md](../tci-final-feature-list.md)의 `티켓 시스템 연동` 기능을 기준으로 잡았다.

기준 기능은 아래다.

- Jira OAuth/API 기반 연동
- 프로젝트/스페이스 선택
- 이슈 유형 및 상태 필터링
- 이슈 메타데이터 수집
  - 제목
  - 설명
  - 작성자
  - 상태
  - 링크

발표에서는 아래 블록을 그대로 복사해서 단계별로 붙여넣으면 된다.

### 1. `specify`

```text
$speckit-specify
TCI의 데이터 수집 영역에서 "티켓 시스템 연동" 기능을 spec으로 작성해줘.

기능 설명:
- Jira 등 외부 티켓 관리 시스템과 연동하여 코드 외 업무 맥락 정보를 수집하고 분석에 활용한다

상세 기능:
- Jira OAuth/API 기반 연동
- 프로젝트/스페이스 선택
- 이슈 유형 및 상태 필터링
- 이슈 메타데이터 수집
  - 제목
  - 설명
  - 작성자
  - 상태
  - 링크

요청사항:
- 구현 방법보다 사용자 가치와 요구사항 중심으로 작성해줘
- 사용자 시나리오, 기능 요구사항, 성공 기준, 제외 범위를 포함해줘
- 지금 단계에서는 구체적인 DB 스키마나 API 엔드포인트는 넣지 말아줘
- TCI 맥락상 이 기능은 이후 문서-코드 추적과 갭 분석 기능의 입력 데이터가 된다는 점을 반영해줘
```

### 2. `clarify`

```text
$speckit-clarify
지금 작성된 티켓 시스템 연동 spec을 기준으로, 설계 착수 전에 반드시 닫아야 할 모호한 항목을 질문으로 드러내고 spec 본문에 반영해줘.

특히 아래 항목을 우선적으로 확인해줘.
- Jira만 1차 지원인지, 다른 티켓 시스템까지 포함하는지
- OAuth와 API 토큰 방식을 모두 지원하는지, 1차 범위가 무엇인지
- 프로젝트/스페이스 선택 단위가 사용자별인지 워크스페이스별인지
- 어떤 이슈 유형과 상태를 필터링 대상으로 볼지
- 수집 주기가 실시간인지 배치인지
- 수집 대상 메타데이터에 댓글, 첨부파일, 담당자, 라벨이 포함되는지
- 권한 부족, 인증 만료, 프로젝트 접근 불가 시 처리 원칙이 무엇인지
- 이번 범위가 "수집"까지인지, "분석 연결"까지인지

질문만 나열하지 말고, 답이 정해지면 spec.md의 요구사항과 비범위 항목에 반영해줘.
```

### 3. `checklist`

```text
$speckit-checklist
티켓 시스템 연동 spec에 대해 체크리스트를 만들어줘.

다음 두 관점을 우선 다뤄줘.
- API/연동 관점
- 보안/권한 관점

체크리스트가 확인해야 할 내용:
- 인증 방식과 권한 범위가 spec에 명시되어 있는가
- 프로젝트/스페이스 선택 범위가 명확한가
- 필터 조건이 테스트 가능하게 정의되어 있는가
- 수집 대상 메타데이터 범위가 명확한가
- 인증 만료, 권한 부족, 연결 실패 시 동작이 명시되어 있는가
- 민감 정보 저장과 마스킹 원칙이 빠지지 않았는가

구현 테스트가 아니라 명세 검토용 체크리스트로 작성해줘.
```

### 4. `plan`

```text
$speckit-plan
티켓 시스템 연동 기능의 기술 계획을 작성해줘.

기능 범위:
- Jira OAuth/API 기반 연동
- 프로젝트/스페이스 선택
- 이슈 유형 및 상태 필터링
- 이슈 메타데이터 수집
  - 제목
  - 설명
  - 작성자
  - 상태
  - 링크

계획 작성 시 반영할 제약:
- TCI에서는 구현보다 설계 입력 문서 품질이 더 중요하다
- 외부 연동이므로 contracts와 data-model을 반드시 분리해 남겨야 한다
- 이후 문서-코드 추적 기능에서 티켓 데이터를 참조할 수 있어야 한다
- 초기 파일럿에서는 Jira를 우선 대상으로 가정해도 된다
- 검증 포인트, 비범위 항목, 운영상 위험을 명확히 적어줘

산출물로는 plan.md, research.md, data-model.md, contracts/, quickstart.md에 들어갈 내용을 정리해줘.
```

### 5. `tasks`

```text
$speckit-tasks
티켓 시스템 연동 기능의 tasks를 작성해줘.

요구사항:
- 구현 가능한 작업 단위로 나눠줘
- 공통 기반 작업과 사용자 가치 기준 작업을 구분해줘
- 인증 연동, 프로젝트 선택, 필터링, 메타데이터 수집, 오류 처리, 검증 작업을 빠뜨리지 말아줘
- 이후 backlog나 issue로 바로 옮길 수 있는 수준으로 작성해줘
- 초기 파일럿이므로 과도하게 세분화된 엔지니어링 하위 작업보다 설계 검토와 구현 경계가 드러나는 수준으로 작성해줘
```

### 6. `analyze`

```text
$speckit-analyze
티켓 시스템 연동 기능의 spec.md, plan.md, tasks.md를 교차 검토해줘.

특히 아래를 중점적으로 봐줘.
- spec에 있는 요구사항이 tasks에 빠지지 않았는가
- plan에 정의한 외부 연동 계약과 데이터 모델이 spec 요구사항과 맞물리는가
- 인증, 권한, 필터링, 메타데이터 수집, 오류 처리 관련 누락이 없는가
- 비범위 항목이 tasks에 잘못 들어간 부분은 없는가
- 추후 문서-코드 추적 기능과 연결될 수 있도록 식별자나 참조 구조가 고려되었는가

치명적, 중요, 보통 수준으로 나눠서 알려줘.
```

### 7. `implement`

```text
$speckit-implement
티켓 시스템 연동 기능은 지금 바로 구현 자동화하지 말고, spec.md, plan.md, tasks.md를 기준으로 실제 구현에 들어갈 준비가 되었는지만 판단해줘.

아래 기준으로 답해줘.
- 구현 시작 가능
- clarify가 더 필요함
- plan 보강 필요
- tasks 보강 필요

그리고 바로 구현하면 위험한 항목이 있으면 함께 적어줘.
```
