# SDD를 모르는 팀원을 위한 Spec Kit 소개

- 문서 목적: SDD와 `github/spec-kit`을 처음 접하는 팀원이 핵심 개념만 빠르게 이해하도록 설명
- 대상 독자: 내일 공유 세션에 참석하는 TCI 팀원

## 한 줄 정의

Spec Kit은 GitHub가 공개한 오픈소스 SDD 워크플로우다. CLI와 템플릿을 이용해 기능 아이디어를 `spec`, `plan`, `tasks` 같은 문서로 정리하게 만든다.

이 문서를 끝까지 읽고 기억해야 할 핵심은 하나다. Spec Kit은 코드를 대신 써주는 도구라기보다, 아이디어를 설계 가능한 문서로 바꾸는 틀이다.

## 먼저 SDD가 무엇인가

SDD는 `Spec-Driven Development`의 약자다. 쉽게 말하면 코드를 먼저 쓰는 대신, 명세를 먼저 만들고 그 명세를 기준으로 설계와 구현을 진행하는 방식이다.

핵심 흐름은 단순하다.

- `spec`: 무엇을 왜 만드는가
- `plan`: 그것을 어떻게 만들 것인가
- `tasks`: 실제 구현을 어떤 작업으로 나눌 것인가

즉, `spec`은 요구사항 문서이고 `plan`은 설계 문서다. 이 둘을 먼저 맞춘 뒤 구현으로 들어가는 것이 SDD의 기본 생각이다.

## 왜 이런 방식이 필요한가

기능 논의가 바로 구현으로 넘어가면 보통 아래 문제가 생긴다.

- 같은 기능을 사람마다 다르게 이해함
- 범위가 덜 닫힌 상태에서 설계가 시작됨
- 구현하다가 빠진 요구사항이 뒤늦게 드러남
- 작업은 시작됐는데 성공 기준이 모호함

Spec Kit은 이 문제를 줄이기 위해, 기획과 구현 사이에 문서 단계를 강제로 넣는다.

## Spec Kit의 핵심 흐름

Spec Kit의 기본 흐름은 아래와 같다.

`constitution → specify → clarify → plan → tasks → implement`

TCI에서 당장 중요하게 볼 단계는 `specify → clarify → plan → tasks`다.

| 단계 | 이 단계에서 답하는 질문 |
| --- | --- |
| `specify` | 이번에 무엇을 왜 만드는가 |
| `clarify` | 아직 모호하거나 빠진 것은 무엇인가 |
| `plan` | 이 기능을 어떤 구조로 설계할 것인가 |
| `tasks` | 실제 구현은 어떤 작업 단위로 나뉘는가 |

여기서 가장 중요한 단계는 `clarify`다. 요구사항이 애매한 상태로 바로 설계에 들어가지 못하게 막아주기 때문이다.

## 산출물은 이것만 기억하면 된다

발표에서 세부 산출물은 직접 시연할 예정이므로, 여기서는 세 가지만 기억하면 충분하다.

| 문서 | 역할 |
| --- | --- |
| `spec.md` | 요구사항 명세 |
| `plan.md` | 기술 설계 |
| `tasks.md` | 구현 작업 목록 |

그 외 `research.md`, `data-model.md`, `contracts/` 같은 문서는 이 세 문서를 보강하는 보조 문서라고 보면 된다.

## TCI에 왜 잘 맞는가

TCI가 원하는 것은 기능 아이디어를 설계 착수 가능한 수준까지 끌어올리는 것이다. 이 기준에서 Spec Kit은 잘 맞는다.

- 기획과 구현 사이에 명세와 설계 단계를 강제로 둠
- 누락 질문을 `clarify` 단계에서 구조적으로 드러냄
- 설계 결과를 `tasks`까지 연결해 실행 계획으로 이어줌

특히 TCI에서는 "설계 착수 가능 여부"를 판단하는 과정이 중요하다. 이 점에서 `clarify` 단계의 가치가 크다.

## 그대로 쓰면 안 되는 이유

Spec Kit이 유용하다고 해서 기본 형태를 그대로 도입하면 되는 것은 아니다.

- 기본 문서 구조는 개발 실행 관점에 더 가깝다
- 현재 TCI 문서 체계와 그대로 맞지는 않는다
- 팀이 이해하기 쉬운 용어와 문서 톤으로 바꿔 써야 한다

즉, Spec Kit은 좋은 기본 뼈대이지만, TCI에 맞게 다듬어서 써야 한다.

## TCI 기준으로는 이렇게 보는 편이 맞다

초기 도입에서는 Spec Kit을 "자동 구현 도구"로 보기보다 "기획을 설계와 작업 계획으로 바꾸는 문서화 워크플로우"로 보는 편이 정확하다.

권장 방식은 아래와 같다.

- `implement`는 당분간 쓰지 않음
- `specify → clarify → plan → tasks`까지만 사용
- 산출물은 TCI 문서 체계에 맞게 옮겨서 관리

## 발표 때 이렇게 설명하면 된다

Spec Kit은 GitHub가 공개한 오픈소스 SDD 워크플로우다. 기능 아이디어를 바로 개발하지 않고, 먼저 요구사항 문서와 설계 문서, 작업 목록으로 정리하게 만든다. 우리에게 중요한 이유는, 기획을 설계 입력 수준까지 끌어올리고, 빠진 질문을 구현 전에 드러낼 수 있기 때문이다.

## 최종 정리

Spec Kit의 핵심은 하나다. 아이디어를 바로 코드로 옮기지 않고, `spec`, `plan`, `tasks`를 거쳐 팀이 같은 기준으로 움직이게 만드는 것이다.

TCI 기준으로는 도입 가치가 높다. 다만 초기에는 구현 자동화보다 문서화 워크플로우로 제한해서 쓰는 것이 가장 현실적이다.
