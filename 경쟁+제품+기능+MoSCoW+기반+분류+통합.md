# 분류 기준 또는 생각하는 제품 방향

### 개인 의견

- @이응규 AI로 개발을 진행함에 따라 코드의 작성 비용이 낮아지고, 직접적으로 코드를 리뷰하는 것이 아니라 개발된 기능을 검증하는 단계로 넘어갈 것이다.
    - 따라서 소스코드의 구조의 시각화 보다는 어떻게 동작하는 가를 보여주는 **비지니스 로직 흐름의 시각화**가 중요해질 것이다.
    - AI가 작성한 코드가 **안전한 코드인가에 대한 보안 검토**가 중요해질 것이다.
    - 작성한 코드가 안정적으로 **의도한 대로 동작하는 가를 검증하기 위한 테스트 코드**가 중요해질 것이다.

### 논의 내용

- 시각화
    - 전체 시스템을 대상으로 하는 시각화는 정보 전달, 인사이트 제공에서 수동적이고 유용하지 않다고 판단 스펙아웃
    - 사용자의 입장에서는 분석 데이터의 시각화가 가장 중요할 것으로 보이고, 이를 타사 제품 대비 어떻게 경쟁력 있고 효율적으로 제공할 지가 관건 논의필요
        - 정적분석의 결과물이 기존의 정적분석 제품들에서 보여주는 뻔한 형태의 다이어그램 나열이라면 시각화가 무의미하다.
        - ‘Code Diff를 대상으로 하는 시각화’와 같은 범위를 갖는 시각화는 유용할 수 있다고 생각하여 Should have로 판단
- 품질 관련 메트릭
    - 품질 메트릭 관련된 정보들은 해당 기능들을 전문적으로 제공하는 제품들 대비 경쟁력을 가져가는 것이 어려울 것 같으니 제외하자. 스펙아웃
    - 코드 품질, Code Health, Hot Spot 같은 기능은 거버넌스와도 연관될 수 있다고 판단하여 스펙 아웃 대신 Could have로 판단 논의필요
- 보안
    - 보안 정적 분석의 필요성에 대해서는 공감하나 제품의 방향성과 일치하는지에 대해 고민, 제품 컨셉 정렬이 필요하다. 논의필요
    - 보안 정적 분석은 제품의 방향성을 고려했을 때 스펙 아웃 대상이라고 판단 스펙아웃
        - 보안 관련 기능을 제공하더라도 SBOM 정도만 제공해주는 레벨로.
- 편의
    - 편의성 기능들은 우선순위 높은 항들 기준으로 내부 협의 후 최종 확정 필요 논의필요
        - ex) 검색 / 탐색 기능?
- 외부 통합
    - IDE 통합은 하면 좋지만 아니어도 제품 사용에 문제는 없기 때문에 Could have로 판단 논의필요
- 조직 생산성 스펙아웃
    - 팀 인력, 조직 생산성과 관련된 기능들도 제품의 방향성과 맞지 않다고 생각하여 스펙 아웃이라 판단 스펙아웃
- 검색/편집 스펙아웃
    - 시스템 전체를 이해하는 데 제품 초점을 맞춘다면 코드 작성 지원, 검색 등의 기능은 제품 스코프에 맞지 않다. 스펙아웃
    - 코드 변경 관련 기능들은 제품의 방향성과 맞지 않다고 생각하여 모두 스펙 아웃 스펙아웃
    - 코드 검색/탐색은 IDE 급의 성능이 나오지 않는 이상 결국엔 IDE를 사용할 것이라 판단하여 스펙 아웃이라 판단 스펙아웃
- 기타
    - ‘지식 자산화’ 카테고리의 설명을 질의/추론 가능한 지식 베이스 구축보다 더 넓은 범위로 볼 필요성이 있어보인다. 논의필요
- 분류 기준 및 제품 방향을 반영한 "제품 컨셉 업데이트" - 전부
- 정리된 기능에 대한 기능 상세화 및 공통 기능 도출 - 전부
