# 아키텍처 설명용 Deployment Diagram 작성 재료

## 목적

이 문서는 아키텍처 설명용 Deployment Diagram을 그리기 전에 무엇이 확정되어 있어야 하는지 정리한 체크리스트다.

여기서 말하는 "아키텍처 설명용"은 운영 인스턴스, 세부 포트, 복제 수까지 모두 담는 운영용 다이어그램이 아니라, 시스템이 어디에서 실행되고 어떻게 연결되는지 설명하는 고수준 다이어그램을 뜻한다.

## 먼저 결론

아키텍처 설명용 Deployment Diagram을 그리려면 최소한 아래 네 가지는 있어야 한다.

- 어떤 소프트웨어가 있는가
- 그 소프트웨어가 어디에 배포되는가
- 어떤 실행 환경에서 동작하는가
- 무엇과 무엇이 연결되는가

이 네 가지 중 두 개 이상이 비어 있으면 Deployment Diagram을 그릴 수는 있어도 설명력이 약하거나 실제와 어긋날 가능성이 크다.

## 필수 재료

### 1. 소프트웨어 목록

먼저 시스템 안에 어떤 소프트웨어 단위가 있는지 알아야 한다.

예시

- 웹 프론트엔드
- 백엔드 API
- 배치 작업
- 관리자 페이지
- 인증 서비스

이 정보는 보통 C4 System Context Diagram이나 Container Diagram에서 가져올 수 있다.

이 단계에서 확인할 질문

- 사용자에게 보이는 주요 애플리케이션은 무엇인가
- 내부적으로 중요한 서비스는 무엇인가
- 외부 시스템과 구분되는 우리 시스템 범위는 어디까지인가

### 2. 배포 대상

각 소프트웨어가 어디에 올라가는지 알아야 한다.

대표 예시

- 브라우저
- CDN
- 웹 서버
- VM
- Kubernetes cluster
- 서버리스 런타임
- 데이터베이스 서버
- 캐시 서버

이 정보가 없으면 Deployment Diagram은 단순한 소프트웨어 관계도에 머무른다.

이 단계에서 확인할 질문

- 프론트엔드는 어디서 서빙되는가
- 백엔드는 어디서 실행되는가
- DB와 캐시는 어디에 위치하는가
- 외부 시스템은 우리 인프라 밖에 있는가

### 3. 실행 환경

배포 대상만으로는 부족하고, 그 위에서 무엇이 실행되는지도 알아야 한다.

대표 예시

- Linux
- Docker
- JVM
- Node.js runtime
- Nginx
- PostgreSQL
- Redis

실무에서는 이 정보가 빠지면 "API 서버"라는 말 하나로 물리 서버, 운영체제, 컨테이너, 런타임이 모두 섞여버린다.

이 단계에서 확인할 질문

- 애플리케이션은 VM에서 직접 도는가
- 컨테이너에서 도는가
- 쿠버네티스에서 도는가
- 서버리스 플랫폼에서 도는가
- DB는 어떤 엔진 위에서 동작하는가

### 4. 배포 단위

무엇이 실제로 배포되는지 알아야 한다.

대표 예시

- `frontend build`
- `api container image`
- `batch.jar`
- `nginx.conf`
- `application-prod.yml`

Deployment Diagram에서는 보통 Component 자체보다 Artifact를 Node에 배포하는 방식으로 이해하는 것이 더 정확하다.

이 단계에서 확인할 질문

- 실제 배포 단위는 바이너리인가
- 컨테이너 이미지인가
- 정적 빌드 결과물인가
- 설정 파일을 분리해서 관리하는가

### 5. 주요 연결 경로

무엇과 무엇이 통신하는지 알아야 한다.

대표 예시

- 브라우저 -> CDN
- CDN -> 웹 앱
- 웹 앱 -> API
- API -> PostgreSQL
- API -> Redis
- API -> 외부 결제사

이 단계에서 확인할 질문

- 누가 누구를 호출하는가
- 동기 호출인가 비동기 호출인가
- 인터넷 구간인가 내부망 구간인가
- 외부 SaaS 연동이 있는가

### 6. 시스템 경계

Deployment Diagram은 단순 연결도보다 경계 표현이 중요하다.

대표 예시

- 외부 사용자 영역
- 인터넷 공개 구간
- 사설 네트워크 구간
- 우리 시스템
- 외부 SaaS
- 고객사 온프레미스

이 정보가 있어야 그림이 단순 연결 관계를 넘어 아키텍처 설명 문서로 기능한다.

이 단계에서 확인할 질문

- 어디까지가 우리 통제 범위인가
- 어떤 노드가 public 영역에 노출되는가
- 어떤 시스템이 외부 시스템인가

## 있으면 좋은 재료

아래 정보는 없어도 아키텍처 설명용 Deployment Diagram 초안은 그릴 수 있다. 하지만 있으면 그림의 정확도와 설명력이 좋아진다.

### 1. 비동기 처리 경로

- 메시지 브로커
- 이벤트 버스
- 큐 기반 배치
- 워커 프로세스

동기 요청 흐름만 보이는 Deployment Diagram은 실제 구조를 지나치게 단순화할 수 있다.

### 2. 보조 인프라

- object storage
- search engine

모든 보조 인프라를 그릴 필요는 없지만, 아키텍처 이해에 중요한 것은 포함하는 편이 좋다.

### 3. 보안 경계

- public subnet
- private subnet
- 내부망

보안 경계가 중요한 시스템이라면 이 정보가 빠진 Deployment Diagram은 설명력이 떨어질 수 있다. 다만 이번 단계에서는 coarse-grained 경계만 있으면 충분하다.

## 아직 그리면 안 되는 상태

아래 상황이면 Deployment Diagram 작성보다 정보 확정이 먼저다.

- 서비스 목록은 있는데 각 서비스의 실행 위치를 모름
- 실행 위치는 아는데 배포 단위를 모름
- 외부 시스템은 아는데 경계와 통신 경로를 모름
- "API 서버" 같은 이름만 있고 VM, 컨테이너, 서버리스 중 무엇인지 모름
- DB와 캐시가 있는 것은 아는데 어디에 붙는지 모름

이 상태에서 그린 그림은 예쁘게 보일 수는 있어도, 실제 아키텍처 설명 자료로는 위험하다.

## 가장 쉬운 판단 기준

아래 질문에 모두 답할 수 있으면 아키텍처 설명용 Deployment Diagram을 그려도 된다.

- 각 주요 소프트웨어는 무엇인가
- 각 소프트웨어는 어디에 배포되는가
- 어떤 실행 환경에서 동작하는가
- 어떤 핵심 인프라와 연결되는가
- 외부 시스템과 내부 시스템의 경계는 어디인가

하나라도 애매하면 먼저 그 부분부터 확정하는 편이 낫다.

## C4 Diagram에서 가져올 수 있는 것

C4 Diagram이 있다면 아래 재료는 이미 확보된 경우가 많다.

- 사용자와 시스템의 관계
- 주요 소프트웨어 시스템
- 주요 컨테이너 또는 서비스
- 서비스 간 상호작용
- 외부 시스템 연동

즉, C4는 Deployment Diagram의 좋은 출발점이다.

## C4만으로는 부족한 것

대부분 아래 정보는 C4만으로는 부족하거나 빠져 있다.

- 실제 배포 대상
- 실제 실행 환경
- 실제 배포 단위
- 네트워크 경계
- 인프라 노드의 역할

그래서 C4 Diagram이 있어도 Deployment Diagram을 바로 그리기 전에 보강 확인이 필요하다.

## 추천 수집 순서

실무에서는 아래 순서로 정보를 모으면 가장 빠르다.

1. C4 Diagram에서 소프트웨어 목록 추출
2. 각 소프트웨어의 배포 위치 확인
3. 각 배포 위치의 실행 환경 확인
4. 주요 데이터 저장소와 보조 인프라 확인
5. 외부 시스템과 네트워크 경계 확인
6. 마지막으로 다이어그램 목적에 맞게 추상화 수준 조정

## 빠른 체크리스트

Deployment Diagram을 그리기 전에 아래 항목을 체크하면 된다.

- [ ] 주요 소프트웨어 목록이 정리되어 있다
- [ ] 각 소프트웨어의 배포 대상이 정리되어 있다
- [ ] 각 배포 대상의 실행 환경이 정리되어 있다
- [ ] 주요 Artifact 또는 배포 단위가 정리되어 있다
- [ ] 주요 통신 경로가 정리되어 있다
- [ ] 외부 시스템과 내부 시스템의 경계가 정리되어 있다
- [ ] public/private 같은 주요 네트워크 구간이 정리되어 있다
- [ ] 다이어그램의 목적이 정해져 있다
- [ ] 운영용 상세가 아니라 아키텍처 설명용 수준으로 추상화 범위를 정했다

## 결론

아키텍처 설명용 Deployment Diagram을 그리기 위한 핵심 재료는 많지 않다.

정리하면 아래 여섯 가지면 충분하다.

- 소프트웨어 목록
- 배포 대상
- 실행 환경
- 배포 단위
- 주요 연결 경로
- 시스템 경계

이 여섯 가지가 정리되면 Deployment Diagram은 자연스럽게 그릴 수 있다. 반대로 이 중 핵심이 비어 있으면, 그림을 그리는 것보다 먼저 아키텍처 사실을 확정하는 편이 낫다.
