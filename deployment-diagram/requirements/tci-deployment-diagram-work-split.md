# TCI Deployment Diagram 작업 분담안

## 목적

이 문서는 [tci-deployment-diagram-readiness-assessment.md](/Users/seokhyunbae_1/Desktop/기획_스프린트/TCI/deployment-diagram/tci-deployment-diagram-readiness-assessment.md)의 최신 기준을 바탕으로, 아키텍처 설명용 Deployment Diagram 작성을 위해 프론트 개발자 1명, 백엔드 개발자 2명, 그리고 내가 각각 무엇을 조사하고 무엇을 확정해야 하는지 정리한 문서다.

이번 문서는 아래 전제를 따른다.

- 초점은 `우리가 정의한 서비스가 어디서 제공되는가`
- 표현 수준은 `Node`와 `Artifact`
- `Execution Environment`는 이번 범위에서 제외
- 통신 경로는 프로토콜 형태로 작성
- 외부 시스템은 일단 개별 서비스로 그리고, 필요하면 역할군으로 묶는다
- `Knowledge Base`는 메인 그림에서 단일 Node로 두되, 내부 Artifact를 함께 구분한다
- 상위 경계는 `public / private / external / data` 4영역을 사용한다
- 운영 배치도보다 아키텍처 설명용 그림을 우선

팀 구성 전제는 다음과 같다.

- 프론트 개발자 1명
- 백엔드 개발자 2명
- 그중 한 명은 C4 Diagram 작성자
- 나도 백엔드 개발자로 함께 작업

## 먼저 결론

이 작업은 기술 스택 기준보다 `표현 기준`으로 나누는 것이 좋다.

- 프론트 개발자
  - 사용자 진입 Node
  - Web Application 제공 위치
  - 업로드와 사용자 접점
  - public 경계 관점의 Node 정리
- C4 Diagram 작성 백엔드 개발자
  - C2 컨테이너를 Deployment Diagram용 Node로 매핑
  - Node 분해 수준 결정
  - 외부 시스템 역할군 기준 정리
  - 상위 경계와 포함 범위 결정
  - 기본 통신 표기 기준 정리
- 나
  - 역할 중심 Artifact 정의
  - Knowledge Base 단일 Node + 내부 Artifact 구분안 정리
  - 외부 시스템 역할군과 실제 접점 정리
  - 내부 연결 정리
  - 최종 통합

즉, 이번 작업은 `실행 환경 조사`보다 `무엇을 어떤 Node와 Artifact로 표현할지`를 닫는 쪽이 핵심이다.

## 역할 분담 원칙

### 1. 프론트 개발자에게 맞는 일

프론트 개발자는 아래를 가장 잘 안다.

- 사용자가 어디로 들어오는가
- Web Application이 어떤 경로에서 제공되는가
- 업로드와 사용자 상호작용이 어느 접점에서 일어나는가
- 사용자에게 직접 보이는 Node 경계가 어디까지인가

즉, `Entry Surface`와 `User-facing Node`를 프론트가 맡는 것이 가장 자연스럽다.

### 2. C4 Diagram 작성자에게 맞는 일

C4 Diagram을 그린 사람은 아래를 가장 잘 안다.

- 현재 논리 구조의 의도
- 컨테이너 경계
- 어떤 외부 시스템을 포함해야 하는지
- 어떤 수준까지 Node를 쪼개거나 묶어야 하는지

즉, C4를 Deployment Diagram의 `서비스 제공 Node 구조`로 번역하는 상위 기준은 이 사람이 맡는 것이 가장 효율적이다.

### 3. 내가 맡아야 하는 일

이번 범위에서 가장 까다로운 부분은 아래다.

- 역할 중심 Artifact 경계
- Knowledge Base 표현 수준
- 외부 시스템 역할군과 실제 접점
- 내부 연결의 통신 표기 정리
- 최종 표현 수준 통일

이건 C4 의도와 백엔드 현실을 같이 봐야 하므로 내가 통합 책임을 갖는 것이 맞다.

## 추천 진행 순서

### 1. 셋이 먼저 짧게 맞출 것

처음부터 각자 자료를 만들면 Node 수준이 어긋날 수 있다.

먼저 아래 결정 사항을 같이 정하는 것이 좋다.

| 결정 사항 | 결정할 질문 | 선택지 예시 | 결정 결과 |
|---|---|---|---|
| 대표 범위 | 대표 환경 하나만 그릴지 | 하나의 대표 환경 / 환경 구분 없이 공통 구조만 표현 | 대표 환경 하나만 그림 |
| 외부 시스템 포함 범위 | 외부 시스템을 어디까지 포함할지 | 핵심 외부 시스템만 / C1 외부 시스템 대부분 포함 | C1 외부 시스템 모두 포함 |
| 외부 시스템 표현 방식 | 외부 시스템을 개별 서비스로 그릴지, 역할군으로 묶을지 | 개별 시스템 유지 / 역할군으로 묶음 | 일단 개별로 그려놓고, 역할군으로 묶을 수 있으면 묶자 |
| Node 분해 수준 | 각 C2 컨테이너를 모두 Node로 드러낼지, 일부를 묶을지 | 전부 분리 / 일부를 묶은 서비스 제공 Node | C2 컨테이너를 모두 노드로 매핑 |
| 서비스 묶음 기준 | `Node 분해 수준`이 `일부를 묶은 서비스 제공 Node`일 때, 어떤 컨테이너를 하나의 Node로 묶을지 | 사용자 접점만 분리 / 내부 서비스 일부 통합 / KB만 별도 유지 / Operations Manager만 admin으로 분리 | C2 컨테이너를 모두 노드로 매핑하기 때문에 결정사항 없음 |
| Knowledge Base 표현 방식 | `Knowledge Base`를 하나의 Node로 둘지 분해할지 | KB 단일 Node / 단일 Node + 내부 Artifact 구분 / 상세 그림에서만 분해 | 단일 Node + 내부 Artifact 구분 |
| 상위 경계 수준 | `public / private / external`만 쓸지, `data / admin`도 추가할지 | 3영역 기본 / 필요 시 4영역 / 필요 시 5영역 | 4영역(public, private, external, data)으로만 결정 |
| 통신 경로 표기 방식 | 통신 경로를 어떤 방식으로 적을지 | 사용자 접근 / 내부 처리 / 지식 접근 / 외부 연동 / 운영 제어 / 프로토콜 표기 | 프로토콜 형태로 작성 |

- 상위 경계 수준은 Deployment Diagram에서 시스템을 큰 구역으로 어떻게 나눠 그릴지를 뜻한다.
- 즉, 개별 Node를 그리기 전에 먼저 "이 Node들이 어느 영역에 속하나"를 정하는 단계이다.
  - public: 사용자나 외부 클라이언트가 직접 접근할 수 있는 영역
  - private: 우리 시스템 내부에서만 쓰이는 서비스 영역
  - external: 우리 시스템 바깥에 있는 외부 서비스 영역
  - data: 저장소나 지식 허브처럼 데이터 중심 역할을 하는 영역
  - admin: 운영자만 쓰는 관리 기능 영역

## 산출물 선행관계

모든 산출물이 완전히 직렬로 이어지는 것은 아니다. 다만 아래 산출물들은 선행관계가 분명하다.

### 1. 가장 먼저 나와야 하는 산출물 묶음

이 산출물 묶음들은 다른 표와 메모의 기준이 된다.

- `Node / 경계 / 서비스 표 묶음` 중 기준 성격 항목
  - `포함할 Node / 제외할 Node 목록`
  - `상위 경계 초안`
- `사용자 진입 / 외부 시스템 표 묶음`
  - `외부 시스템 역할군 표` 상단 기준 섹션
- `통신 표 묶음`
  - `Node 간 연결 표` 상단 기준 섹션

이 단계가 먼저 필요한 이유는 아래와 같다.

- 어떤 Node를 그릴지 정하지 않으면 서비스 제공 위치가 흔들린다
- 외부 시스템을 어떻게 묶을지 정하지 않으면 메인 그림 복잡도가 흔들린다
- 경계가 정해지지 않으면 public / private / external / data 구분이 흔들린다
- 통신 표기 기준이 없으면 연결 표가 사람마다 다른 표현으로 작성된다

### 2. 그다음 나와야 하는 산출물 묶음

이 산출물 묶음들은 1단계 결과를 바탕으로 정리할 수 있다.

- `Node / 경계 / 서비스 표 묶음` 중 배치 항목
  - `서비스 Node 매핑 초안`
  - `서비스별 제공 위치 표`
  - `우리 시스템 / 외부 서비스 구분표`
- `사용자 진입 / 외부 시스템 표 묶음`
  - `사용자 진입 경로 표`
  - `외부 시스템 역할군 표`
  - `외부 시스템별 접속 지점 표`
  - `외부 채널 접점 표`
- `Artifact / Knowledge Base 표 묶음`
  - `Artifact 이름 및 Node 매핑표`
  - `Knowledge Base 표현 결정표`

이 단계가 1단계 뒤에 와야 하는 이유는 아래와 같다.

- 서비스 제공 위치는 Node 구조가 먼저 정해져야 적을 수 있다
- 사용자 진입 경로는 상위 경계와 진입 Node 정의가 먼저 있어야 정리된다
- Artifact는 어떤 Node가 존재하는지 먼저 정해져야 붙일 수 있다
- 외부 시스템 역할군 표는 메인 그림 복잡도를 정하는 기준이라 초반에 정리하는 편이 안전하다
- Knowledge Base 표현 수준은 전체 Node 분해 수준과 맞물린다
- 외부 시스템 접속 지점은 어떤 서비스 Node가 외부와 맞닿는지 먼저 알아야 쓸 수 있다

### 3. 그다음 나와야 하는 산출물 묶음

이 산출물 묶음들은 앞선 표와 메모가 정리된 뒤에 안정적으로 작성할 수 있다.

- `통신 표 묶음`
  - `Node 간 연결 표`

이 단계가 뒤로 가야 하는 이유는 아래와 같다.

- Node 간 연결은 통신 경로 표기 기준과 Node 구조가 먼저 정리돼야 일관되게 작성할 수 있다
- Knowledge Base를 한 Node로 볼지 나눌지에 따라 연결 수와 Artifact 경계가 함께 바뀐다

### 4. 마지막 산출물

아래 산출물은 앞선 산출물들이 어느 정도 정리된 뒤에 그릴 수 있다.

- `최종 Deployment Diagram 초안`

이 산출물은 아래 내용을 모두 흡수해야 한다.

- 어떤 Node를 그릴지
- 외부 시스템을 어떤 역할군으로 묶을지
- 각 Node가 어떤 경계에 속하는지
- 각 Node에 어떤 Artifact를 둘지
- Node 사이를 어떤 통신 표기 기준으로 연결할지

### 병렬로 가능한 산출물 묶음

완전한 직렬만 있는 것은 아니다. 아래는 병렬로 진행해도 된다.

- `사용자 진입 / 외부 시스템 표 묶음`과 `Node / 경계 / 서비스 표 묶음`의 일부 항목
  - `사용자 진입 경로 표`
  - `서비스별 제공 위치 표`
- `Node / 경계 / 서비스 표 묶음`과 `통신 표 묶음`의 기준 항목
  - `상위 경계 초안`
  - `Node 간 연결 표` 상단 기준 섹션
- `사용자 진입 / 외부 시스템 표 묶음`과 `Artifact / Knowledge Base 표 묶음`의 일부 항목
  - `외부 시스템 역할군 표`
  - `Knowledge Base 표현 결정표`

다만 아래 산출물들은 병렬로 시작할 수는 있어도, 확정은 선행 산출물 이후가 안전하다.

- `Artifact / Knowledge Base 표 묶음`
  - `Artifact 이름 및 Node 매핑표`
- `통신 표 묶음`
  - `Node 간 연결 표`
- `최종 Deployment Diagram 초안`

### 한 줄 흐름

산출물 선행관계는 아래 순서로 이해하면 된다.

1. `Node 구조와 경계 기준 정리`
2. `서비스 제공 위치 / 사용자 진입 / Artifact / 외부 시스템 역할군 / Knowledge Base 표현 / 외부 접점 정리`
3. `연결 정리`
4. `최종 Deployment Diagram 초안 작성`

## 주의할 점

### 프론트 개발자에게 과도하게 넘기면 안 되는 것

- Knowledge Base 표현 수준
- Artifact 전체 경계
- 내부 서비스 연결 체계
- 외부 채널 중 AI Agent / IDE Plugin / MCP 접점 정리

### C4 Diagram 작성자에게 혼자 맡기면 안 되는 것

- Artifact naming과 묶음 기준
- Knowledge Base 표현 현실화
- 외부 시스템 역할군 기준 이후의 실제 접점 정리

### 내가 혼자 다 가져가면 안 되는 이유

- 사용자 진입 경로와 Web Application 제공 위치는 프론트가 더 정확하다
- Node 분해 수준과 구조적 추상화는 C4 작성자가 가장 잘 안다

즉, 내가 통합 책임을 가지되, 사용자 접점과 구조 원칙은 각 담당자가 정보를 내는 방식이 가장 좋다.

## 바로 착수할 수 있는 담당별 체크리스트

### 프론트 개발자

- [ ] 사용자 진입 Node 후보 정리
- [ ] Web Application 제공 위치 설명 정리
- [ ] 업로드가 어떤 Node를 통해 들어오는지 정리
- [ ] public 경계에서 보이는 외부 접점 정리

### C4 Diagram 작성 백엔드 개발자

- [ ] C2 컨테이너를 어떤 Node로 표현할지 초안 작성
- [ ] 포함/제외 Node 기준 정리
- [ ] 외부 시스템 역할군 표 상단의 기준 섹션 정리
- [ ] `Knowledge Base`, `Workflow & Integration`, `Operations Manager`의 분해 수준 초안 작성
- [ ] 상위 경계 초안 작성
- [ ] Node 간 연결 표 상단의 통신 표기 기준 정리

### 나

- [ ] 역할 중심 Artifact 후보 정리
- [ ] Knowledge Base 표현 결정표 상단의 기본 원칙 정리
- [ ] 외부 시스템 역할군별 실제 접점 정리
- [ ] 내부 연결을 통신 표기 기준으로 재작성
- [ ] 최종 Deployment Diagram 초안 통합

## 한 줄 정리

가장 좋은 분배는 아래다.

- 프론트 개발자: Entry Surface Owner
- C4 Diagram 작성 백엔드 개발자: Node Mapping Owner
- 나: Artifact & Integration Owner

이렇게 나누면 readiness 문서의 최신 기준과 싱크를 맞추면서, 메인 그림 복잡도를 낮춘 Deployment Diagram에 필요한 정보를 중복 없이 모을 수 있다.
