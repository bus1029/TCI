# 개발자 페르소나 — AI 개발 시대 재정의

> 작성일: 2026-03-03
> 목적: 기존 After 버전의 시나리오가 여전히 Pre-AI 맥락임을 인지하고, AI Agent가 코드를 생성하는 시대에 맞게 개발자 페르소나를 재정의
> 참조: 경쟁 제품 기능 MoSCoW 기반 분류 통합, 개발자 페르소나 정리(Notion)

---

| 구분 | Before | After (AI 개발 시대) |
|------|--------|----------------------|
| 이름 | | |
| 세부사항 | 개발자 (개발자)<br>• 레거시 투입 직후 빠르게 구조와 도메인을 파악하고 개발해야 함 | 개발자<br>• AI Agent(Cursor, Copilot 등)에게 기능 구현을 지시하고, AI가 생성한 코드를 검증해 머지 여부를 판단하는 입장 |
| TCI 도입 전 불만사항 | • 코드 구조 파악<br>    ◦ 외주 개발사가 바뀔 때마다 시스템 설명에 2~3주 소요<br>• 영향 범위 파악<br>    ◦ 영향 범위 파악 서비스 부재로 인한 리스크<br>    ◦ 변경 리스크 평가가 개인 역량과 구전 지식에 의존<br>    ◦ 변경 범위를 매번 수작업으로 추적 → 시간·비용 증대 및 누락 발생<br>• 문서 작성 리스크<br>    ◦ 기존 문서(설계서/ERD)와 실제 코드가 불일치<br>    ◦ 코드를 보고 문서 초안을 직접 쓰는 데 하루 소요<br>    ◦ QA/PM 설득을 위한 근거 자료 생산에 시간 과다<br>• 지식/맥락 단절<br>    ◦ 문서·티켓·지식이 파편화되어 누구에게 물어봐야 하는지 난감<br>    ◦ 시스템 전체 구조를 아는 사람이 본인 포함 2명뿐 | 1. AI가 생성한 코드가 기존 비즈니스 규칙을 따르는지 확인할 방법이 없음<br>    ◦ AI는 자신 있게 틀린 코드를 만들어냄<br>    ◦ "맞는 것 같긴 한데…" 수준의 불확실한 승인이 반복되고, 검증 실패 시 장애로 돌아옴<br>2. AI 변경의 영향 범위를 여전히 사람이 수작업으로 추적해야 함<br>    ◦ AI가 코드를 더 빠르게 더 많이 생성할수록 검증해야 할 양이 폭발적으로 증가<br>    ◦ 영향 추적 속도가 AI 생성 속도를 따라가지 못함<br>3. AI 생성 코드에는 "왜 이렇게 짰는지"가 처음부터 없음<br>    ◦ AI는 코드를 만들지만 설계 의도·비즈니스 컨텍스트를 남기지 않음<br>    ◦ PR 리뷰어가 판단할 근거가 없어서 리뷰 시간이 오히려 늘어남<br>4. AI에게 지시할 때 시스템 컨텍스트를 매번 수동으로 제공해야 함<br>    ◦ "결제 모듈 수정해줘"라고 해도 AI가 시스템 구조를 모름<br>    ◦ 컨텍스트 없이 지시하면 AI가 엉뚱한 곳을 수정하거나 기존 패턴을 무시한 코드를 생성<br>5. AI 생성 코드가 팀의 아키텍처 규약·패턴을 준수하는지 확인하기 어려움<br>    ◦ AI는 팀의 코딩 규칙을 모르고 일반적인 방식으로 코드를 생성<br>    ◦ 리뷰 시점에 규약 위반을 발견해도 수정 비용이 이미 발생한 상태 |
| 목표 | • 코드 구조 파악<br>    ◦ 신규 팀원이 별도 설명 없이 1주일 내 시스템 구조 파악 가능<br>    ◦ 코드 분석에 쓰는 시간을 줄이고 실제 개발 시간 확보<br>• 영향 범위 파악<br>    ◦ 수정 전 영향 범위를 사전에 파악해 리뷰 시 설명<br>    ◦ 변경 전 영향 범위 자동 산출 → 안전하게 수정 및 리팩토링<br>    ◦ PR 반려율 감소 / 변경으로 인한 장애·회귀 건수 감소<br>• 문서 작성 리스크 완화<br>    ◦ 문서 작성에 들이는 시간 절감<br>    ◦ 코드 변경과 문서가 항상 동기화된 상태 유지<br>• 지식/맥락 단절 해결<br>    ◦ 시스템 구조 지식을 개인이 아닌 조직 자산으로 전환<br>    ◦ 온보딩 기간 단축 | 1. AI 생성 코드가 기존 비즈니스 규칙대로 동작하는지 빠르게 검증 가능한 상태<br>2. AI 변경의 영향 범위를 자동으로 파악해 누락 없이 머지<br>3. AI 생성 코드에도 "왜 이렇게 짰는지"가 남도록 맥락을 자동 보완<br>4. AI Agent에게 시스템 컨텍스트를 자동으로 제공해 더 정확한 코드 생성 유도<br>5. 팀 아키텍처 규약 준수 여부를 리뷰 이전에 자동 검증 |
| 니즈 | • 코드 구조 파악<br>    ◦ 컴포넌트/레이어/의존성 시각화<br>    ◦ 자연어 설명 + 그래프<br>    ◦ 핵심 트랜잭션/업무 흐름 다이어그램 생성 및 근거 링크 제공<br>    ◦ 자연어 질문에 대한 근거 기반 답변<br>• 영향 범위 파악<br>    ◦ PR 생성 시 규칙 영향 자동 분석<br>    ◦ 변경 Diff를 넣으면 영향받는 모듈/트랜잭션/API를 추적해 체크리스트로 제시<br>    ◦ 변경 객체 기준 영향 (모듈, 컴포넌트, 의존성, DB)<br>    ◦ 데이터·서비스 흐름 기준 영향<br>• 문서 작성 리스크 완화<br>    ◦ 문서 내보내기 기능<br>• 지식/맥락 단절 해결<br>    ◦ 비즈니스 규칙 추출<br>    ◦ 멀티 소스 수집 (Git, Jira, Confluence, 업로드) 스냅샷 | 1. AI 생성 코드의 비즈니스 규칙 검증<br>    ◦ 코드에서 추출한 기존 규칙과 AI 생성 코드의 구현을 대조<br>    ◦ "이 AI가 짠 환불 로직이 기존 정산 규칙과 맞아?"를 근거와 함께 확인<br>    ◦ 연결 기능: 비즈니스 규칙 추출(Must), 자연어 질의(Must), Knowledge Base(Must)<br>2. AI 변경 Diff 기반 영향 분석 자동화<br>    ◦ AI가 생성한 Diff 기준으로 영향 받는 모듈·트랜잭션·API·DB를 자동 추적<br>    ◦ PR 생성 시 CI/CD 파이프라인에서 자동 실행되어 누락을 줄임<br>    ◦ 연결 기능: Change Impact Analysis(Must), CI/CD 연동(Must), 영향 범위 분석(Should)<br>3. AI 생성 코드에 맥락·근거 자동 주입<br>    ◦ AI가 만든 PR에 비즈니스 컨텍스트·영향 범위·검증 포인트를 자동 보완<br>    ◦ "왜 이렇게 짰는지"를 코드·흐름 링크 기반으로 채워줌<br>    ◦ 연결 기능: PR to Doc(Should), 비즈니스 규칙 추출(Must), 자동 문서·가이드 생성(Should)<br>4. TCI가 AI Agent의 컨텍스트 소스(MCP)로 동작<br>    ◦ Cursor/Copilot이 TCI에서 구조·흐름·규칙을 가져와 정확한 위치에 정확한 코드 생성<br>    ◦ 개발자가 매번 컨텍스트를 수동 제공하는 부담 제거<br>    ◦ 연결 기능: Knowledge Base(Must), IDE 통합 > MCP 서버(Could), Static Structure Analysis(Must)<br>5. 아키텍처 규약 준수 자동 검증<br>    ◦ AI 생성 코드가 팀 코딩 규칙·레이어 규약·패턴에서 벗어났는지 리뷰 전에 자동 탐지<br>    ◦ 규약 위반을 머지 전 단계에서 잡아 재작업 비용 방지<br>    ◦ 연결 기능: Virtual Code Review(Could), 변경 감지 및 요약(Should), Risk Insight(Should) |
