## 매핑 기준

'제품 기능 리스트 필터링(Must/Should/Could)'에서 대응 기능을 추출했다.
리스트에 정의되지 않은 기능은 카테고리를 **미정의**로 표기한다.

|---|---|---|
| 새 세션마다 동일 컨텍스트 반복 설명 | 세션 간 컨텍스트 연속성 확보 | 영속적 코드베이스 지식 저장소 |
| deprecated 경로·컨벤션 위반 반복 | 처음부터 올바른 컨텍스트 위에서 작업 | 에이전트 컨텍스트 패키지 자동 생성 |

## 기능 매핑 결과

| **카테고리** | **기능명** | **설명** | **매핑 근거** |
|---|---|---|---|
| 지식 자산화 | Knowledge Base Construction | 분석 결과를 구조화해 지식 모델/그래프를 구성하고 재탐색 가능하게 만든다 | 새 세션마다 코드베이스 컨텍스트를 반복 설명해야 하는 불만을 해소하려면, 에이전트 세션과 독립된 영속적 지식 저장소가 필요하다. 분석 결과를 누적·구조화하면 에이전트가 대화 리셋 없이 기존 지식을 재사용할 수 있다. |
| 미정의 | 에이전트 컨텍스트 패키지 자동 생성 | 특정 기능/모듈 수정 시작 시, 에이전트에게 주입할 컨텍스트 번들(구조 맵·흐름·규칙·주의사항 등)을 자동으로 생성한다 | 페르소나의 핵심 니즈인 "컨텍스트 번들을 작업 시작 시 자동으로 에이전트에게 주입"에 직접 대응하는 기능이나, 기존 기능 리스트에 정의된 기능이 없다. 별도 기획이 필요한 항목이다. |
| 시스템 구조 분석 | Static Structure Analysis | 코드 기반 정적 구조를 식별하고 구조 메타데이터/관계를 생성한다 | 에이전트가 deprecated 경로나 팀 컨벤션을 위반하는 이유는 코드베이스 구조를 파악하지 못하기 때문이다. 정적 구조 메타데이터가 있어야 컨텍스트 패키지에 정확한 구조 정보를 담을 수 있다. |
| 시스템 구조 분석 | Codebase Explanations | 정적 분석으로 모듈/컴포넌트를 자동 발견하고, 모듈별 파일 목록·콜 트리·AI 생성 설명을 제공한다 | 에이전트가 암묵적 컨벤션을 인식하지 못하는 문제를 해소하려면, 모듈별 역할과 구조를 자연어 설명으로 제공해야 한다. 이 설명이 에이전트 컨텍스트로 주입되면 잘못된 제안 반복을 사전에 차단할 수 있다. |
| 호출/의존성 분석 | Dependency Analysis | 정적 호출 기준으로 호출/의존 관계를 그래프로 구성하고, 순환 의존성 등 이상 패턴을 탐지한다 | 에이전트가 이미 존재하는 유사 함수·모듈을 인식하지 못해 중복 구현하는 근본 원인은 의존성 그래프가 없기 때문이다. 호출/의존 관계를 미리 파악하면 에이전트 작업 시 중복 생성을 예방할 수 있다. |
| 리스크 인사이트 | 리팩토링 | 코드 유사도/유틸리티 탐색으로 중복 코드 모듈화 후보를 제시한다 | 이미 중복 구현이 발생한 코드베이스에 대해, 유사 로직을 탐지하고 모듈화 후보를 제시함으로써 산재된 중복을 정리하고 유지보수 비용을 줄인다. |
| 영향도 분석 | Change Impact Analysis | 변경 Diff를 기준으로 영향 범위를 구조/트랜잭션 관점에서 추적한다 | 에이전트가 만든 변경의 영향 범위를 개발자가 수작업으로 검증해야 하는 병목을 해소하는 핵심 기능이다. Diff를 기준으로 영향받는 모듈·트랜잭션을 자동 추적하면 검증 시간을 절감할 수 있다. |
| 제어/데이터 흐름 분석 | 영향도 분석 (변수/화면 흐름 추적) | 변수/화면 흐름을 따라 값의 설정 → 전파 → 출력까지 데이터 흐름을 추적하고, 변경 Diff와 연동한다 | 구조 단위 영향도만으로는 데이터 흐름 경로의 파급 범위를 파악하기 어렵다. 변수·화면 단위의 흐름 추적을 Diff와 연동하면 영향 검증의 정밀도가 높아진다. |
| 영향도 분석 | 코드 수정 사항 제안 | 변경(스펙/UI)이 미치는 영향을 분석해 백엔드·프론트엔드 수정 포인트를 제시한다 | 영향 범위 파악에서 나아가 실제 수정이 필요한 지점까지 제시해 줌으로써, 개발자가 직접 영향 범위를 탐색하고 수정 포인트를 찾는 수작업을 줄인다. |
| 영향도 분석 | Trade-off 시각화 | 변경 전후 주요 변경점·영향 범위·지표 변화를 트레이드오프 보고서로 생성한다 | PR 작성 시 변경의 근거와 영향 범위를 개발자가 직접 작성해야 하는 부담을 줄인다. 보고서 형태로 변경 전후를 정량화하면 팀에 설명하는 비용도 함께 낮아진다. |
| 워크플로우 자동화 | Documentation as Code (PR to Doc) | PR Diff 기반으로 문서 초안(스니펫·AI 본문 포함)을 자동 생성하고, 수동 작성 옵션도 지원한다 | 에이전트가 코드를 생성했더라도 PR 설명·변경 근거는 개발자가 직접 써야 하는 불만에 직접 대응한다. Diff 기반으로 문서 초안을 자동 생성하면 추적 가능한 근거를 에이전트 결과물에 자동으로 부착할 수 있다. |

## 추가 기능 (페르소나 검토 후 추가)

기존 매핑 결과(11개)를 페르소나 불만사항·니즈 흐름과 재대조한 결과, 아래 4개 기능이 누락된 것으로 확인됐다.

| **카테고리** | **기능명** | **설명** | **대응 불만사항** | **추가 근거** |
|---|---|---|---|---|
| 워크플로우 자동화 | 변경 감지 및 요약 | 코드/문서/라이브러리 변경을 감지하고 요약·알림·의도 추론을 제공한다 | deprecated 경로·컨벤션 위반 반복 | 정적 구조 분석은 현재 상태 스냅샷만 제공한다. 변경 감지 없이는 코드베이스가 바뀌어도 지식 저장소가 갱신되지 않아 에이전트가 deprecated 경로를 계속 따라가게 된다. "버전 변경 요약·Deprecated 예정 알림"이 컨텍스트 최신성 유지를 직접 담당한다. |
| 외부 채널 연동 | IDE 통합 | IDE 안에서 문서 생성·편집·탐색·질의를 지원하며, MCP 서버를 통해 변경 영향 범위 등을 실시간으로 제공한다 | deprecated 경로·컨벤션 위반 반복 | 페르소나 니즈에 명시된 "Cursor Rules·Claude Project Knowledge·MCP Tool Response 포맷에 맞춰 바로 주입 가능한 형태로 제공"은 에이전트 컨텍스트 패키지 자동 생성의 실질적 딜리버리 채널이다. 미정의 기능의 구현 방법론과 직결되므로 함께 매핑한다. |
| 리스크 인사이트 | Risk Insight | 구조/변경 기반으로 리스크 요소를 설명하고 우선 점검 대상을 제시한다 | 영향 범위를 개발자가 수작업으로 검증 | 영향 범위가 넓을 때 어디부터 검증해야 하는지 우선순위가 없으면 자동화가 절반짜리가 된다. Change Impact Analysis가 영향 범위의 "무엇"을 알려준다면, Risk Insight는 "얼마나 위험한지"를 알려주는 보완재로 수작업 검증 부담을 실질적으로 낮춘다. |
| 리스크 인사이트 | 테스트 | 변경/분석 결과를 바탕으로 테스트 품질 개선 포인트와 신규 테스트 시나리오를 제시한다 | 영향 범위를 개발자가 수작업으로 검증 | 영향 범위를 자동으로 파악했더라도 그 범위에 대한 검증 수단이 없으면 개발자의 수작업 검증 부담은 완전히 해소되지 않는다. 영향 분석 결과를 테스트 시나리오로 연결해야 검증 자동화 루프가 닫힌다. |

## 검증 결과 — 서브에이전트 추가 검토 (2026-03-05)

기존 매핑 전체(15개)를 Notion 원문과 재대조한 결과, 추가로 **4개 누락 기능**이 확인됐다.
자연어 기반 질의 응답(Ask Swimm, Ask Kodesage)은 이 페르소나에서 자연어 Q&A 인터페이스 역할을 Cursor/Claude가 이미 담당하므로 제외한다. TCI의 역할은 지식 저장소 구축과 컨텍스트 공급이며, 딜리버리 채널은 IDE 통합(MCP)이 커버한다.

### 누락 기능

| **카테고리** | **기능명** | **MoSCoW** | **대응 불만사항** | **누락 이유** |
|---|---|---|---|---|
| 제어/데이터 흐름 분석 | 비즈니스 규칙 추출 (Create specs) | Must | deprecated 경로·컨벤션 위반 반복 | 페르소나 니즈는 컨텍스트 패키지 구성 요소를 "구조 맵 + 흐름 + **규칙** + 주의사항"으로 명시했다. 구조 맵·흐름은 커버됐으나 비즈니스 규칙을 추출하는 기능이 빠져 컨텍스트 패키지의 "규칙" 구성 요소에 공백이 생긴다. |
| 시스템 구조 분석 | 아키텍처 맵 생성 | Should | deprecated 경로·컨벤션 위반 반복 / 중복 구현 | 컨텍스트 패키지의 "구조 맵"에 직접 대응한다. Static Structure Analysis가 파일·클래스 단위 메타데이터를 생성한다면, 아키텍처 맵은 이를 시스템 전체 모듈 간 관계로 종합하는 상위 산출물로 중복 구현 예방의 핵심 인풋이다. |
| 호출/의존성 분석 | Change Coupling | Should | 영향 범위를 개발자가 수작업으로 검증 | Change Impact Analysis(정적 구조 기반)만으로는 커밋 이력 기반의 암묵적·시간적 의존성(temporal coupling)을 발견할 수 없다. 두 기능이 함께 있어야 영향 범위 자동 검증의 사각지대가 줄어든다. |
| 워크플로우 자동화 | CI/CD 연동 | Must | 영향 범위 수작업 검증 / PR 근거 개발자 혼자 작성 | 영향 분석·PR 문서 자동 생성 기능이 CI/CD 파이프라인과 연동되어야 PR 제출 시점에 자동으로 분석·문서화가 트리거된다. 연동 없이는 IDE에서만 실행 가능한 반쪽짜리 자동화가 된다. |

### 기존 매핑 중 연결이 상대적으로 약한 기능

| **기능명** | **약한 이유** |
|---|---|
| 리팩토링 | 불만 #3(중복 구현)의 **사후 처리**에 해당. 페르소나가 원하는 건 예방이지만, 리팩토링은 이미 발생한 중복을 정리하는 remediation 기능으로 핵심 니즈에서 한 단계 거리가 있다. |
| 코드 수정 사항 제안 | 불만 #4(영향 범위 검증)에서 파생됐으나, 실제 니즈는 "영향 범위 파악 자동화"이지 "수정 포인트 제시"까지는 아니다. 유용한 부가 기능이나 핵심 불만사항에 직접 대응하는 기능은 아니다. |
