## 1. 프로젝트 개요

### 제품명
**TCI (Tmax Code Intelligence)**

### 제품 정체성
TCI는 엔터프라이즈 애플리케이션의 코드·문서·티켓 등을 연결해 시스템을 **'설명 가능한 구조'**로 재구성하고, 변경이 발생했을 때 영향·리스크·검증 포인트를 **근거 기반**으로 제시하는 **분석·검증 중심 코드 인텔리전스** 제품.

### 핵심 파이프라인
```
데이터 수집/연동
  → 정적 구조·관계 생성
  → 비즈니스 규칙/스펙 추출
  → 지식 베이스화
  → 변경 diff 기반 영향 분석
  → 자연어 질의 응답
  → 엔터프라이즈 운영/자동화
```

### 개발 배경
- 시장에는 '코드를 작성해주는 툴'과 '바꿔주는 툴'은 있지만, **'정확하게 파악하고 가이드해주는 엔진(Intelligence)'은 부재**
- IDE·정적 분석 툴·AI agent 툴에서 분석된 코드 정보가 연결되지 않고 휘발 → 지속 가능한 자산으로 관리할 지식 베이스 필요
- 티맥스소프트가 미들웨어(JEUS)로 시장 장악 → 이제는 애플리케이션의 생산과 진화를 책임지는 **AI 코드 분석 인프라**로 포트폴리오 확장

### 품질/메트릭 분석 방침
- **품질/메트릭 분석은 TCI가 자체 개발하지 않음**
- 외부 툴(SonarQube 등)이나 오픈소스를 활용하여 나온 결과를 표시만 할 예정
- TCI 자체 역량: 구조 분석, 영향 분석, 지식 베이스, 비즈니스 규칙 추출, 자연어 질의, 흐름 분석, 코드 중요도 분류, AI 코드 감사, 규정 준수 검증

### ⚠️ AI Agent 시대 제품 방향 논의 (세션 3 신규)
팀장 논의에서 제기된 중요한 방향성 이슈:
- 기존 MoSCoW 분류는 **AI Agent 이전 시대**에 개발된 경쟁 제품(CAST Imaging, SWIMM, CodeScene, Kodesage) 분석 기반
- TCI는 **AI Agent 이후 시대**에 출시될 제품
- 따라서 MoSCoW 기반 기능 목록은 유효하지만, **페르소나의 맥락(사용 시나리오)은 AI Agent 시대로 업데이트 필요**

| 구분 | Pre-AI Agent | Post-AI Agent |
|------|-------------|---------------|
| 개발자 핵심 활동 | 코드 직접 작성 + 이해 | AI 생성 코드 검토 + 검증 |
| 병목 | 코드 이해·탐색 시간 | 검증·리뷰 시간 (오히려 증가) |
| 지식 단절 원인 | 사람 퇴사 시 지식 소멸 | AI 생성 코드는 처음부터 "왜 이렇게 짰는지"가 없음 |
| TCI 역할 | 사람이 코드를 이해하도록 보조 | AI가 생성한 코드를 검증 + AI Agent에게 시스템 컨텍스트 제공(MCP) |

> 결론: MoSCoW 기능 자체는 유효하거나 일부 더 중요해지지만, 페르소나 시나리오의 맥락을 AI Agent 이후로 업데이트하는 작업이 필요하다. 이 방향은 팀 내부 정렬 후 페르소나 재검증에 반영 예정.

> 이 버전이 현재 팀의 최신 방향. 개인별 상세 페르소나(5-1)보다 우선함.

**메인 페르소나 (2종)**
| 역할 | 핵심 pain point | 핵심 니즈 |
|------|----------------|----------|
| 개발자 (개발자) | 진입점 탐색·영향 범위 추적·문서 낡음·지식 단절 | 구조 시각화, Diff 기반 영향 분석, 비즈니스 규칙 추출, 멀티소스 수집 |
| 개발자 (리뷰어) | PR 영향 범위 수동 추적, AI 코드 리뷰 부하, 승인 근거 분산 | Diff 기반 영향 추적, 리뷰 체크리스트 자동화, PR 리포트 공유 |

**서브 페르소나 (2종, 진행 중)**
| 역할 | 상태 | 비고 |
|------|------|------|
| BA (Business Analyst) | 초안 작성 중 | 로컬 `BA_페르소나_초안.md` 참조 |

### 5-3. BA 페르소나 초안 (세션 3 작성, Notion 미반영)

로컬 파일 `BA_페르소나_초안.md` 내용 요약:

**TCI 도입 전 불만사항**
1. 비즈니스 규칙이 코드에 어떻게 구현됐는지 확인할 수단 없음
2. 요구사항 변경 영향 분석이 개발자 인터뷰에 의존해 병목
3. 코드와 기획서(스펙) 사이의 갭을 인지하는 타이밍이 항상 늦음
4. 레거시 시스템의 현행 업무 규칙 파악에 너무 오래 걸림

**목표**
- 기획서·스펙과 실제 구현 사이의 갭을 개발자 없이 직접 확인
- 요구사항 변경 시 영향 범위를 사전에 파악해 근거 기반 일정 협의
- 현재 시스템 업무 규칙을 자연어로 탐색 가능
- 이해관계자 보고 시 코드 근거가 붙은 신뢰도 높은 산출물 생산

**니즈 (MoSCoW 기능 연결 포함)**
- 비즈니스 규칙 파악: 비즈니스 규칙 추출(Must), 자연어 질의(Must), Knowledge Base(Must)
- 요구사항 변경 영향 파악: Change Impact Analysis(Must), Trade-off 시각화(Should), 코드 수정 사항 제안(Should)
- 스펙 vs 구현 갭 탐지: 문서와 코드 간 불일치 분석(Could), Data Sources 티켓 연동(Must)
- 비개발자 친화 리포팅: 자동 문서·가이드 생성(Should), 비즈니스 임팩트 번역(미확정 제안), 비기술자 다이어그램(미확정 제안)

**BA 페르소나 평가 (시니어 BA 관점)**
- 잘 된 부분: 핵심 pain point(요구사항-구현 갭, 개발자 인터뷰 의존 병목) 포착
- 보강 필요: 감리/검수 대응(공공/금융 타겟 시 필수), UAT 지원, 변경 감지 니즈 반영
- 감리/검수는 타겟 산업군(공공/금융 여부)에 따라 중요도 달라짐 — 팀 내 결정 필요

### 5-4. 제거된 페르소나
