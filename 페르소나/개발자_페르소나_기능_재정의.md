# 기능 재정의 — 에이전트 컨텍스트 패키지 자동 생성 → IDE 통합(MCP)

> 시나리오 검토 결과, '미정의' 상태로 남아 있던 "에이전트 컨텍스트 패키지 자동 생성" 기능의 상호작용 모델이
> 시나리오와 어긋난다는 점이 확인됐다. 이 문서는 그 근거와 대체 기능 정의를 기록한다.

---

## 기존 기능의 가정

| 항목 | 내용 |
|------|------|
| **기능명** | 에이전트 컨텍스트 패키지 자동 생성 |
| **카테고리** | 미정의 |
| **설명** | 특정 기능/모듈 수정 시작 시, 에이전트에게 주입할 컨텍스트 번들(구조 맵·흐름·규칙·주의사항)을 자동으로 생성한다 |
| **전달 방식** | 개발자가 번들을 직접 생성 → Cursor Rules·Claude Project Knowledge 등에 수동으로 붙여넣기 |
| **행위 주체** | 개발자가 트리거 |

---

## 시나리오에서 드러난 실제 상호작용 모델

시나리오(VIP 할인율 수정 작업 40분)의 해당 장면:

> 전트가 "VIP 할인 정책을 수정해야 해"라고 입력하자,
> **에이전트**는 TCI MCP 서버에 해당 기능과 관련된 컨텍스트를 요청한다.
> TCI는 할인 로직 관련 모듈 경로, 파일 목록, 팀 위키의 주의사항 등이 포함된
> 컨텍스트 번들을 생성하여 **에이전트에게 반환**한다.

기존 기능이 가정한 흐름과 실제 시나리오의 흐름을 비교하면 다음과 같다.

| 구분 | 기존 가정 | 시나리오 실제 |
|------|-----------|---------------|
| 트리거 주체 | 개발자 | 에이전트 |
| 전달 방식 | 개발자가 번들 생성 후 수동 붙여넣기 | 에이전트가 MCP 서버에 질의 → 온디맨드 반환 |
| 컨텍스트 포맷 | Cursor Rules·Claude Project Knowledge 등 정적 포맷 | MCP Tool Response (에이전트가 런타임에 수신) |
| 개발자 개입 | 필요 | 없음 |

---

## 결론 — 기능 재정의

"에이전트 컨텍스트 패키지 자동 생성"이 해결하려던 문제(세션 간 컨텍스트 연속성·올바른 컨텍스트 위에서 즉시 작업)는 시나리오에서 **IDE 통합(MCP 서버)** 이 담당하고 있다.

따라서 이 기능을 독립 항목으로 유지하는 대신, **IDE 통합** 기능의 범위를 확장·구체화하는 방향으로 재정의한다.

## 변경 기능 정의

### 제거

| **카테고리** | **기능명** | **제거 이유** |
|---|---|---|
| 미정의 | 에이전트 컨텍스트 패키지 자동 생성 | 시나리오의 실제 상호작용 모델(에이전트 → MCP 질의 → 온디맨드 반환)과 기능 설계가 어긋난다. IDE 통합(MCP)이 동일 문제를 더 정확한 모델로 커버한다. |

### 대체

| **카테고리** | **기능명** | **설명** | **대응 니즈** |
|---|---|---|---|
| 외부 채널 연동 | IDE 통합 (MCP 서버) | 에이전트가 작업 의도에 따라 TCI MCP 서버에 질의하면, TCI가 관련 모듈 경로·파일 목록·팀 컨벤션·주의사항을 컨텍스트 번들로 온디맨드 반환한다. 개발자의 수동 개입 없이 에이전트가 올바른 컨텍스트 위에서 즉시 작업할 수 있는 상태를 만든다. | 에이전트 세션과 독립된 영속적 코드베이스 지식 저장소 / 에이전트 컨텍스트 번들 자동 생성 |

### MCP 서버가 온디맨드로 반환해야 하는 컨텍스트 구성 요소

| **구성 요소** | **소스 기능** |

| 파일·클래스 단위 메타데이터 | Static Structure Analysis |

| 팀 위키·주의사항 | Data Sources |
| deprecated 경로·버전 변경 알림 | 변경 감지 및 요약 |
