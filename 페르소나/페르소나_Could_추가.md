# Could + 제품 방향 기반 페르소나

> 전제: 품질/메트릭 분석은 TCI가 자체 개발하지 않고, 외부 툴(SonarQube 등) 결과를 표시만 함.
> 따라서 아래 페르소나의 핵심 니즈는 모두 TCI 자체 분석 역량에 기반하며, 외부 품질 데이터는 보조 참고 수준.

---

## 페르소나 선정 근거

| 페르소나 | 선정 이유 |
|----------|-----------|
| 최원호 (QA 리드) | TCI의 "검증" 축을 가장 직접적으로 대표. AI 시대에 코드 검증/규정 준수 자동화 수요 급증 |
| 김태영 (마이그레이션 PM) | 엔터프라이즈(금융/공공) 차세대 프로젝트는 TCI의 핵심 영업 시나리오. TCI 자체 분석만으로 핵심 가치 전달 가능 |
| 서지훈 (개발 조직장) | 오진석(보안팀장) 제거 후 부재한 "구매 결정권자" 역할. 조직 전체 수준의 기술 의사결정자 |

## QA 리드 / 검증 자동화 담당

| 항목 | 내용 |
|------|------|
| 이름 | 최원호 |
| 세부 정보 | • 36세, 대형 보험사 IT센터 9년차 QA 리드 • 담당 시스템: 보험 계약/심사 플랫폼 (Java 기반, 약 90만 LOC) • QA 엔지니어 5명 리딩, 분기 릴리즈마다 회귀 테스트 계획 및 품질 게이트 운영 담당 |
| TCI 도입 전 불만사항 | • 변경이 발생할 때 "어디까지 테스트해야 하는지" 근거 없이 전체 회귀 테스트를 돌림 ◦ "테스트 범위를 줄이고 싶은데, 줄였다가 장애 나면 QA 책임이니까 결국 전부 돌립니다" • AI 코파일럿으로 생성된 코드가 기존 아키텍처 규약에 맞는지 일일이 수동 리뷰 ◦ "AI가 만든 코드가 우리 패턴이랑 안 맞아서 오히려 리뷰 시간이 더 걸립니다" • 금융감독원 감사 때마다 코딩 표준 준수 여부를 수작업으로 취합 ◦ "감사 대응 리포트 만드는 데만 2주 걸립니다" |
| 목표 | • 변경 영향 기반 테스트 범위 자동 산정으로 회귀 테스트 비용 50% 절감 • AI 생성 코드의 규약 준수 여부를 자동 검증해 리뷰 부담 50% 감소 • 감사 대응 리포트를 자동 생성해 대응 기간 2주 → 2일 단축 |
| 니즈 | • **테스트** (Should): 변경 기반 누락 테스트 자동 제시, 우선 테스트 대상 추천, 테스트 시나리오 제안 — TCI 자체 역량 • **AI 생성 코드 감사** (추가 제안): AI가 작성한 코드가 아키텍처 패턴/보안 규약을 준수하는지 자동 검증 — TCI 자체 역량 • **규정/코딩 표준 준수 검증** (추가 제안): 산업 규정·내부 코딩 표준 자동 체크 + 감사 리포트 생성 — TCI 자체 역량 • **Virtual Code Review** (Could): TCI 지식 베이스 기반 문제 원인/리팩토링 권장 리포트 — TCI 자체 역량 • **Change Impact Analysis** (Must): 변경 범위 기반 검증 포인트 자동 도출 — TCI 자체 역량 • 품질 점수/기술 부채 현황 (Could): 외부 툴 결과를 표시해 참고 데이터로 활용 — 외부 연동 |

> 기존 페르소나와의 차별점: 한민석(테크리드)은 "릴리즈 승인/PR 리뷰" 관점, 최원호는 "테스트 전략 + AI 코드 검증 + 감사 대응" 관점.

## 마이그레이션/현대화 프로젝트 PM

| 항목 | 내용 |
|------|------|
| 이름 | 김태영 |
| 세부 정보 | • 42세, IT 컨설팅 기업 15년차 시니어 PM • 현재 대형 공공기관 차세대 프로젝트 (COBOL/Pro*C → Java/Spring 전환) 총괄 • 대상 시스템: 약 200만 LOC, 20년 운영. 전환 기간 2년, 6개 협력사 80명 참여 |
| TCI 도입 전 불만사항 | • 레거시 시스템의 전체 구조를 파악하는 데만 3개월 소요 ◦ "200만 줄 코드를 누가 어떻게 다 읽습니까. 구조를 모르니 전환 전략을 못 세웁니다" • 어떤 컴포넌트부터 전환해야 하는지 근거 없이 "쉬운 것부터" 전환 ◦ "전환 순서를 잘못 잡아서 의존성 때문에 2개월 지연된 적 있습니다" • 전환된 부분과 미전환 부분 사이 인터페이스 영향을 수동 추적 ◦ "전환한 모듈 A가 아직 안 바꾼 모듈 B, C에 어떤 영향을 주는지 확인하는 게 악몽입니다" |
| 목표 | • 레거시 시스템 구조를 1개월 내 파악하고 전환 로드맵 수립 • 컴포넌트별 전환 난이도/의존성 기반 최적 전환 순서 도출 • 전환 진행 중 잔여 레거시와의 영향 범위를 실시간 추적 |
| 니즈 | • **Static Structure Analysis** (Must): 200만 LOC의 컴포넌트/레이어/의존성을 자동 식별 — TCI 자체 역량 • **Control & Data Flow Analysis** (Could): 레거시 코드의 비즈니스 트랜잭션 흐름을 추적해 전환 단위 식별 — TCI 자체 역량 • **코드 중요도 라벨링** (Could): 핵심 비즈니스 로직 vs 유틸리티를 자동 분류해 전환 우선순위 판단 — TCI 자체 역량 • **Change Impact Analysis** (Must): 전환 완료 모듈 ↔ 미전환 모듈 간 영향 범위 실시간 추적 — TCI 자체 역량 • **문서와 코드 간 불일치 분석** (Could): 20년 된 설계서와 실제 코드의 괴리를 자동 식별 — TCI 자체 역량 • **Knowledge Base** (Must): 분석 결과를 축적해 협력사 80명이 동일 수준으로 시스템 이해 — TCI 자체 역량 • **마이그레이션 영향 분석 & 로드맵** (추가 제안): 의존 관계 기반 최적 전환 순서 및 단계별 로드맵 자동 생성 — TCI 자체 역량 |

> 기존 페르소나와의 차별점: 박정우(아키텍트)는 "운영 중 시스템의 유지보수/변경" 관점, 김태영은 "레거시 → 차세대 전환 프로젝트" 관점. 엔터프라이즈(금융/공공) 차세대 프로젝트는 TCI의 핵심 영업 시나리오.

## 개발 조직장 / VP of Engineering

| 항목 | 내용 |
|------|------|
| 이름 | 서지훈 |
| 세부 정보 | • 47세, 중견 금융IT 자회사 개발본부장 (VP) • 산하 4개 팀 (개발 50명, QA 10명), 관리 시스템 5개 (총 약 400만 LOC) • TCI 도입 예산 승인/ROI 판단 권한 보유 • 분기 1회 경영진 보고 (기술 현황/리스크) |

| 목표 | • 산하 전체 시스템의 구조적 건전성과 변경 리스크를 조직 레벨 대시보드로 파악 • 기술 투자 (리팩토링/현대화) 우선순위를 TCI 분석 데이터 기반으로 결정 • 인력 변동에도 흔들리지 않는 조직 지식 자산 체계 구축 |
| 니즈 | • **조직 레벨 구조 분석 대시보드** (추가 제안): 시스템별 구조 복잡도, 의존성 밀도, 변경 영향 반경 추이를 경영진 관점으로 요약 — TCI 자체 역량 (구조 분석의 집계) • **기술 부채 우선순위 제안** (Could): TCI의 구조/영향도/사용빈도 분석 기반으로 해결 우선순위 자동 정렬 — TCI 자체 역량 (+ 외부 품질 데이터 보조 참고) • **Knowledge Base** (Must): 분석 결과와 판단 근거를 축적해 인력 변동에도 운영이 흔들리지 않는 자산화 — TCI 자체 역량 • **Risk Insight** (Should): 시스템별 구조/변경 기반 리스크 요소를 식별해 사전 대응 — TCI 자체 역량 • **AI 생성 코드 감사** (추가 제안): 조직 전체 AI 생성 코드의 규약 준수율/리스크 현황 집계 — TCI 자체 역량 • **Enterprise Operation** (Must): 멀티 시스템/대규모 조직 운영, 접근 제어, 감사 로그 — TCI 자체 역량 • 외부 품질 메트릭 (Could): SonarQube 등 외부 도구 결과를 참고 데이터로 함께 표시 — 외부 연동 (보조) |

> 기존 페르소나와의 차별점: 개별 시스템을 직접 다루는 다른 페르소나들과 달리, "조직 전체 수준"에서 기술 의사결정을 하는 경영층. 오진석 제거 후 부재했던 구매 결정권자 역할도 겸함.
