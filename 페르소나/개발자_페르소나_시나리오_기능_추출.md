# 시나리오 기반 추가 기능 추출 — 개발자 페르소나

> 시나리오(VIP 할인율 수정 작업 40분)를 장면 단위로 분해해, 기존 기능 매핑에서 누락된 기능을 추가로 도출했다.

---

## 시나리오 장면별 분석

### 장면 1 — 새 세션 시작, TCI MCP에서 어제 작업 컨텍스트 복원

> "어제 전트가 탐색했던 모듈 경로, 관련 파일 목록, 팀 위키의 주의사항을 그대로 반환"

**팀 위키의 주의사항** → Data Sources(Must Have) 기능이 필요하다. 이전에 인프라성 전제조건으로 제외했지만, 팀 위키가 컨텍스트 패키지의 핵심 소스로 쓰이고 있다. 위키를 수집하지 않으면 이 장면 자체가 불가능하다.

**어제 탐색했던 모듈 경로·파일 목록** → 기존 "에이전트 컨텍스트 패키지 자동 생성"과 다르다. 그 기능은 분석 결과를 패키징하는 것인데, 이건 개발자/에이전트가 실제로 탐색한 이력을 저장하고 복원하는 것이다. 세션이 리셋돼도 "어디까지 봤는지"를 TCI가 기억하는 별도 능력이 필요하다.

---

### 장면 2 — TCI 플러그인에서 코드 스냅샷 전송 → 영향도 분석 실행

> "전트는 TCI 플러그인에서 '변경 코드 영향도 분석' 버튼을 클릭하고 현재 코드의 스냅샷을 TCI로 전송한다.
> 웹 UI의 영향 분석 화면을 열자 변경된 파일 목록이 자동으로 불러와져 있고 '분석 실행' 버튼을 누르자..."

Change Impact Analysis는 기존에 매핑됐지만, 그 분석이 실행되려면 **IDE에서 코드 변경을 TCI 서버로 먼저 전달**해야 한다. 이 전송 행위 자체가 별도 기능이다. 에이전트가 MCP를 통해 TCI에서 컨텍스트를 가져오는 흐름(TCI → IDE)과 달리, 스냅샷 전송은 반대 방향(IDE → TCI)이다. 플러그인이 이 역할을 담당하며, 이 기능 없이는 Change Impact Analysis의 입력값 자체가 존재하지 않는다.

---

### 장면 3 — 위험 신호 배지 클릭 → 호출 경로 그래프 확인

> "해당 항목을 클릭해 호출 경로를 그래프로 확인"

이전에 시각화/탐색(Could)을 "개발자가 직접 탐색하는 UX 기능"이라고 제외했으나, 이 장면의 맥락은 다르다. 리스크 신호를 눌러 포인트 모듈이 실제로 영향받는지 검증하는 것이다. 탐색 목적이 아니라 영향 분석 결과의 시각적 검증 수단으로 쓰이고 있어서 제외 근거가 약해졌다.

---

### 장면 4 — PR 초안 생성 → Jira 티켓 링크 자동 연결

> "각 항목이 코드 위치와 Jira 티켓 링크로 연결된 PR 본문이 생성"

Documentation as Code(PR to Doc)는 이미 매핑됐지만, Jira 티켓 링크를 PR 본문에 자동으로 연결하려면 티켓 플랫폼 연동이 별도로 필요하다. Issue Analysis(Could)가 이 역할을 담당하는데, 이전에는 pain point와 연결이 없다고 제외했다. 하지만 시나리오에서 PR 근거 자동 생성(불만 #5)의 구성 요소로 명확히 사용되고 있다.

## 추출 기능 목록

| **카테고리** | **기능명** | **시나리오 장면** | **대응 불만사항** | **추가 근거** |
|---|---|---|---|---|
| 데이터 수집 | Data Sources | 팀 위키 주의사항을 TCI가 수집·반환 | deprecated 경로·컨벤션 위반 반복 | 팀 위키를 데이터 소스로 수집하지 않으면 "팀 컨벤션 주입"이 불가능하다. 비즈니스 규칙 추출·컨텍스트 패키지 자동 생성의 전제 소스로, 인프라가 아닌 컨텍스트 품질의 핵심 인풋이다. |
| 외부 채널 연동 | 코드 스냅샷 전송 (플러그인) | TCI 플러그인에서 '변경 코드 영향도 분석' 버튼 클릭 → 코드 스냅샷을 TCI 서버로 전송 | 영향 범위를 개발자가 수작업으로 검증 | Change Impact Analysis의 전제 조건이다. 에이전트가 MCP를 통해 TCI에서 컨텍스트를 가져오는 방향(TCI → IDE)과 반대로, 플러그인이 코드 변경을 서버로 밀어 넣는 역할(IDE → TCI)을 담당한다. 이 기능 없이는 영향도 분석의 입력값 자체가 존재하지 않는다. |
| 미정의 | 작업 세션 이력 저장 및 복원 | 어제 탐색한 모듈 경로·파일 목록을 다음 세션에서 복원 | 새 세션마다 동일 컨텍스트 반복 설명 | 에이전트 컨텍스트 패키지 자동 생성이 "분석 결과 기반 패키징"이라면, 이 기능은 "탐색 이력 기반 복원"이다. 두 기능이 합쳐져야 "반복 설명 없이 바로 작업 가능한 상태"가 완성된다. |
| 제어/데이터 흐름 분석 | 시각화/탐색 | 위험 신호 클릭 → 호출 경로 그래프로 영향 검증 | 영향 범위를 수작업으로 검증 | 탐색 목적이 아닌 영향 분석 결과의 검증 수단으로 사용된다. Risk Insight가 "어디가 위험한지"를 알려준다면, 시각화/탐색은 그 위험 신호를 개발자가 직접 눈으로 확인하는 접점이다. |
| 워크플로우 자동화 | Issue Analysis | Jira 티켓 링크를 PR 본문에 자동 연결 | PR 근거·설명은 개발자 혼자 작성 | PR 본문에 Jira 티켓 링크가 자동으로 연결되어야 "추적 가능한 근거"가 완성된다. Documentation as Code가 PR 본문 생성을 담당한다면, Issue Analysis는 그 본문과 티켓을 연결하는 고리다. |
