| TCI 도입 전 불만사항 | **비즈니스 규칙이 코드에 어떻게 구현됐는지 확인할 수단이 없다**<br>• "내가 작성한 요구사항이 코드에 제대로 반영됐는지 확인하려면 개발자를 따로 불러야 합니다"<br>• "3년 전 스펙 문서에 적힌 업무 규칙이 지금도 살아있는지, 언제 바뀌었는지 아무도 몰라요"<br>• "기능 인수 테스트를 했는데 6개월 후에 스펙과 다르게 구현된 로직을 발견했습니다"<br><br>**요구사항 변경 영향 분석이 개발자 인터뷰에 의존해 병목**<br>• "변경 요청을 올리면 '다른 모듈과 엮여 있어서 공수가 크다'는 답이 돌아오는데, 그 말을 검증할 방법이 없습니다"<br>• "영향 범위 추정이 늘 개발자의 주관적 판단이어서 일정이 틀렸을 때 근거 자료가 없습니다"<br>• "요구사항 변경 → 개발자 인터뷰 → 일정 협의까지 기본 3일은 걸립니다"<br><br>**코드와 기획서(스펙) 사이의 갭을 인지하는 타이밍이 항상 늦다**<br>• "Jira 티켓이 Done으로 닫혀있어도 내가 원한 대로 구현됐는지 알 수가 없습니다"<br>• "갭을 발견하는 시점이 개발 완료 후 QA 단계 혹은 릴리즈 이후여서 수정 비용이 큽니다"<br><br>**레거시 시스템의 현행 업무 규칙을 파악하는 데 너무 오래 걸린다**<br>• "10년 된 시스템에 신규 요구사항을 추가해야 하는데, 기존에 어떤 업무 규칙들이 있는지 아무도 정확히 모릅니다"<br>• "담당 개발자도 코드를 뒤져야 해서, 분석 결과를 며칠씩 기다리는 게 일상입니다" |
| 목표 | • 기획서·스펙과 실제 구현 사이의 갭을 개발자 없이 직접 확인 가능한 수준 달성<br>• 요구사항 변경 시 영향 범위를 사전에 파악해 일정 협의 시 근거 기반 대화<br>• 현재 시스템에 어떤 업무 규칙이 살아있는지 자연어로 탐색 가능<br>• 이해관계자(경영진·고객사) 보고 시 코드 근거가 붙은 신뢰도 높은 산출물 생산 |
| 니즈 | **비즈니스 규칙 파악**<br>• 코드에서 도메인 로직을 자연어 스펙으로 자동 추출 — 기획서와 비교 가능한 형태로 제공 (Must: 비즈니스 규칙 추출)<br>• "보험 해지 환급금 계산이 어떤 조건으로 분기되나요?" 처럼 업무 언어로 질의 → 근거 링크 포함 답변 (Must: 자연어 질의)<br>• 추출된 업무 규칙을 도메인별로 카탈로그화해 팀 자산으로 관리 (Must: Knowledge Base Construction)<br><br>**요구사항 변경 영향 파악**<br>• 스펙·업무 규칙 변경 시 영향받는 모듈·API·트랜잭션을 자동 추적 (Must: Change Impact Analysis)<br>• 변경안 A vs B의 영향 범위를 비교해 의사결정 근거 제공 (Should: Trade-off 시각화)<br>• 비즈니스 로직 변경이 소스 코드 어디에 영향을 주는지 역방향 추적 (Should: 코드 수정 사항 제안)<br><br>**스펙 vs 구현 갭 탐지**<br>• 업로드한 스펙 문서와 실제 코드 구현 사이의 불일치를 자동 탐지 (Could: 문서와 코드 간 불일치 분석)<br>• Jira 티켓이 실제 어떤 코드 변경으로 구현됐는지 연결해 인수 검증에 활용 (Must: Data Sources — 티켓 플랫폼 연동)<br><br>**비개발자 친화 리포팅**<br>• 핵심 트랜잭션·업무 흐름을 비기술자도 이해 가능한 다이어그램 + 자연어 요약으로 제공<br>• 기술 변경 영향을 업무 영향 언어로 번역 — "결제 모듈 변경 → 월 거래 N만 건 영향 가능성"<br>• 자동 리포트 생성으로 이해관계자 보고 자료 작성 시간 절감 (Should: 자동 문서, 가이드 생성) |
