# Feature Specification: ZIP 업로드 스냅샷과 워크스페이스 삭제

**Feature Branch**: `004-zip-upload-workspace-delete`

**Created**: 2026-05-06

**Status**: Draft

**Input**: User description: "기존 GitLab/GitHub 연동 기능에 추가로 ZIP 형태로 로컬 파일로 업로드 했을 때 스탭샷이 생성되는 기능을 추가하고 싶다. 그리고 기존에 생성된 워크스페이스를 삭제하는 기능을 추가하고 싶다. 사내망에서 외부 GitHub 연결도 안되고 자체 운영중인 GitLab도 없을 때, 로컬에서 프로젝트를 업로드할 필요성이 생김. 이 외에 자체 테스트에도 바로 프로젝트를 업로드하는게 편하다고 생각함. ZIP 형태로 업로드된 파일은 압축을 풀었을 때 형태 그대로 Snapshot을 떠서 보관. 기존 GitLab/GitHub 연동 기능과의 호환성을 모두 고려해야 함. 기존에 생성된 워크스페이스의 삭제 기능이 필요해짐"

## Clarifications

### Session 2026-05-06

- Q: 워크스페이스 삭제는 사용자 기능에서 어떤 의미여야 하는가? → A: Soft delete: 활성 목록과 작업 흐름에서는 제거하고, 최소 감사 메타데이터는 보존한다.
- Q: 로컬 ZIP 업로드는 제품에서 어떤 도메인으로 표현해야 하는가? → A: 별도 Local Upload 출처로 관리하고, Repository Connection으로 만들지 않는다.
- Q: 삭제된 워크스페이스의 프로젝트 내용과 스냅샷 파일은 보존해야 하는가? → A: 프로젝트 내용과 스냅샷 파일은 제거하고 최소 감사 메타데이터만 보존한다.
- Q: 같은 워크스페이스에서 ZIP을 여러 번 업로드하면 어떻게 처리해야 하는가? → A: 각 업로드를 독립 스냅샷으로 보존하고 최신 스냅샷을 기본 표시한다.
- Q: 워크스페이스 삭제는 누가 수행할 수 있어야 하는가? → A: 워크스페이스 소유자 또는 관리자만 삭제할 수 있다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ZIP 파일로 프로젝트 스냅샷 생성 (Priority: P1)

워크스페이스 관리자는 외부 GitHub에 연결할 수 없고 자체 GitLab도 없는 환경에서 로컬 프로젝트 ZIP 파일을 업로드해, 압축을 풀었을 때의 폴더와 파일 구조 그대로 워크스페이스 스냅샷을 생성할 수 있어야 한다.

**Why this priority**: 사내망이나 테스트 환경에서 원격 저장소 연결 없이도 프로젝트 분석을 시작할 수 있어야 이번 기능의 핵심 가치가 성립한다.

**Independent Test**: 새 워크스페이스에서 유효한 ZIP 파일을 업로드했을 때, 업로드 완료 후 생성된 스냅샷에서 압축 해제된 폴더와 파일 구조가 원본과 일치하고 스냅샷 목록과 상세에서 확인되는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 워크스페이스 관리자가 새 워크스페이스를 열고 로컬 업로드를 선택한 상태에서, **When** 유효한 프로젝트 ZIP 파일을 업로드하면, **Then** 시스템은 압축 해제 기준의 폴더와 파일 구조를 보존한 스냅샷을 생성해야 한다.
2. **Given** ZIP 파일에 루트 폴더가 포함되어 있을 때, **When** 스냅샷이 생성되면, **Then** 시스템은 사용자가 압축을 풀었을 때 보게 되는 최상위 구조를 스냅샷의 기준 구조로 보여줘야 한다.
3. **Given** ZIP 파일 업로드와 스냅샷 생성이 완료된 상태에서, **When** 사용자가 워크스페이스의 스냅샷 목록 또는 상세를 열면, **Then** 시스템은 로컬 업로드로 생성된 스냅샷임을 식별 가능하게 표시해야 한다.
4. **Given** 사용자가 테스트 목적으로 작은 프로젝트 ZIP 파일을 업로드할 때, **When** 업로드가 성공하면, **Then** 사용자는 추가 원격 저장소 설정 없이 해당 워크스페이스에서 스냅샷 기반 기능을 시작할 수 있어야 한다.

### User Story 2 - 생성된 워크스페이스 삭제 (Priority: P2)

워크스페이스 소유자 또는 관리자는 더 이상 사용하지 않는 워크스페이스를 명시적으로 삭제하고, 삭제 후 해당 워크스페이스와 관련 연결 및 스냅샷이 일반 작업 흐름에서 더 이상 사용되지 않도록 할 수 있어야 한다. 삭제는 사용자 기능에서는 즉시 제거되는 soft delete로 처리하며, 프로젝트 내용과 스냅샷 파일은 제거하고 최소 감사 메타데이터만 보존해야 한다.

**Why this priority**: 로컬 ZIP 업로드가 테스트와 임시 검증을 쉽게 만들면 불필요한 워크스페이스도 늘어나므로, 사용자가 워크스페이스 수명주기를 직접 정리할 수 있어야 한다.

**Independent Test**: 워크스페이스 소유자 또는 관리자가 확인 절차를 거쳐 워크스페이스를 삭제했을 때, 삭제된 워크스페이스가 활성 목록과 연결/스냅샷 작업 흐름에서 제외되고 재접근 시 삭제 상태가 명확히 안내되는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 워크스페이스 소유자 또는 관리자가 기존 워크스페이스 상세를 열었을 때, **When** 삭제를 선택하고 확인 절차를 완료하면, **Then** 시스템은 해당 워크스페이스를 활성 작업 대상에서 제거해야 한다.
2. **Given** 삭제된 워크스페이스에 GitHub, GitLab 또는 로컬 ZIP 스냅샷이 연결되어 있을 때, **When** 사용자가 활성 워크스페이스 목록을 조회하면, **Then** 시스템은 해당 워크스페이스와 하위 연결 및 스냅샷을 일반 작업 대상으로 표시하지 않고 프로젝트 내용과 스냅샷 파일을 재사용하지 않아야 한다.
3. **Given** 삭제된 워크스페이스 주소나 식별자로 접근을 시도할 때, **When** 사용자가 상세 화면을 열려고 하면, **Then** 시스템은 삭제된 워크스페이스임을 안내하고 새 작업을 시작하지 못하게 해야 한다.
4. **Given** 워크스페이스 소유자 또는 관리자가 아닌 사용자가 워크스페이스 삭제를 시도할 때, **When** 삭제 동작을 요청하면, **Then** 시스템은 삭제를 수행하지 않고 권한 문제를 명확히 알려줘야 한다.

### User Story 3 - 기존 GitHub/GitLab 연결과 로컬 업로드 호환성 유지 (Priority: P3)

워크스페이스 관리자는 같은 제품 흐름 안에서 GitHub 연결, GitLab 연결, 로컬 ZIP 업로드 스냅샷을 구분해 관리할 수 있어야 하며, 새 로컬 업로드 기능과 삭제 기능이 기존 GitHub/GitLab 연결의 상태, 이벤트, 스냅샷 조회 경험을 깨뜨리지 않아야 한다. 로컬 ZIP 업로드는 Repository Connection으로 만들지 않고 별도 Local Upload 출처의 스냅샷으로 표현해야 한다.

**Why this priority**: 로컬 ZIP 업로드가 새 입력 경로를 추가하더라도 이미 운영 중인 GitHub/GitLab 연결 신뢰성을 훼손하면 기능 채택이 어렵다.

**Independent Test**: GitHub 연결, GitLab 연결, 로컬 ZIP 스냅샷이 함께 있는 워크스페이스에서 각 출처가 명확히 구분되고 기존 GitHub/GitLab 목록, 상세, 스냅샷 조회가 기존 기준선과 동일하게 동작하는지 검증한다.

**Acceptance Scenarios**:

1. **Given** 기존 GitHub 연결이 있는 워크스페이스에 로컬 ZIP 스냅샷이 추가된 상태에서, **When** 사용자가 연결과 스냅샷 목록을 조회하면, **Then** 시스템은 GitHub 연결 상태와 Local Upload 출처의 스냅샷을 혼동 없이 구분해 보여줘야 한다.
2. **Given** 기존 GitLab 연결이 있는 워크스페이스에 로컬 ZIP 스냅샷이 추가된 상태에서, **When** 사용자가 연결과 스냅샷 상세를 조회하면, **Then** 시스템은 GitLab 연결의 기존 운영 정보와 로컬 업로드 출처를 각각 보존해야 한다.
3. **Given** 사용자가 로컬 ZIP 업로드 스냅샷을 생성한 뒤 같은 워크스페이스에 GitHub 또는 GitLab 저장소를 연결하려 할 때, **When** 연결 흐름을 시작하면, **Then** 시스템은 로컬 업로드 이력을 원격 저장소 연결의 필수 조건으로 요구하지 않아야 한다.
4. **Given** 사용자가 GitHub/GitLab 연결이 있는 워크스페이스를 삭제할 때, **When** 삭제 확인을 완료하면, **Then** 시스템은 provider별 기존 연결 운영 의미를 바꾸지 않고 해당 워크스페이스 범위만 비활성화해야 한다.

### Edge Cases

- ZIP 파일이 손상되었거나 압축을 풀 수 없는 경우, 시스템은 스냅샷을 생성하지 않고 사용자가 이해할 수 있는 실패 사유를 알려줘야 한다.
- ZIP 파일 안에 워크스페이스 밖의 위치를 가리키는 위험한 경로나 사용할 수 없는 항목이 포함된 경우, 시스템은 해당 업로드를 안전하게 거부해야 한다.
- ZIP 파일이 허용된 크기나 파일 수 한도를 초과하는 경우, 시스템은 업로드를 중단하고 허용 기준과 재시도 방법을 안내해야 한다.
- ZIP 파일 안에 빈 폴더, 숨김 파일, 중첩 폴더가 포함된 경우, 시스템은 압축 해제 시 보이는 프로젝트 구조를 기준으로 스냅샷에 반영해야 한다.
- 같은 워크스페이스에서 여러 ZIP 스냅샷을 순차적으로 업로드하는 경우, 시스템은 각 스냅샷의 생성 시점과 출처를 구분해야 한다.
- 같은 워크스페이스에서 여러 ZIP 스냅샷이 있는 경우, 시스템은 각 업로드를 독립 스냅샷으로 보존하고 최신 Local Upload 스냅샷을 기본 표시 대상으로 삼아야 한다.
- 삭제 진행 중인 워크스페이스에 새 ZIP 업로드 또는 GitHub/GitLab 연결 생성 요청이 들어오면, 시스템은 새 작업을 시작하지 않아야 한다.
- 이미 삭제된 워크스페이스에 다시 삭제 요청이 들어오면, 시스템은 중복 삭제를 수행하지 않고 현재 삭제 상태를 알려줘야 한다.
- 워크스페이스 삭제가 완료된 뒤에도 최소 감사 메타데이터는 보존하되, 프로젝트 내용과 스냅샷 파일은 제거하고 일반 사용자 작업 흐름에서 재사용하지 않아야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 워크스페이스에서 로컬 프로젝트 ZIP 파일을 업로드하는 입력 경로를 제공해야 한다.
- **FR-002**: 시스템은 유효한 ZIP 파일 업로드가 완료되면 압축 해제 기준의 폴더와 파일 구조를 보존한 스냅샷을 생성해야 한다.
- **FR-003**: 시스템은 로컬 ZIP 업로드로 생성된 스냅샷을 Repository Connection으로 만들지 않고, 기존 스냅샷 목록과 상세 흐름에서 별도 Local Upload 출처로 조회 가능하게 해야 한다.
- **FR-004**: 시스템은 로컬 ZIP 업로드 스냅샷의 Local Upload 출처, 업로드 시점, 업로드한 사용자, 원본 파일 식별 정보를 사용자가 이해할 수 있는 수준으로 표시해야 한다.
- **FR-005**: 시스템은 손상된 ZIP, 허용되지 않는 압축 항목, 한도 초과 업로드에 대해 스냅샷을 생성하지 않고 명확한 실패 안내를 제공해야 한다.
- **FR-006**: 시스템은 로컬 ZIP 업로드 스냅샷 생성 실패 시 부분 생성된 스냅샷을 활성 스냅샷으로 보여주지 않아야 한다.
- **FR-007**: 시스템은 한 워크스페이스 안에서 여러 로컬 ZIP 업로드 스냅샷을 생성하고 각 업로드를 독립 스냅샷으로 보존하며 생성 시점별로 구분할 수 있게 해야 한다.
- **FR-007a**: 시스템은 한 워크스페이스에 여러 Local Upload 스냅샷이 있을 때 최신 스냅샷을 기본 표시 대상으로 사용해야 한다.
- **FR-008**: 시스템은 기존 GitHub 연결과 GitLab 연결의 목록, 상세, 상태, 이벤트, 스냅샷 조회 의미를 로컬 ZIP 업로드 추가 이후에도 유지해야 한다.
- **FR-009**: 시스템은 GitHub Repository Connection, GitLab Repository Connection, Local Upload 출처가 같은 워크스페이스 안에 함께 있을 때 각 출처를 혼동 없이 구분해 보여줘야 한다.
- **FR-010**: 시스템은 로컬 ZIP 업로드 스냅샷을 GitHub 또는 GitLab 원격 연결의 필수 선행 조건으로 요구하지 않아야 한다.
- **FR-011**: 시스템은 워크스페이스 소유자 또는 관리자만 워크스페이스 삭제를 시작하고 확인할 수 있는 흐름을 제공해야 한다.
- **FR-012**: 시스템은 워크스페이스 삭제 전 삭제 대상과 영향을 받는 연결, 스냅샷, 작업 범위를 사용자가 확인할 수 있게 해야 한다.
- **FR-013**: 시스템은 워크스페이스 삭제를 soft delete로 처리하고, 삭제가 완료되면 해당 워크스페이스를 활성 목록, 연결 생성, ZIP 업로드, 스냅샷 생성, 일반 상세 조회 대상에서 제외해야 한다.
- **FR-014**: 시스템은 삭제된 워크스페이스에 접근하려는 사용자에게 삭제 상태와 가능한 다음 행동을 안내해야 한다.
- **FR-015**: 시스템은 워크스페이스 소유자 또는 관리자가 아닌 사용자의 워크스페이스 삭제 요청을 수행하지 않고 권한 문제를 알려줘야 한다.
- **FR-016**: 시스템은 삭제 중이거나 삭제된 워크스페이스에 대해 새 GitHub 연결, GitLab 연결, 로컬 ZIP 업로드, 스냅샷 생성 작업을 시작하지 않아야 한다.
- **FR-017**: 시스템은 워크스페이스 삭제 후에도 최소 감사 메타데이터를 일반 작업 흐름과 분리해 유지해야 한다.
- **FR-018**: 시스템은 워크스페이스 삭제가 기존 GitHub/GitLab provider 자체의 의미나 다른 워크스페이스의 연결 상태를 변경하지 않도록 해야 한다.
- **FR-019**: 시스템은 워크스페이스 삭제 완료 후 해당 워크스페이스의 프로젝트 내용과 스냅샷 파일을 일반 작업 흐름에서 재사용할 수 없게 제거해야 한다.

### Key Entities

- **Workspace**: 사용자가 프로젝트 연결, 로컬 업로드, 스냅샷을 관리하는 작업 공간. 활성, 삭제 중, 삭제됨 상태를 포함한다.
- **Local Upload**: 사용자가 워크스페이스에 제출한 ZIP 파일 기반 프로젝트 입력. Repository Connection이 아니며, 업로드한 사용자, 업로드 시점, 원본 파일 식별 정보, 처리 결과를 포함한다.
- **Snapshot**: 특정 시점의 프로젝트 파일 구조와 내용을 보관한 결과. GitHub, GitLab, 로컬 ZIP 업로드 중 어떤 출처에서 생성되었는지 구분되어야 하며, Local Upload 스냅샷은 업로드마다 독립적으로 생성된다.
- **Repository Connection**: 워크스페이스에 연결된 GitHub 또는 GitLab 저장소. Local Upload와 병존할 수 있으나 Local Upload를 포함하지 않으며 기존 provider별 운영 의미를 유지한다.
- **Deletion Record**: 워크스페이스 soft delete 요청과 결과를 설명하는 이력. 삭제 요청자, 확인 시점, 삭제 상태, 프로젝트 내용 및 스냅샷 파일 제거 결과, 일반 작업 흐름과 분리해 보존되는 최소 감사 메타데이터를 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 대표 운영자 3명이 각자 GitHub/GitLab 없이 ZIP 업로드만으로 프로젝트 스냅샷 생성을 1회씩 수행했을 때, 3회 중 3회 모두 5분 이내에 완료해야 한다.
- **SC-002**: 유효한 ZIP 업로드 수락 테스트의 100%에서 압축 해제 기준의 최상위 폴더와 파일 구조가 생성된 스냅샷 상세와 일치해야 한다.
- **SC-003**: 손상된 ZIP, 위험한 경로, 한도 초과 ZIP을 포함한 실패 시나리오의 100%가 활성 스냅샷을 남기지 않고 사용자가 이해할 수 있는 실패 안내를 제공해야 한다.
- **SC-004**: 기존 GitHub Cloud 및 온프레미스 GitLab 기준선 시나리오의 100%가 로컬 ZIP 업로드 기능 추가 이후에도 추가 수동 우회 없이 성공해야 한다.
- **SC-005**: GitHub, GitLab, 로컬 ZIP 업로드가 함께 있는 워크스페이스에서 대표 운영자 3명이 총 30개 출처 식별 과제를 수행했을 때, 29개 이상을 올바르게 구분해야 한다.
- **SC-006**: 워크스페이스 소유자 또는 관리자의 워크스페이스 삭제 수락 테스트의 100%에서 삭제된 워크스페이스가 활성 목록과 새 작업 시작 흐름에서 제외되어야 한다.
- **SC-007**: 워크스페이스 소유자 또는 관리자가 아닌 사용자의 워크스페이스 삭제 시도 100%가 워크스페이스 상태를 바꾸지 않고 권한 문제를 표시해야 한다.
- **SC-008**: 삭제된 워크스페이스 접근 시나리오의 100%가 새 연결, 새 업로드, 새 스냅샷 생성을 허용하지 않고 삭제 상태를 안내해야 한다.
- **SC-009**: 워크스페이스 삭제 완료 수락 테스트의 100%에서 프로젝트 내용과 스냅샷 파일이 일반 작업 흐름에서 재사용 불가능하고 최소 감사 메타데이터만 조회 가능해야 한다.
- **SC-010**: 같은 워크스페이스에 ZIP 파일을 3회 연속 업로드하는 수락 테스트의 100%에서 3개의 독립 Local Upload 스냅샷이 생성되고 최신 스냅샷이 기본 표시되어야 한다.

## Assumptions

- 로컬 ZIP 업로드는 워크스페이스 관리 권한이 있는 사용자가 수행하고, 워크스페이스 삭제는 워크스페이스 소유자 또는 관리자만 수행한다고 가정한다.
- 1차 범위의 로컬 업로드 형식은 ZIP 파일로 한정하며, 다른 압축 형식이나 폴더 직접 업로드는 범위에 포함하지 않는다.
- ZIP 업로드 스냅샷은 압축 해제 후 일반적으로 보이는 폴더와 파일 구조를 기준으로 하며, 위험한 경로나 사용할 수 없는 압축 항목은 안전을 위해 거부한다.
- ZIP 파일 크기, 파일 수, 개별 파일 크기 한도는 계획 단계에서 기존 운영 용량과 사용자 경험 기준에 맞춰 정하되, 초과 시에는 스냅샷을 생성하지 않는다.
- 로컬 ZIP 업로드는 GitHub/GitLab 원격 저장소 연결을 대체할 수 있는 별도 Local Upload 입력 경로이며, Repository Connection, 원격 provider 이벤트, 자동 동기화를 제공한다고 가정하지 않는다.
- 워크스페이스 삭제는 soft delete로 정의하며, 일반 사용자 작업 흐름에서 워크스페이스와 하위 연결/스냅샷을 더 이상 사용할 수 없게 만든다.
- 법적, 보안, 감사 목적상 필요한 최소 삭제 이력은 보존하되, 삭제된 워크스페이스의 프로젝트 내용과 스냅샷 파일은 제거하고 일반 사용자 기능에서 재사용되지 않는다고 가정한다.
- 기존 GitHub/GitLab 연결의 provider별 검증, 이벤트, 스냅샷 조회 의미는 이번 기능에서 재설계하지 않는다.
