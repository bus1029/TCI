# Feature Specification: 코드 저장소 연동

**Feature Branch**: `[006-git-repo-connection]`  
**Created**: 2026-04-16  
**Status**: Draft  
**Input**: User description: "TCI의 데이터 수집 영역에서 Git 기반 형상 관리 시스템과 연동하여 코드베이스와 변경 이력을 수집하고 분석 가능한 코드 스냅샷을 생성한다. Git 기반 레포지토리 연결(SSH/HTTPS), 분석 대상 브랜치/태그 선택, 제외/포함 경로 및 파일 타입 설정, 코드 변경 이벤트 감지(Commit, Push, PR), Webhook 기반 실시간 이벤트 수신."

## Design Input Traceability *(mandatory)*

- **Planning Source**: 2026-04-16 사용자 요청 "TCI 데이터 수집 영역의 코드 저장소 연동"
- **Why now**: TCI가 코드 변경과 산출물을 분석 대상으로 삼기 위해서는 기획 입력과 연결되는 신뢰 가능한 코드 수집 기반이 먼저 필요하다.
- **Scope baseline**: Git 기반 저장소 연결, 수집 대상 ref 선택, 수집 범위 필터 설정, 변경 이벤트 수신, 분석 가능한 코드 스냅샷 생성
- **Out of scope**: Git 이외 형상 관리 시스템 지원, 저장소에 대한 쓰기 작업, CI/CD 실행, 코드 품질 점수화나 리뷰 자동화 자체

## Clarifications

### Session 2026-04-16

- Q: 어떤 이벤트를 실제 코드 스냅샷 갱신 트리거로 사용할 것인가? → A: Push와 PR은 스냅샷 갱신 트리거로 처리하고, Commit은 이벤트 기록만 수행한다.
- Q: PR 이벤트가 스냅샷을 갱신할 때 어떤 기준 ref를 사용할 것인가? → A: PR의 source branch 최신 HEAD를 기준으로 스냅샷을 생성한다.
- Q: 기본 수집 제외 정책은 무엇으로 둘 것인가? → A: 텍스트 기반 소스 파일만 기본 수집하고, 바이너리·생성 산출물·대용량 파일은 기본 제외한다.
- Q: 바이너리, 생성 산출물, 대용량 파일에 사용자 예외 포함 규칙을 허용할 것인가? → A: v1에서는 사용자 예외 규칙이 있어도 이 범주의 파일은 수집하지 않고, 후속 범위로 남긴다.
- Q: 저장소 연결 1건이 몇 개의 분석 대상 ref를 가질 것인가? → A: 저장소 연결 1건은 기본 분석 대상 ref 1개만 가지며, PR source branch는 이벤트 기반 예외 처리로 다룬다.
- Q: 웹훅 이벤트의 신뢰성은 어떻게 검증할 것인가? → A: 저장소 연결별 webhook secret을 등록하고, 서명 검증에 성공한 이벤트만 처리한다.
- Q: Code Snapshot을 어떤 보존 단위로 저장할 것인가? → A: 각 수집 성공 시점마다 필터 적용 후 전체 파일 집합을 완전한 스냅샷으로 저장한다.
- Q: 1차 릴리스에서 공식 지원할 Git 저장소 서비스 범위는 무엇인가? → A: GitHub Cloud만 1차 공식 지원 대상으로 한다.
- Q: 저장소 접근 자격 증명은 누구에게 귀속되는가? → A: 자격 증명은 저장소 연결 단위의 공유 비밀정보로 저장하고, 연결을 아는 운영자가 공용으로 관리한다.
- Q: 중복 수신되거나 순서가 바뀐 이벤트는 어떤 기준으로 처리할 것인가? → A: webhook delivery ID로 중복 이벤트를 제거하고, ref 또는 PR source branch별 최신 HEAD SHA만 스냅샷 반영 대상으로 인정한다.
- Q: 저장소 접근 자격 증명은 어떤 권한 수준만 허용할 것인가? → A: SSH와 HTTPS 모두 읽기 전용 자격 증명만 허용한다.
- Q: 기존 저장소 연결의 기본 분석 대상 ref를 바꾸면 기존 스냅샷과 이후 수집은 어떻게 처리할 것인가? → A: 기존 스냅샷과 이벤트 이력은 유지하고, ref 변경 이후 새로 실행되는 수집과 스냅샷만 변경된 기본 ref를 기준으로 처리한다.
- Q: 추가 브랜치나 태그를 상시 분석하려는 사용자는 어떤 선택지를 받아야 하는가? → A: 시스템은 현재 연결이 기본 ref 1개만 지원함을 명확히 안내하고, 새 연결 생성 또는 기존 기본 ref 교체 중 하나를 선택하게 한다.
- Q: secret 불일치 이벤트는 일반 서명 검증 실패와 어떻게 구분할 것인가? → A: 시스템은 secret 누락, secret 불일치, 기타 서명 검증 실패를 서로 다른 거부 사유로 기록하고, 운영자가 각각에 맞는 재설정 필요 상태를 확인할 수 있게 한다.
- Q: 저장소 자격 증명이 만료되거나 취소된 경우 어떤 연결 상태와 차단 규칙을 적용할 것인가? → A: 시스템은 연결 상태를 `reauth_required`로 전환하고, 운영자가 자격 증명을 다시 등록하기 전까지 신규 검증과 수집을 차단해야 한다.
- Q: 기본 분석 대상 ref가 삭제되거나 이름이 변경된 경우 어떤 연결 상태와 차단 규칙을 적용할 것인가? → A: 시스템은 연결 상태를 `ref_missing`으로 전환하고, 운영자가 새 기본 ref를 선택하기 전까지 신규 수집을 차단해야 한다.
- Q: 최신 성공 수집 시각, 마지막 실패 시각, 마지막 처리 이벤트는 어디서 보장할 것인가? → A: 시스템은 connection detail에서 이 값을 요약 필드로 항상 제공하고, 상세 이력은 별도 event timeline 조회에서 제공해야 한다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 저장소 연결과 초기 스냅샷 생성 (Priority: P1)

수집 담당자는 분석 대상 Git 저장소를 연결하고, 기본 분석 대상이 되는 브랜치 또는 태그 1개를 선택한 뒤 첫 번째 코드 스냅샷을 생성할 수 있어야 한다.

**Why this priority**: 저장소 연결과 초기 스냅샷이 성립해야 이후의 변경 감지와 분석 파이프라인이 모두 가능해진다.

**Independent Test**: 유효한 저장소 주소와 접근 정보를 입력하고 기본 분석 대상 ref 1개를 선택한 뒤, 사용자가 첫 스냅샷 생성 완료와 수집 결과를 확인할 수 있으면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 사용자가 유효한 Git 저장소 주소와 접근 방식을 제공했고 연결 권한이 유효한 상태에서, **When** 저장소 연결을 저장하면, **Then** 시스템은 연결 가능 여부를 확인하고 분석 가능한 저장소 연결을 생성한다.
2. **Given** 활성화된 저장소 연결과 기본 분석 대상 브랜치 또는 태그 1개가 선택된 상태에서, **When** 사용자가 초기 수집을 실행하면, **Then** 시스템은 해당 ref 기준의 코드 스냅샷을 생성하고 수집 완료 상태를 보여준다.
3. **Given** 활성화된 저장소 연결에 기존 기본 분석 대상 ref가 설정된 상태에서, **When** 사용자가 다른 브랜치 또는 태그로 기본 ref를 변경하면, **Then** 시스템은 기존 스냅샷 이력을 유지한 채 이후 새로 생성되는 수집과 스냅샷부터 변경된 기본 ref를 기준으로 처리해야 한다.

### User Story 2 - 수집 범위 제어 (Priority: P2)

수집 담당자는 포함 경로, 제외 경로, 파일 타입 조건을 설정하여 분석 대상 코드 범위를 제어할 수 있어야 한다. 초기 기본 정책은 텍스트 기반 소스 파일만 수집하고, 바이너리·생성 산출물·대용량 파일은 기본 제외한다.

**Why this priority**: 저장소 전체를 수집하면 분석 비용과 노이즈가 커질 수 있으므로, 실제로 의미 있는 코드만 대상으로 삼는 범위 제어가 필요하다.

**Independent Test**: 연결된 저장소에 대해 경로 및 파일 타입 규칙을 저장하고 새 스냅샷을 생성했을 때, 결과에 포함된 파일 목록이 규칙과 일치하면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 활성화된 저장소 연결이 있을 때, **When** 사용자가 포함 경로, 제외 경로, 파일 타입 규칙을 설정하고 저장하면, **Then** 이후 생성되는 스냅샷은 해당 규칙을 기준으로 수집 범위를 결정한다.
2. **Given** 저장된 범위 규칙이 실제로 수집 가능한 파일을 남기지 않는 상태에서, **When** 사용자가 규칙을 확정하거나 스냅샷 생성을 시도하면, **Then** 시스템은 빈 수집 결과 위험을 알려주고 사용자가 규칙을 조정할 수 있게 한다.
3. **Given** 저장소에 바이너리, 생성 산출물, 과도하게 큰 파일이 포함되어 있을 때, **When** 사용자가 별도 예외 규칙을 두지 않고 스냅샷을 생성하면, **Then** 시스템은 해당 파일들을 기본 제외 대상으로 처리해야 한다.

### User Story 3 - 변경 이벤트 기반 최신화 (Priority: P3)

시스템 운영자는 연결된 저장소에서 발생하는 Commit, Push, PR 이벤트를 실시간으로 받아 코드 변경 이력을 기록하고, Push 및 PR 이벤트를 기준으로 후속 스냅샷 최신화를 관리할 수 있어야 한다. 저장소 연결은 기본 분석 대상 ref 1개만 유지하며, PR 기반 최신화는 예외적으로 PR의 source branch 최신 HEAD를 기준으로 수행되어야 한다. 웹훅 이벤트는 저장소 연결별 secret 서명 검증을 통과한 경우에만 처리되어야 한다.

**Why this priority**: 초기 수집만으로는 최신 상태 분석이 불가능하므로, 변경 이벤트를 통해 스냅샷과 이력 데이터를 지속적으로 갱신해야 한다.

**Independent Test**: 연결된 저장소에서 Commit, Push, PR 이벤트를 발생시키고 시스템이 secret 서명 검증에 성공한 이벤트만 수용하며, Commit은 기록 전용으로, Push와 PR은 스냅샷 최신화 트리거로 구분 처리하면 이 스토리는 독립적으로 검증된다.

**Acceptance Scenarios**:

1. **Given** 웹훅이 연결된 활성 저장소가 있을 때, **When** Commit 이벤트가 수신되면, **Then** 시스템은 이벤트를 기록하지만 별도의 스냅샷 최신화 작업은 시작하지 않는다.
2. **Given** 웹훅이 연결된 활성 저장소가 있을 때, **When** Push 이벤트가 수신되면, **Then** 시스템은 이벤트를 기록하고 해당 ref 기준의 최신화 작업을 시작한다.
3. **Given** Pull Request 이벤트가 수신된 상태에서, **When** 시스템이 관련 스냅샷 최신화를 시작하면, **Then** 시스템은 PR의 source branch 최신 HEAD를 기준으로 스냅샷을 생성해야 한다.
4. **Given** 웹훅 요청의 서명 검증에 실패한 상태에서, **When** 시스템이 이벤트를 수신하면, **Then** 시스템은 해당 이벤트를 처리 대상으로 인정하지 않고 거부 상태를 기록해야 한다.
5. **Given** 웹훅 요청에 서명 헤더는 존재하지만 등록된 secret과 일치하지 않는 상태에서, **When** 시스템이 이벤트를 수신하면, **Then** 시스템은 이를 일반 서명 검증 실패나 secret 누락과 다른 별도의 거부 사유로 기록하고 운영자가 재설정 필요 상태를 확인할 수 있게 해야 한다.

### Edge Cases

- 저장소 연결 후 접근 권한이 만료되거나 취소되면 연결 상태를 `reauth_required`로 전환하고, 자격 증명 재등록 전까지 신규 검증과 수집을 어떻게 차단하고 안내할 것인가?
- 사용자가 선택한 브랜치 또는 태그가 삭제되거나 이름이 변경되면 연결 상태를 `ref_missing`으로 전환하고, 운영자가 새 기본 ref를 선택하기 전까지 신규 수집을 어떻게 차단하고 안내할 것인가?
- 한 저장소 연결이 기본 분석 대상 ref 1개만 가질 때, 추가 브랜치나 태그 분석이 필요하면 새 연결을 만들지 기존 연결을 바꿀지 어떻게 안내할 것인가?
- 사용자가 기본 분석 대상 ref를 다른 브랜치 또는 태그로 변경할 때 기존 스냅샷 이력과 이후 수집 기준은 어떻게 분리되는가?
- 동일한 Commit, Push, PR 이벤트가 중복 수신되거나 순서가 바뀌어 들어오면 어떤 기준으로 최신 상태를 판단하는가?
- Commit 이벤트는 기록만 하고 Push/PR만 스냅샷을 갱신할 때, 같은 변경이 서로 다른 이벤트로 들어오면 어떤 우선순위로 합쳐 처리할 것인가?
- PR의 source branch가 force push 되거나 target branch가 변경될 때 기존 PR 기반 스냅샷을 어떻게 갱신하거나 대체할 것인가?
- 포함 규칙과 제외 규칙이 충돌하거나 결과적으로 분석 가능한 파일이 하나도 남지 않으면 어떤 경고와 차단이 필요한가?
- 바이너리, 생성 산출물, 대용량 파일은 v1에서 사용자 예외 포함 규칙이 있더라도 수집하지 않을 때, 시스템은 이를 어떤 경고와 안내로 설명할 것인가?
- webhook secret이 누락되었거나 회전된 경우 어떤 연결 상태와 재설정 절차를 보여줄 것인가?
- 웹훅 요청에 서명은 있으나 등록된 secret과 일치하지 않는 경우 이를 어떤 재설정 필요 상태로 보여줄 것인가?
- 운영자가 webhook secret을 회전하는 동안 GitHub 설정 반영 시차가 있으면, 이전 secret과 새 secret 중 무엇을 언제까지 허용해야 하는가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자가 Git 기반 저장소를 SSH 또는 HTTPS 방식으로 연결 등록할 수 있게 해야 한다.
- **FR-001a**: 시스템은 1차 릴리스에서 GitHub Cloud 저장소만 공식 지원 대상으로 받아들여야 한다.
- **FR-001b**: 시스템은 저장소 접근 자격 증명을 사용자 개인 세션이 아닌 저장소 연결 단위의 공유 비밀정보로 저장하고, 백그라운드 수집과 재시도 작업이 해당 연결 자격 증명을 사용하게 해야 한다.
- **FR-001c**: 시스템은 1차 릴리스에서 SSH와 HTTPS 모두 읽기 전용 자격 증명만 허용해야 하며, 쓰기 권한이 포함된 자격 증명은 연결 대상으로 승인하지 않아야 한다.
- **FR-002**: 시스템은 저장소 연결 등록 시 접근 가능 여부를 확인하고, 연결 실패 시 원인을 이해할 수 있는 상태를 제공해야 한다.
- **FR-002a**: 시스템은 저장소 연결 이후 저장된 자격 증명이 만료되거나 취소되면 연결 상태를 `reauth_required`로 전환하고, 운영자가 자격 증명을 다시 등록하기 전까지 신규 검증과 수집을 차단해야 한다.
- **FR-003**: 시스템은 각 저장소 연결에 대해 기본 분석 대상으로 사용할 브랜치 또는 태그 1개를 선택하고 변경할 수 있게 해야 한다.
- **FR-003a**: 시스템은 기본 분석 대상 ref가 변경되더라도 기존 스냅샷과 이벤트 이력은 유지해야 하며, 변경 이후 새로 생성되는 수집과 스냅샷만 새 기본 ref를 기준으로 처리해야 한다.
- **FR-003b**: 시스템은 저장된 기본 분석 대상 ref가 삭제되거나 이름이 변경되어 더 이상 유효하지 않으면 연결 상태를 `ref_missing`으로 전환하고, 운영자가 새 기본 ref를 선택하기 전까지 신규 수집을 차단해야 한다.
- **FR-004**: 시스템은 저장소 연결마다 포함 경로, 제외 경로, 파일 타입 조건을 설정하고 수정할 수 있게 해야 한다.
- **FR-005**: 시스템은 생성되는 모든 코드 스냅샷에 저장소 식별 정보, 선택된 ref, 수집 시각, 적용된 범위 규칙을 함께 기록해야 한다.
- **FR-005a**: 시스템은 각 수집 성공 시점마다 해당 시점의 필터 적용 결과 전체 파일 집합을 하나의 완전한 코드 스냅샷으로 고정 저장해야 한다.
- **FR-006**: 시스템은 텍스트 기반 소스 파일을 기본 수집 대상으로 삼고, 바이너리·생성 산출물·대용량 파일은 1차 릴리스(v1)에서는 사용자가 예외 규칙을 두더라도 수집 대상에 포함하지 않아야 한다.
- **FR-007**: 시스템은 설정된 범위 규칙을 일관되게 적용하여 분석 가능한 파일만 코드 스냅샷에 포함해야 한다.
- **FR-008**: 시스템은 Push 및 Pull Request 이벤트가 수신되면 해당 이벤트와 연결되는 코드 스냅샷 최신화 흐름을 시작해야 한다.
- **FR-009**: 시스템은 Commit 이벤트를 변경 이력 추적용으로 기록하되, Commit 이벤트만으로는 별도의 코드 스냅샷 최신화 흐름을 시작하지 않아야 한다.
- **FR-010**: 시스템은 Pull Request 이벤트로 생성되는 코드 스냅샷을 해당 Pull Request의 source branch 최신 HEAD 기준으로 생성해야 한다.
- **FR-011**: 시스템은 동일한 변경 상태를 반복 반영하여 중복 스냅샷을 대량 생성하지 않도록 이벤트 처리 결과를 구분할 수 있어야 한다.
- **FR-011a**: 시스템은 webhook delivery ID를 기준으로 중복 수신 이벤트를 식별하고 재처리를 방지해야 한다.
- **FR-011b**: 시스템은 ref 및 PR source branch별로 가장 최신 HEAD SHA만 스냅샷 반영 대상으로 인정하고, 더 오래된 SHA를 가리키는 지연 도착 이벤트는 이력만 남기고 스냅샷 갱신 대상에서 제외해야 한다.
- **FR-012**: 시스템은 사용자가 connection detail에서 최신 성공 수집 시각, 마지막 실패 시각, 마지막 처리 이벤트를 요약 필드로 확인할 수 있게 해야 한다.
- **FR-012a**: 시스템은 마지막 처리 이벤트를 포함한 상세 이벤트 이력을 별도 event timeline 조회에서 제공해야 한다.
- **FR-013**: 시스템은 범위 규칙으로 인해 수집 대상 파일이 없거나 저장소 접근이 불가능한 경우, 스냅샷 생성을 완료 처리하지 않고 수정 가능한 안내를 제공해야 한다.
- **FR-014**: 시스템은 계획 입력, 저장소 연결 설정, 이벤트 기록, 코드 스냅샷 사이의 추적 관계를 유지해 변경 이력을 재구성할 수 있게 해야 한다.
- **FR-015**: 시스템은 저장소 연결 1건당 기본 분석 대상 ref를 1개만 유지해야 하며, 추가 브랜치 또는 태그를 상시 분석하려면 별도 연결 생성 또는 기본 ref 변경으로 처리해야 한다.
- **FR-015a**: 시스템은 사용자가 추가 브랜치 또는 태그를 상시 분석하려는 경우, 현재 연결이 기본 ref 1개만 지원함을 명확히 안내하고 새 연결 생성 또는 기존 기본 ref 변경 중 하나를 선택할 수 있게 해야 한다.
- **FR-016**: 시스템은 저장소 연결별 webhook secret을 등록하고, 서명 검증에 성공한 이벤트만 처리해야 한다.
- **FR-016a**: 시스템은 운영자가 webhook secret을 교체할 때, 명시된 유예 기간 동안 직전 secret과 새 secret 중 어느 것으로 서명된 요청인지 구분하면서 둘 다 유효한 secret으로 인정할 수 있어야 한다.
- **FR-017**: 시스템은 secret 누락, secret 불일치, 기타 서명 검증 실패를 서로 다른 거부 사유로 구분해 기록하고 사용자가 각각의 재설정 필요 상태를 확인할 수 있게 해야 한다.
- **FR-017a**: 시스템은 webhook secret 회전이 진행 중인 연결에 대해 유예 기간 종료 시각, 현재 회전 상태, 그리고 유예 기간 동안 직전 secret으로 수신된 이벤트 여부를 사용자가 확인할 수 있게 해야 한다.

### Key Entities *(include if feature involves data)*

- **Repository Connection**: 분석 대상으로 등록된 GitHub Cloud 저장소 연결 정보. 저장소 주소, 연결 방식, `active`/`reauth_required`/`ref_missing`를 포함한 연결 상태, 기본 분석 대상 ref 1개, 저장소 연결 단위의 공유 자격 증명, webhook secret 상태를 포함한다.
- **Collection Scope Rule**: 저장소에서 어떤 파일을 수집할지 결정하는 범위 규칙. 포함 경로, 제외 경로, 파일 타입 조건을 포함한다.
- **Code Snapshot**: 특정 시점의 분석 가능한 코드 수집 결과. 저장소, 기준 ref, 수집 시각, 적용된 범위 규칙, 포함된 전체 파일 집합을 갖는 완전한 스냅샷으로 저장된다.
- **Repository Event**: Commit, Push, Pull Request 등 저장소에서 발생한 변경 이벤트 기록. 이벤트 유형, webhook delivery ID, 관련 ref, PR의 경우 source/target ref와 HEAD SHA, 스냅샷 갱신 여부, 처리 상태, 연결된 최신화 결과를 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 권한이 유효한 사용자는 저장소 주소 입력부터 첫 코드 스냅샷 생성 완료 확인까지 10분 이내에 끝낼 수 있다.
- **SC-002**: 유효하게 전달된 저장소 변경 이벤트의 95% 이상이 1분 이내에 시스템 상의 처리 상태로 확인 가능하다.
- **SC-003**: 생성된 코드 스냅샷의 100%는 어떤 저장소, 어떤 브랜치 또는 태그, 어떤 범위 규칙, 어떤 시점의 수집 결과인지 추적 가능하다.
- **SC-004**: 사용자가 설정한 포함/제외 규칙 및 파일 타입 규칙과 실제 스냅샷 결과의 일치율이 100%여야 한다.
- **SC-005**: 계획된 webhook secret 회전이 유예 기간 안에 완료된 경우, 유효한 변경 이벤트는 중단 없이 처리 상태로 확인 가능해야 한다.

## Assumptions

- 초기 릴리스의 주 사용자는 저장소 연결 권한과 웹훅 설정 권한을 가진 내부 운영자 또는 수집 담당자다.
- 지원 대상은 1차 릴리스 기준으로 GitHub Cloud 저장소이며, 브랜치/태그 선택과 webhook 이벤트 제공이 가능한 저장소를 전제로 한다.
- 분석 가능한 코드 스냅샷은 사람이 읽고 후속 분석에 활용할 수 있는 텍스트 기반 파일을 우선 대상으로 하며, 바이너리나 생성 산출물, 대용량 파일은 기본 제외 대상이다.
- 저장소 연결에 사용하는 SSH 키와 HTTPS 토큰/자격 증명은 읽기 전용 권한만 가진다고 가정한다.
- TCI는 이 기능에서 저장소 데이터를 읽고 수집하지만, 원격 저장소의 코드를 수정하거나 배포를 수행하지 않는다.
